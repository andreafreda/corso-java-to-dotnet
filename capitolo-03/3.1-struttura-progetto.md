# 3.1 Struttura di un progetto ASP.NET Core

> **Analogia Java**: equivalente di un progetto Spring Boot

---

## Introduzione

Un progetto ASP.NET Core è strutturalmente molto simile a un progetto Spring Boot: un entry point, un file di configurazione, una cartella per i controller, una per i modelli. Se hai mai creato un progetto con Spring Initializr, riconoscerai subito l'idea — cambiano i nomi, non la logica.

---

## Come si faceva in Java

Con Spring Initializr o `spring-boot:run` ottieni questa struttura:

```
my-service/
├── src/
│   └── main/
│       ├── java/com/example/myservice/
│       │   ├── MyServiceApplication.java   ← entry point (@SpringBootApplication)
│       │   ├── controller/
│       │   ├── service/
│       │   ├── repository/
│       │   └── model/
│       └── resources/
│           ├── application.yml             ← configurazione
│           └── application-dev.yml
├── pom.xml
└── target/
```

---

## Il progetto ASP.NET Core

```bash
dotnet new webapi -n OrderService
```

```
OrderService/
├── Controllers/
│   └── WeatherForecastController.cs    ← controller di esempio (da eliminare)
├── Properties/
│   └── launchSettings.json             ← porte, profili di avvio (equivalente di application-dev.yml per il dev)
├── appsettings.json                    ← configurazione (equivalente di application.yml)
├── appsettings.Development.json        ← override per l'ambiente di sviluppo
├── OrderService.csproj                 ← dipendenze e configurazione build (equivalente di pom.xml)
├── OrderService.http                   ← file per testare gli endpoint con REST Client
└── Program.cs                          ← entry point — tutto parte da qui
```

---

## La struttura che userai davvero

Il template genera un progetto flat — va bene per esempi, non per un microservizio reale. La struttura che adotteremo dal capitolo 4 in poi è multi-progetto:

```
OrderService.sln
├── OrderService.Api/               ← Web API, entry point, controller/endpoint
│   ├── Controllers/ (o Endpoints/)
│   ├── appsettings.json
│   ├── Program.cs
│   └── OrderService.Api.csproj
│
├── OrderService.Core/              ← dominio: entità, interfacce, logica di business
│   ├── Entities/
│   ├── Interfaces/
│   ├── Services/
│   └── OrderService.Core.csproj
│
├── OrderService.Infrastructure/    ← implementazioni: EF Core, HTTP client, messaggistica
│   ├── Persistence/
│   ├── Repositories/
│   └── OrderService.Infrastructure.csproj
│
└── OrderService.Tests/             ← test unitari e di integrazione
    └── OrderService.Tests.csproj
```

Le dipendenze scorrono in una direzione sola:

```
Api  →  Infrastructure  →  Core
Tests  →  (tutti)
```

`Core` non dipende da nulla di esterno — è il cuore del dominio, indipendente da framework e database. Questo è il principio della **Clean Architecture** che guiderà tutto il corso.


N.B. Esistono sicuramente altre architetture, tutte valide a modo loro. 

La Onion, ad esempio, separa esplicitamente Domain (entità pure) da Application (use cases e orchestrazione), dove Clean Architecture tende a unirli in Core. 

La Vertical Slice invece non separa per layer tecnici — ogni feature contiene tutto il suo stack, dalla request al database. 

Negli esempi che seguiranno proponiamo la Clean Architecture perché è un ottimo punto di partenza per capire la separazione delle responsabilità — e la struttura che riconoscerai spesso nei progetti su cui lavorerai.

## Confronto con Java

| Spring Boot | ASP.NET Core |
|---|---|
| `MyServiceApplication.java` | `Program.cs` |
| `application.yml` | `appsettings.json` |
| `application-dev.yml` | `appsettings.Development.json` |
| `pom.xml` | `OrderService.csproj` |
| `target/` | `bin/` e `obj/` |
| Struttura `src/main/java/...` | Struttura piatta — nessuna convenzione di cartelle forzata |
| Multi-module Maven | Solution multi-progetto (`.sln`) |

La differenza più evidente: Spring Boot impone una struttura di cartelle precisa (`src/main/java`, `src/main/resources`). ASP.NET Core non ha convenzioni forzate sulle cartelle — sei libero di organizzarle come vuoi. Libertà totale, ma anche responsabilità totale.
