# 3.4 Dependency Injection nativa

> **Analogia Java**: equivalente del container Spring IoC

---

## Introduzione

ASP.NET Core ha un container DI integrato — non serve aggiungere nulla. Il principio è identico a Spring: dichiari le dipendenze nel costruttore, il framework le risolve. La differenza principale è che in ASP.NET Core la registrazione è **esplicita** — non ci sono annotation magiche come `@Component` o `@Service` che fanno la scansione automatica del classpath.

---

## Come si faceva in Java

```java
// Spring — registrazione implicita tramite annotation
@Service
public class OrdineService {
    private final OrdineRepository repository;

    @Autowired  // o costruttore injection senza annotation in Spring moderno
    public OrdineService(OrdineRepository repository) {
        this.repository = repository;
    }
}

@Repository
public class OrdineRepositoryImpl implements OrdineRepository { }

// Spring trova e registra tutto automaticamente con il component scan
```

---

## Registrazione dei servizi in ASP.NET Core

```csharp
// Program.cs
builder.Services.AddScoped<IOrdineService, OrdineService>();
builder.Services.AddScoped<IOrdineRepository, OrdineRepository>();
builder.Services.AddSingleton<ICache, MemoryCache>();
builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();
```

```csharp
// OrdineService.cs — nessuna annotation, solo costruttore
public class OrdineService : IOrdineService
{
    private readonly IOrdineRepository _repository;
    private readonly ILogger<OrdineService> _logger;

    public OrdineService(IOrdineRepository repository, ILogger<OrdineService> logger)
    {
        _repository = repository;
        _logger = logger;
    }

    public async Task<IEnumerable<OrdineDto>> GetAllAsync()
    {
        _logger.LogInformation("Recupero tutti gli ordini");
        var ordini = await _repository.GetAllAsync();
        return ordini.Select(o => new OrdineDto(o.Id, o.Cliente, o.Totale));
    }
}
```

---

## Spiegazione

**I tre cicli di vita** — equivalenti diretti di Spring:

| ASP.NET Core | Spring | Comportamento |
|---|---|---|
| `AddScoped` | `@RequestScope` / default `@Bean` in Spring MVC | Una istanza per richiesta HTTP |
| `AddSingleton` | `@Singleton` / `@Bean` senza scope | Una istanza per tutta la vita dell'applicazione |
| `AddTransient` | `@Prototype` | Una nuova istanza ogni volta che viene richiesta |

**`AddScoped`** è quello che userai il 90% delle volte per service e repository — una istanza per richiesta HTTP è il comportamento giusto per quasi tutto.

**Registrazione esplicita vs component scan** — in Spring basta mettere `@Service` sulla classe e Spring la trova. In ASP.NET Core devi registrarla esplicitamente in `Program.cs`. È più verboso, ma più esplicito: aprendo `Program.cs` sai esattamente cosa è registrato nel container.

**`ILogger<T>`** — è già registrato automaticamente dal framework, non devi aggiungerlo. Puoi iniettarlo in qualsiasi classe senza configurazione aggiuntiva.

## Injection: costruttore, property e method

In ASP.NET Core l'injection avviene quasi sempre tramite **costruttore** — è l'unico modo supportato nativamente dal container built-in. Non esiste l'equivalente di `@Autowired` su un campo o su una property.

```csharp
// ✅ costruttore — il modo corretto
public class OrdineService : IOrdineService
{
    private readonly IOrdineRepository _repository;

    public OrdineService(IOrdineRepository repository)
    {
        _repository = repository;
    }
}

// ✅ primary constructor (C# 12) — ancora più compatto
public class OrdineService(IOrdineRepository repository) : IOrdineService
{
    public async Task<OrdineDto?> GetByIdAsync(int id)
    {
        var ordine = await repository.GetByIdAsync(id);
        return ordine is null ? null : new OrdineDto(ordine.Id, ordine.Cliente, ordine.Totale);
    }
}
```

Il **primary constructor** di C# 12 è la forma più moderna — il parametro è disponibile in tutta la classe senza dichiarare un campo esplicitamente.

## Registrare più implementazioni della stessa interfaccia

```csharp
// Registra più implementazioni
builder.Services.AddScoped<INotificatore, EmailNotificatore>();
builder.Services.AddScoped<INotificatore, SmsNotificatore>();
builder.Services.AddScoped<INotificatore, PushNotificatore>();

// Inietta tutte le implementazioni come IEnumerable
public class NotificatoreComposito
{
    private readonly IEnumerable<INotificatore> _notificatori;

    public NotificatoreComposito(IEnumerable<INotificatore> notificatori)
    {
        _notificatori = notificatori;
    }

    public async Task NotificaTuttiAsync(string msg)
    {
        foreach (var n in _notificatori)
            await n.InviaAsync(msg);
    }
}
```

Equivalente di iniettare una `List<INotificatore>` in Spring con tutti i bean che implementano l'interfaccia.

## Keyed services — C# 8 / .NET 8

Da .NET 8 puoi registrare e risolvere servizi con una chiave, senza creare factory manuali:

```csharp
// Registrazione con chiave
builder.Services.AddKeyedScoped<INotificatore, EmailNotificatore>("email");
builder.Services.AddKeyedScoped<INotificatore, SmsNotificatore>("sms");

// Iniezione con chiave
public class OrdineService
{
    private readonly INotificatore _emailNotificatore;

    public OrdineService([FromKeyedServices("email")] INotificatore notificatore)
    {
        _emailNotificatore = notificatore;
    }
}
```

## Registrazione con factory

Quando la costruzione di un servizio richiede logica:

```csharp
builder.Services.AddScoped<IOrdineService>(serviceProvider =>
{
    var repo = serviceProvider.GetRequiredService<IOrdineRepository>();
    var logger = serviceProvider.GetRequiredService<ILogger<OrdineService>>();
    var config = serviceProvider.GetRequiredService<IOptions<OrdineOptions>>();

    return new OrdineService(repo, logger, config.Value);
});
```

Equivalente del metodo `@Bean` in Spring con parametri espliciti.

## Organizzare le registrazioni con extension methods

In un microservizio reale, registrare tutto in `Program.cs` diventa rapidamente caotico. Il pattern più diffuso in .NET è spostare le registrazioni in **extension methods** su `IServiceCollection`, raggruppate per area funzionale, e richiamarle in `Program.cs` con una riga sola.

```csharp
// Infrastructure/DependencyInjection.cs
public static class InfrastructureServiceCollectionExtensions
{
    public static IServiceCollection AddInfrastructure(
        this IServiceCollection services,
        IConfiguration configuration)
    {
        services.AddScoped<IOrdineRepository, OrdineRepository>();
        services.AddScoped<IClienteRepository, ClienteRepository>();
        services.AddDbContext<AppDbContext>(options =>
            options.UseNpgsql(configuration.GetConnectionString("Default")));

        return services;
    }
}

// Core/DependencyInjection.cs
public static class CoreServiceCollectionExtensions
{
    public static IServiceCollection AddCore(this IServiceCollection services)
    {
        services.AddScoped<IOrdineService, OrdineService>();
        services.AddScoped<IClienteService, ClienteService>();

        return services;
    }
}
```

```csharp
// Program.cs — pulito e leggibile
builder.Services
    .AddCore()
    .AddInfrastructure(builder.Configuration);
```

È lo stesso pattern che usa ASP.NET Core internamente — `AddControllers()`, `AddSwaggerGen()`, `AddAuthentication()` sono tutti extension methods su `IServiceCollection`. Adottarlo fin da subito mantiene `Program.cs` snello e le registrazioni vicine al codice che le riguarda.

## Quando il container built-in non basta
N.B. Il container DI nativo di ASP.NET Core è intenzionalmente semplice — copre il 95% dei casi. Se ti serve qualcosa di più avanzato (decoratori, interceptor, lazy injection, convention-based registration) puoi sostituirlo con **Autofac** o **Lamar**, che si integrano in `Program.cs` con poche righe.

Per il corso useremo il container nativo — è più che sufficiente per un microservizio.

## Confronto con Java

| Spring | ASP.NET Core |
|---|---|
| `@Component`, `@Service`, `@Repository` | `AddScoped<I, T>()` in `Program.cs` |
| `@Autowired` (costruttore) | Costruttore normale — nessuna annotation |
| `@Singleton` | `AddSingleton<I, T>()` |
| `@RequestScope` | `AddScoped<I, T>()` |
| `@Prototype` | `AddTransient<I, T>()` |
| `@Bean` con logica | Factory lambda in `AddScoped` |
| `@Qualifier("nome")` | `[FromKeyedServices("nome")]` (.NET 8+) |
| `List<INotificatore>` iniettata | `IEnumerable<INotificatore>` iniettata |
| Component scan automatico | Registrazione esplicita in `Program.cs` |
