# 3.5 Middleware pipeline

> **Analogia Java**: equivalente dei Filter e Interceptor di Spring

---

## Introduzione

Ogni richiesta HTTP che arriva ad ASP.NET Core attraversa una **pipeline di middleware** — una catena di componenti che si eseguono in sequenza, ognuno con la possibilità di fare qualcosa prima e dopo il componente successivo. Logging, autenticazione, gestione degli errori, CORS, compressione: tutto passa di qui.

---

## Come si faceva in Java

In Spring hai due meccanismi principali per intercettare le richieste:

```java
// Filter — livello Servlet, intercetta tutto
@Component
public class LoggingFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        // prima
        System.out.println("Richiesta in arrivo: " + ((HttpServletRequest) req).getRequestURI());
        chain.doFilter(req, res);
        // dopo
        System.out.println("Risposta inviata");
    }
}

// HandlerInterceptor — livello Spring MVC, solo per richieste gestite dai controller
@Component
public class AuthInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse res, Object handler) {
        // false = blocca la richiesta
        return verificaToken(req);
    }
}
```

## La pipeline middleware in ASP.NET Core

```csharp
// Program.cs — l'ordine conta
var app = builder.Build();

app.UseExceptionHandler("/error");   // deve essere il primo — cattura tutto
app.UseHttpsRedirection();
app.UseCors();
app.UseAuthentication();             // prima di Authorization
app.UseAuthorization();
app.MapControllers();

app.Run();
```

## Scrivere un middleware custom

```csharp
// Middleware come classe
public class LoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<LoggingMiddleware> _logger;

    public LoggingMiddleware(RequestDelegate next, ILogger<LoggingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        _logger.LogInformation("→ {Method} {Path}", 
            context.Request.Method, 
            context.Request.Path);

        var stopwatch = Stopwatch.StartNew();

        await _next(context); // chiama il middleware successivo nella pipeline

        stopwatch.Stop();
        _logger.LogInformation("← {StatusCode} in {Ms}ms",
            context.Response.StatusCode,
            stopwatch.ElapsedMilliseconds);
    }
}

// Registrazione in Program.cs
app.UseMiddleware<LoggingMiddleware>();
```

Per middleware semplici esiste anche la forma inline con `app.Use`:

```csharp
app.Use(async (context, next) =>
{
    // prima
    context.Response.Headers.Append("X-Request-Id", Guid.NewGuid().ToString());

    await next(context); // chiama il successivo

    // dopo
    if (context.Response.StatusCode == 401)
        context.Response.Headers.Append("WWW-Authenticate", "Bearer");
});
```

## Spiegazione

La pipeline funziona come una **cipolla**: ogni middleware avvolge il successivo. La richiesta entra dal middleware più esterno, scende fino al controller, poi risale attraverso gli stessi middleware in ordine inverso.

```
Richiesta →  ExceptionHandler
          →  HttpsRedirection
          →  Cors
          →  Authentication
          →  Authorization
          →  Controller
          ←  Authorization
          ←  Authentication
          ←  Cors
          ←  HttpsRedirection  
          ←  ExceptionHandler
← Risposta
```

**`_next(context)`** — è il cuore del pattern. Chiamarlo significa "passa la richiesta al middleware successivo". Non chiamarlo significa cortocircuitare la pipeline — la risposta viene generata qui e non procede oltre. Utile per auth middleware che bloccano richieste non autorizzate.

**L'ordine è fondamentale** — `UseAuthentication` deve venire prima di `UseAuthorization`, altrimenti l'identità dell'utente non è ancora stata stabilita quando si controlla se ha i permessi. `UseExceptionHandler` deve essere il primo per catturare le eccezioni di tutti i middleware successivi.

## `app.Run` vs `app.Use` vs `app.Map`

```csharp
// Use — aggiunge un middleware che passa al successivo
app.Use(async (ctx, next) =>
{
    // ...
    await next(ctx);
});

// Run — middleware terminale, non chiama il successivo
app.Run(async ctx =>
{
    await ctx.Response.WriteAsync("Fine della pipeline");
});

// Map — branch della pipeline in base al path
app.Map("/health", healthApp =>
{
    healthApp.Run(async ctx =>
        await ctx.Response.WriteAsync("OK"));
});
```

## Confronto con Java

| Spring | ASP.NET Core |
|---|---|
| `Filter` (Servlet) | Middleware con `app.UseMiddleware<T>()` |
| `HandlerInterceptor` (Spring MVC) | Middleware o Action Filter |
| `chain.doFilter(req, res)` | `await _next(context)` |
| `@Order(1)` per l'ordine | Ordine di registrazione in `Program.cs` |
| `FilterRegistrationBean` | `app.UseMiddleware<T>()` |
| `OncePerRequestFilter` | Middleware — già eseguito una volta per richiesta |

La differenza principale: in Spring hai due livelli separati (Filter e Interceptor) con scopi diversi. In ASP.NET Core c'è un solo meccanismo — il middleware — che copre entrambi i casi. Più semplice da capire, meno cose da ricordare.
