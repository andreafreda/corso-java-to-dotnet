# 3.2 `Program.cs` e il nuovo Minimal API style

> **Analogia Java**: equivalente di `@SpringBootApplication` + `main` + `WebMvcConfigurer` — tutto in un unico file

---

## Introduzione

`Program.cs` è il cuore dell'applicazione ASP.NET Core: è qui che si avvia il web server, si registrano i servizi nel container DI, si configura la pipeline delle richieste HTTP. In Spring Boot queste responsabilità sono distribuite tra `@SpringBootApplication`, `@Configuration`, `@Bean` e vari `*Configurer`. In ASP.NET Core stanno tutte qui, in un unico posto.

---

## Come si faceva in Java

```java
// Entry point
@SpringBootApplication
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}

// Configurazione dei bean e dei servizi
@Configuration
public class AppConfig {
    @Bean
    public OrdineService ordineService(OrdineRepository repo) {
        return new OrdineServiceImpl(repo);
    }
}

// Configurazione del web layer
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**").allowedOrigins("*");
    }
}
```

Tre file, tre classi, tre annotazioni diverse per fare cose che concettualmente appartengono allo stesso posto.

## `Program.cs` in ASP.NET Core

```csharp
var builder = WebApplication.CreateBuilder(args);

// 1. Registrazione dei servizi nel container DI
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

builder.Services.AddScoped<IOrdineService, OrdineService>();
builder.Services.AddScoped<IOrdineRepository, OrdineRepository>();

builder.Services.AddCors(options =>
{
    options.AddDefaultPolicy(policy =>
        policy.AllowAnyOrigin().AllowAnyMethod().AllowAnyHeader());
});

// 2. Build dell'applicazione
var app = builder.Build();

// 3. Configurazione della pipeline HTTP (middleware)
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseCors();
app.UseAuthorization();
app.MapControllers();

// 4. Avvio
app.Run();
```

---

## Spiegazione

Il file si divide in due fasi nette, separate dal `builder.Build()`:

**Fase 1 — `builder`**: configuri tutto ciò che deve esistere prima che l'applicazione parta. `builder.Services` è il container DI — qui registri i tuoi servizi, i repository, le opzioni di configurazione. È l'equivalente del contesto Spring: tutto ciò che aggiungi qui sarà disponibile per l'injection.

**Fase 2 — `app`**: configuri la pipeline delle richieste HTTP. L'ordine conta — ogni `Use*` aggiunge un middleware che viene eseguito in sequenza per ogni richiesta in arrivo. `MapControllers()` dice ad ASP.NET Core dove trovare i controller.

`app.Environment.IsDevelopment()` legge la variabile d'ambiente `ASPNETCORE_ENVIRONMENT` — equivalente dei profili Spring (`spring.profiles.active`).

## Confronto con Java

| Spring Boot | ASP.NET Core |
|---|---|
| `@SpringBootApplication` | `WebApplication.CreateBuilder(args)` |
| `@Bean` in `@Configuration` | `builder.Services.Add*()` |
| `@Component`, `@Service`, `@Repository` | `AddScoped`, `AddSingleton`, `AddTransient` |
| `WebMvcConfigurer` | Middleware con `app.Use*()` |
| `spring.profiles.active=dev` | `ASPNETCORE_ENVIRONMENT=Development` |
| `@RestController` + metodi | `MapGet`, `MapPost`… (Minimal API) |

La differenza filosofica: Spring Boot è dichiarativo — annoti le classi e il framework le trova. ASP.NET Core è imperativo — registri esplicitamente tutto in `Program.cs`. Più verboso all'inizio, ma più esplicito e più facile da seguire quando il progetto cresce.
