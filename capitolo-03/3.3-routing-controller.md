# 3.3 Routing e Controller

> **Analogia Java**: equivalente di `@RestController` e `@RequestMapping`

---

## Introduzione

I controller in ASP.NET Core sono il punto di ingresso delle richieste HTTP â€” esattamente come in Spring. Ricevono la richiesta, delegano la logica al service layer, restituiscono una risposta. La struttura Ă¨ talmente simile a Spring MVC che dopo i primi dieci minuti ti sentirai a casa.

---

## Come si faceva in Java

```java
@RestController
@RequestMapping("/ordini")
public class OrdineController {

    private final OrdineService ordineService;

    public OrdineController(OrdineService ordineService) {
        this.ordineService = ordineService;
    }

    @GetMapping
    public List<OrdineDto> getAll() {
        return ordineService.getAll();
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrdineDto> getById(@PathVariable int id) {
        return ordineService.getById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }

    @PostMapping
    public ResponseEntity<OrdineDto> crea(@RequestBody @Valid CreaOrdineRequest request) {
        var ordine = ordineService.crea(request);
        return ResponseEntity.created(URI.create("/ordini/" + ordine.getId())).body(ordine);
    }

    @PutMapping("/{id}")
    public ResponseEntity<OrdineDto> aggiorna(
            @PathVariable int id,
            @RequestBody @Valid AggiornaOrdineRequest request) {
        return ordineService.aggiorna(id, request)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> elimina(@PathVariable int id) {
        ordineService.elimina(id);
        return ResponseEntity.noContent().build();
    }
}
```

## Il controller in ASP.NET Core

```csharp
[ApiController]
[Route("ordini")]
public class OrdiniController : ControllerBase
{
    private readonly IOrdineService _ordineService;

    public OrdiniController(IOrdineService ordineService)
    {
        _ordineService = ordineService;
    }

    [HttpGet]
    public async Task<IActionResult> GetAll()
    {
        var ordini = await _ordineService.GetAllAsync();
        return Ok(ordini);
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> GetById(int id)
    {
        var ordine = await _ordineService.GetByIdAsync(id);
        return ordine is null ? NotFound() : Ok(ordine);
    }

    [HttpPost]
    public async Task<IActionResult> Crea([FromBody] CreaOrdineRequest request)
    {
        var ordine = await _ordineService.CreaAsync(request);
        return CreatedAtAction(nameof(GetById), new { id = ordine.Id }, ordine);
    }

    [HttpPut("{id}")]
    public async Task<IActionResult> Aggiorna(int id, [FromBody] AggiornaOrdineRequest request)
    {
        var ordine = await _ordineService.AggiornaAsync(id, request);
        return ordine is null ? NotFound() : Ok(ordine);
    }

    [HttpDelete("{id}")]
    public async Task<IActionResult> Elimina(int id)
    {
        await _ordineService.EliminaAsync(id);
        return NoContent();
    }
}
```

## Spiegazione

**`[ApiController]`** â€” abilita una serie di comportamenti automatici: validazione automatica del model con risposta 400 se fallisce, inferenza delle sorgenti dei parametri (`[FromBody]`, `[FromRoute]` spesso non servono), gestione automatica dei binding errors. Mettilo sempre sui controller REST.

**`[Route("ordini")]`** â€” definisce il prefisso del route. Equivalente di `@RequestMapping("/ordini")`. Puoi usare `[Route("api/[controller]")]` per usare il nome della classe come segmento automatico â€” `OrdiniController` diventa `/api/ordini`.

**`: ControllerBase`** â€” la classe base per i controller API. Non estendere `Controller` (che aggiunge il supporto alle Razor Views) â€” per le API REST usa sempre `ControllerBase`.

**`IActionResult`** â€” il tipo di ritorno generico che incapsula stato HTTP + body. I metodi helper `Ok()`, `NotFound()`, `CreatedAtAction()`, `NoContent()` costruiscono le risposte con i codici HTTP corretti.

**`CreatedAtAction`** â€” genera la risposta 201 con l'header `Location` che punta al nuovo resource. Equivalente di `ResponseEntity.created(URI.create(...))` in Spring.

## Routing: attributi e convenzioni

Il routing funziona solo tramite attributi (`[HttpGet]`, `[HttpPost]`â€¦) â€” non esistono convenzioni implicite come in Spring MVC con `@RequestMapping` sul metodo.

```csharp
// Route parametrica
[HttpGet("{id}")]                        // GET /ordini/42

// Route con constraint di tipo
[HttpGet("{id:int}")]                    // GET /ordini/42 (solo interi)
[HttpGet("{slug:alpha}")]                // GET /ordini/abc (solo lettere)

// Route con query string â€” non serve attributo, viene letta automaticamente
[HttpGet]                                // GET /ordini?pagina=2&dimensione=10
public async Task<IActionResult> GetAll(int pagina = 1, int dimensione = 20) { }

// Route multipli sullo stesso metodo
[HttpGet("attivi")]
[HttpGet("in-corso")]                    // entrambi puntano allo stesso metodo
public async Task<IActionResult> GetAttivi() { }
```

## Sorgenti dei parametri

Con `[ApiController]` abilitato, ASP.NET Core inferisce automaticamente la sorgente dei parametri:

| Sorgente | Attributo esplicito | Inferenza automatica |
|---|---|---|
| Route (`/ordini/{id}`) | `[FromRoute]` | âś… se il nome corrisponde |
| Query string (`?pagina=1`) | `[FromQuery]` | âś… per tipi semplici |
| Body JSON | `[FromBody]` | âś… per tipi complessi |
| Header | `[FromHeader]` | âťŚ sempre esplicito |
| Form | `[FromForm]` | âťŚ sempre esplicito |

```csharp
// Tutto inferito automaticamente con [ApiController]
[HttpGet("{id}")]
public async Task<IActionResult> GetById(int id) { }       // id da route

[HttpGet]
public async Task<IActionResult> GetAll(int pagina) { }    // pagina da query string

[HttpPost]
public async Task<IActionResult> Crea(CreaOrdineRequest r) { } // r da body
```

## Typed results â€” la via piĂą moderna

Con .NET 7+ puoi usare `TypedResults` invece di `IActionResult` per avere informazioni sul tipo di ritorno anche in Swagger, senza attributi aggiuntivi:

```csharp
[HttpGet("{id}")]
public async Task<Results<Ok<OrdineDto>, NotFound>> GetById(int id)
{
    var ordine = await _ordineService.GetByIdAsync(id);
    return ordine is null ? TypedResults.NotFound() : TypedResults.Ok(ordine);
}
```

Swagger capisce automaticamente che questo endpoint puĂ˛ restituire 200 con `OrdineDto` o 404 â€” senza scrivere `[ProducesResponseType]` a mano.

## Confronto con Java

| Spring MVC | ASP.NET Core |
|---|---|
| `@RestController` | `[ApiController]` + `: ControllerBase` |
| `@RequestMapping("/ordini")` | `[Route("ordini")]` |
| `@GetMapping("/{id}")` | `[HttpGet("{id}")]` |
| `@PathVariable int id` | `int id` (inferito da route) |
| `@RequestBody` | `[FromBody]` (o inferito con `[ApiController]`) |
| `@Valid` | Validazione automatica con `[ApiController]` |
| `ResponseEntity.ok(body)` | `Ok(body)` |
| `ResponseEntity.notFound().build()` | `NotFound()` |
| `ResponseEntity.created(uri).body(dto)` | `CreatedAtAction(...)` |
| `ResponseEntity.noContent().build()` | `NoContent()` |
