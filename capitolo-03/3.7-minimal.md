# 3.7 Minimal API e Carter: organizzazione modulare

> **Analogia Java**: nessun equivalente diretto in Spring — il più vicino è il pattern Feature Slice con `@RestController`

---

## Introduzione

Da .NET 6 esiste un secondo modo di esporre endpoint HTTP in ASP.NET Core: le **Minimal API**. Invece di creare classi controller, registri gli endpoint direttamente come lambda in `Program.cs`. È un approccio più compatto, pensato per microservizi semplici e scenari dove la cerimonia dei controller è eccessiva. Quando gli endpoint crescono, **Carter** risolve il problema dell'organizzazione.

---

## Come si faceva in Java

In Spring non esiste un equivalente diretto delle Minimal API — hai sempre bisogno di una classe `@RestController`. Il più vicino come filosofia sono i functional endpoints di Spring WebFlux:

```java
// Spring WebFlux — RouterFunction (il più simile alle Minimal API)
@Configuration
public class OrdineRouter {
    @Bean
    public RouterFunction<ServerResponse> route(OrdineHandler handler) {
        return RouterFunctions.route()
            .GET("/ordini", handler::getAll)
            .GET("/ordini/{id}", handler::getById)
            .POST("/ordini", handler::crea)
            .build();
    }
}
```

Ma è comunque più verboso e richiede classi dedicate. Le Minimal API di .NET sono più dirette.

---

## Minimal API — endpoint senza controller

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddScoped<IOrdineService, OrdineService>();

var app = builder.Build();

app.MapGet("/ordini", async (IOrdineService service) =>
    await service.GetAllAsync());

app.MapGet("/ordini/{id:int}", async (int id, IOrdineService service) =>
{
    var ordine = await service.GetByIdAsync(id);
    return ordine is null ? Results.NotFound() : Results.Ok(ordine);
});

app.MapPost("/ordini", async (CreaOrdineRequest request, IOrdineService service) =>
{
    var ordine = await service.CreaAsync(request);
    return Results.Created($"/ordini/{ordine.Id}", ordine);
});

app.Run();
```

---

## Spiegazione

I parametri del lambda vengono risolti automaticamente:
- **Tipi registrati nel container DI** (`IOrdineService`) → iniettati dal framework
- **Tipi semplici che corrispondono a segmenti di route** (`int id`) → letti dalla route
- **Tipi complessi** (`CreaOrdineRequest`) → deserializzati dal body JSON
- **`CancellationToken`** → iniettato automaticamente dalla richiesta

Non serve `[FromRoute]`, `[FromBody]`, `[FromQuery]` — ASP.NET Core deduce la sorgente dal tipo e dal contesto.

**Controller-based vs Minimal API — quale scegliere:**

| | Controller-based | Minimal API |
|---|---|---|
| Microservizi semplici | ✅ | ✅ preferibile |
| API complesse con molti endpoint | ✅ preferibile | Richiede Carter |
| Familiarità per chi viene da Spring | Alta | Bassa |
| Testabilità | Consolidata | Ottima con `WebApplicationFactory` |

Nel corso useremo i **controller** per la familiarità con Spring. Le Minimal API con Carter sono la direzione verso cui si sta muovendo l'ecosistema .NET — vale la pena conoscerle.

## Il problema quando gli endpoint crescono

```csharp
// Program.cs — diventa rapidamente ingestibile
app.MapGet("/ordini", ...);
app.MapGet("/ordini/{id}", ...);
app.MapPost("/ordini", ...);
app.MapGet("/clienti", ...);
app.MapGet("/clienti/{id}", ...);
// ... altri 30 endpoint
```

**Carter** risolve questo: ogni gruppo di endpoint vive nel proprio modulo, `Program.cs` torna ad avere due righe.

## Un modulo Carter

```csharp
// Modules/OrdiniModule.cs
public class OrdiniModule : ICarterModule
{
    public void AddRoutes(IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/ordini")
                       .WithTags("Ordini")
                       .RequireAuthorization();

        group.MapGet("/", GetAll);
        group.MapGet("/{id:int}", GetById);
        group.MapPost("/", Crea);
        group.MapPut("/{id:int}", Aggiorna);
        group.MapDelete("/{id:int}", Elimina);
    }

    private static async Task<IResult> GetAll(IOrdineService service)
        => Results.Ok(await service.GetAllAsync());

    private static async Task<IResult> GetById(int id, IOrdineService service)
    {
        var ordine = await service.GetByIdAsync(id);
        return ordine is null ? Results.NotFound() : Results.Ok(ordine);
    }

    private static async Task<IResult> Crea(
        CreaOrdineRequest request, IOrdineService service)
    {
        var ordine = await service.CreaAsync(request);
        return Results.Created($"/ordini/{ordine.Id}", ordine);
    }

    private static async Task<IResult> Aggiorna(
        int id, AggiornaOrdineRequest request, IOrdineService service)
    {
        var ordine = await service.AggiornaAsync(id, request);
        return ordine is null ? Results.NotFound() : Results.Ok(ordine);
    }

    private static async Task<IResult> Elimina(int id, IOrdineService service)
    {
        await service.EliminaAsync(id);
        return Results.NoContent();
    }
}
```

```csharp
// Program.cs — due righe
builder.Services.AddCarter();
app.MapCarter(); // trova e registra automaticamente tutti i moduli ICarterModule
```

## Confronto con Java

| Spring | ASP.NET Core |
|---|---|
| `@RestController` + `@GetMapping` | Minimal API con `MapGet` / Carter module |
| Nessun equivalente | `MapGroup` per prefisso e configurazione comune |
| `RouterFunction` (WebFlux) | Minimal API (concettualmente simile) |
| Classe per ogni controller | `ICarterModule` per ogni gruppo di endpoint |

---

## Controller o Minimal API: cosa usiamo nel corso?

Hai due strumenti validi. La scelta dipende dal contesto:

| | Controller | Minimal API / Carter |
|---|---|---|
| Struttura familiare da Spring MVC | ✅ | ❌ |
| Built-in, zero dipendenze extra | ✅ | ❌ (Carter è un package) |
| Ideale per API complesse e numerose | ✅ | ✅ |
| Sintassi più compatta | ❌ | ✅ |
| Adatto a microservizi piccoli | ✅ | ✅ |

Nel progetto pratico del capitolo 4 in poi useremo i **controller** — la struttura è immediata per chi viene da Spring MVC, il mapping concettuale è diretto, e non aggiunge dipendenze esterne. Le Minimal API con Carter restano un'ottima scelta che troverai spesso nei progetti .NET moderni: ora sai come funzionano e quando usarle.
