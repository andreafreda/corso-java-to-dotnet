# 9.7 Approccio TDD applicato al microservizio di esempio

> **Analogia Java**: pratica hands-on sull'intero progetto

---

## Introduzione

Il TDD non è "scrivi test dopo". È un ciclo di design: **Red → Green → Refactor**. Scrivi il test che descrive il comportamento che vuoi, vedi fallire, scrivi il codice minimo per farlo passare, poi pulisci. Questo paragrafo applica TDD su una nuova feature di `OrderService`: la conferma di un ordine con cambio di stato.

---

## Il ciclo Red → Green → Refactor

```
Red    → scrivi un test che fallisce (il codice non esiste ancora)
Green  → scrivi il codice minimo per farlo passare
Refactor → pulisci il codice senza rompere i test
```

---

## La feature: `ConfirmOrderAsync`

Vogliamo aggiungere la possibilità di confermare un ordine — da `Pending` a `Confirmed`. Regole:
1. L'ordine deve esistere
2. Solo gli ordini in stato `Pending` possono essere confermati
3. La conferma aggiorna lo stato e persiste

Iniziamo con i test.

---

## Step 1 — Red: scrivi i test

```csharp
// Unit/Services/OrderServiceConfirmTests.cs
using FluentAssertions;
using Moq;
using OrderService.Core.Entities;
using OrderService.Core.Interfaces.Repositories;
using OrderService.Core.Services;
using Xunit;

namespace OrderService.Tests.Unit.Services;

public class OrderServiceConfirmTests
{
    private readonly Mock<IOrderRepository> _repositoryMock = new();
    private readonly OrderService           _sut;

    public OrderServiceConfirmTests()
    {
        _sut = new OrderService(_repositoryMock.Object, Mock.Of<ILogger<OrderService>>());
    }

    // Test 1 — ordine non trovato
    [Fact]
    public async Task ConfirmAsync_OrderNotFound_ThrowsNotFoundException()
    {
        _repositoryMock
            .Setup(r => r.GetByIdAsync(99, It.IsAny<CancellationToken>()))
            .ReturnsAsync((Order?)null);

        var act = () => _sut.ConfirmAsync(99);

        await act.Should().ThrowAsync<KeyNotFoundException>()
            .WithMessage("*99*");
    }

    // Test 2 — ordine già confermato
    [Fact]
    public async Task ConfirmAsync_AlreadyConfirmed_ThrowsInvalidOperationException()
    {
        _repositoryMock
            .Setup(r => r.GetByIdAsync(1, It.IsAny<CancellationToken>()))
            .ReturnsAsync(new Order { Id = 1, Status = OrderStatus.Confirmed });

        var act = () => _sut.ConfirmAsync(1);

        await act.Should().ThrowAsync<InvalidOperationException>()
            .WithMessage("*già confermato*");
    }

    // Test 3 — conferma valida
    [Fact]
    public async Task ConfirmAsync_PendingOrder_ChangesStatusToConfirmed()
    {
        var order = new Order { Id = 1, Customer = "Mario", Total = 100m, Status = OrderStatus.Pending };

        _repositoryMock
            .Setup(r => r.GetByIdAsync(1, It.IsAny<CancellationToken>()))
            .ReturnsAsync(order);
        _repositoryMock
            .Setup(r => r.UpdateAsync(It.IsAny<Order>(), It.IsAny<CancellationToken>()))
            .ReturnsAsync((Order o, CancellationToken _) => o);

        var result = await _sut.ConfirmAsync(1);

        result.Status.Should().Be(OrderStatus.Confirmed);
    }

    // Test 4 — verifica che l'update venga chiamato
    [Fact]
    public async Task ConfirmAsync_PendingOrder_CallsUpdateOnRepository()
    {
        var order = new Order { Id = 1, Status = OrderStatus.Pending };

        _repositoryMock.Setup(r => r.GetByIdAsync(1, It.IsAny<CancellationToken>())).ReturnsAsync(order);
        _repositoryMock.Setup(r => r.UpdateAsync(It.IsAny<Order>(), It.IsAny<CancellationToken>()))
                       .ReturnsAsync(order);

        await _sut.ConfirmAsync(1);

        _repositoryMock.Verify(
            r => r.UpdateAsync(
                It.Is<Order>(o => o.Status == OrderStatus.Confirmed),
                It.IsAny<CancellationToken>()),
            Times.Once);
    }
}
```

Esegui: tutti e quattro i test falliscono — `ConfirmAsync` non esiste ancora. ✓ **Red**.

```csharp
// Core/Services/OrderService.cs — aggiungi ConfirmAsync
public async Task<Order> ConfirmAsync(int id, CancellationToken ct = default)
{
    var order = await _repository.GetByIdAsync(id, ct)
        ?? throw new KeyNotFoundException($"Ordine {id} non trovato");

    if (order.Status != OrderStatus.Pending)
        throw new InvalidOperationException($"L'ordine {id} è già confermato o in uno stato non confermabile");

    order.Status = OrderStatus.Confirmed;
    return await _repository.UpdateAsync(order, ct);
}
```

Esegui: tutti e quattro i test passano. ✓ **Green**.

## Step 3 — Refactor: pulisci senza rompere

```csharp
// Il messaggio "già confermato" era troppo generico — aggiusta per leggibilità
if (order.Status != OrderStatus.Pending)
    throw new InvalidOperationException(
        $"Impossibile confermare l'ordine {id}: stato attuale '{order.Status}', atteso 'Pending'");
```

Aggiorna il test per riflettere il nuovo messaggio:

```csharp
await act.Should().ThrowAsync<InvalidOperationException>()
    .WithMessage("*stato attuale*");
```

Esegui: tutti i test passano ancora. ✓ **Refactor completato**.

## Step 4 — Aggiungi il test di integrazione

```csharp
// Integration/OrderConfirmApiTests.cs
public class OrderConfirmApiTests : IClassFixture<TestWebApplicationFactory>
{
    private readonly HttpClient _client;

    public OrderConfirmApiTests(TestWebApplicationFactory factory)
    {
        _client = factory.CreateClient();
        _client.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", TestTokenHelper.Generate("Admin"));
    }

    [Fact]
    public async Task ConfirmOrder_ExistingPendingOrder_Returns200()
    {
        // Crea un ordine
        var createResponse = await _client.PostAsJsonAsync("/orders",
            new { customer = "Mario Rossi", total = 150.00 });
        var created = await createResponse.Content.ReadFromJsonAsync<OrderResponse>();

        // Confermalo
        var confirmResponse = await _client.PostAsync($"/orders/{created!.Id}/confirm", null);

        confirmResponse.StatusCode.Should().Be(HttpStatusCode.OK);
        var confirmed = await confirmResponse.Content.ReadFromJsonAsync<OrderResponse>();
        confirmed!.Status.Should().Be("Confirmed");
    }

    [Fact]
    public async Task ConfirmOrder_NonExistingOrder_Returns404()
    {
        var response = await _client.PostAsync("/orders/9999/confirm", null);
        response.StatusCode.Should().Be(HttpStatusCode.NotFound);
    }
}
```

## Step 5 — Aggiungi l'endpoint al controller

```csharp
// Api/Controllers/OrderController.cs — aggiungi
[HttpPost("{id:int}/confirm")]
[Authorize(Policy = "WriteOrders")]
public async Task<IActionResult> Confirm(int id, CancellationToken ct)
{
    try
    {
        var order = await _orderService.ConfirmAsync(id, ct);
        return Ok(order);
    }
    catch (KeyNotFoundException)
    {
        return NotFound();
    }
    catch (InvalidOperationException ex)
    {
        return BadRequest(new ProblemDetails
        {
            Title  = "Operazione non valida",
            Detail = ex.Message,
            Status = 400
        });
    }
}
```

## Il valore del TDD applicato

In questo ciclo il TDD ha prodotto:

1. **Design forzato**: prima di scrivere il codice hai definito l'interfaccia (`ConfirmAsync`), il contratto (eccezioni attese) e il comportamento — il codice è emerso dai test, non viceversa.

2. **Regression coverage automatica**: se domani qualcuno rompe `ConfirmAsync`, i test lo rilevano immediatamente.

3. **Documentazione eseguibile**: i test descrivono il comportamento del sistema in modo più preciso di qualsiasi commento.

---

## Spiegazione

**Quando NON usare TDD** — TDD è ideale per logica business complessa e regole di dominio. Per glue code (controller, DI wiring, migrations) il costo supera il beneficio. Un approccio pragmatico: TDD per `Core`, test di integrazione per `Api`, Testcontainers per i repository.

**Test first non significa test only** — se scopri un bug in produzione, il flusso è: scrivi un test che riproduce il bug (Red), fixa il bug (Green), verifica che gli altri test passino ancora. Il test diventa la prova che il bug non ritornerà.

**`InMemoryOrderRepository` come fake nei test unitari** — invece di mockare `IOrderRepository` con Moq, puoi usare il fake del capitolo 4. I fake sono preferibili ai mock quando la logica di interazione è complessa — rendono i test più leggibili.
