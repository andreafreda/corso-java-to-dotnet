# 9.1 I framework di test in .NET: xUnit, NUnit e MSTest a confronto

> **Analogia Java**: equivalente di JUnit 5 — quale scegliere e perché

---

## Introduzione

In Java hai JUnit 5 come standard de facto, con TestNG come alternativa. In .NET hai tre framework principali: **xUnit**, **NUnit** e **MSTest**. Tutti e tre sono supportati da Visual Studio, VS Code e la CLI `dotnet test`. La differenza è nello stile delle API — non nelle capacità.

---

## Come si faceva in Java

```java
// JUnit 5 — il framework standard
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository repository;

    @InjectMocks
    private OrderService service;

    @Test
    void shouldCreateOrder() {
        // Arrange
        var request = new CreateOrderRequest("Mario Rossi", 150.00);
        when(repository.save(any())).thenReturn(new Order(1L, "Mario Rossi", 150.00));

        // Act
        var order = service.createOrder(request);

        // Assert
        assertThat(order.getId()).isEqualTo(1L);
        verify(repository).save(any());
    }

    @ParameterizedTest
    @ValueSource(doubles = {0, -1, -100})
    void shouldRejectInvalidTotal(double total) {
        assertThrows(ValidationException.class,
            () -> service.createOrder(new CreateOrderRequest("Mario", total)));
    }
}
```

## I tre framework a confronto

### xUnit — il più moderno, il più usato

```csharp
public class OrderServiceTests
{
    [Fact]                    // ← equivalente di @Test
    public void CreateOrder_ValidRequest_ReturnsOrder()
    {
        // ...
    }

    [Theory]                  // ← equivalente di @ParameterizedTest
    [InlineData(0)]
    [InlineData(-1)]
    [InlineData(-100)]
    public void CreateOrder_InvalidTotal_ThrowsException(decimal total)
    {
        // ...
    }
}
```

**Caratteristiche**: istanzia una nuova classe per ogni test (isolamento garantito), nessun `[SetUp]`/`[TearDown]` — usa il costruttore e `IDisposable`, integrazione nativa con `ITestOutputHelper` per output. È il default per i nuovi progetti .NET e il framework usato dal team ASP.NET Core.

### NUnit — il più simile a JUnit

```csharp
[TestFixture]                 // ← equivalente di @TestInstance
public class OrderServiceTests
{
    private OrderService _service;

    [SetUp]                   // ← equivalente di @BeforeEach
    public void Setup() => _service = new OrderService(/* ... */);

    [Test]                    // ← equivalente di @Test
    public void CreateOrder_ValidRequest_ReturnsOrder() { }

    [TestCase(0)]             // ← equivalente di @ValueSource
    [TestCase(-1)]
    [TestCase(-100)]
    public void CreateOrder_InvalidTotal_ThrowsException(decimal total) { }
}
```

**Caratteristiche**: API molto simile a JUnit 4/5, `[SetUp]`/`[TearDown]` espliciti, `[TestFixture]` come contenitore. Buona scelta se vieni da JUnit e vuoi una transizione più lineare.

### MSTest — il framework Microsoft

```csharp
[TestClass]
public class OrderServiceTests
{
    private OrderService _service;

    [TestInitialize]          // ← equivalente di @BeforeEach
    public void Init() => _service = new OrderService(/* ... */);

    [TestMethod]              // ← equivalente di @Test
    public void CreateOrder_ValidRequest_ReturnsOrder() { }

    [DataTestMethod]
    [DataRow(0)]
    [DataRow(-1)]
    public void CreateOrder_InvalidTotal_ThrowsException(decimal total) { }
}
```

**Caratteristiche**: integrato in Visual Studio, meno usato nei progetti open source, API più verbosa. Buona scelta se lavori in ambienti enterprise con strong tie a Visual Studio.

## La scelta per il corso: xUnit

Il capitolo 9 usa **xUnit** — è il framework di riferimento per ASP.NET Core, usato dal team Microsoft internamente, con il supporto migliore per i test di integrazione con `WebApplicationFactory`.

---

## Setup del progetto di test

```bash
# Crea il progetto di test
dotnet new xunit -n OrderService.Tests
dotnet sln add OrderService.Tests

# Referenzia i progetti da testare
dotnet add OrderService.Tests reference OrderService.Core
dotnet add OrderService.Tests reference OrderService.Infrastructure
dotnet add OrderService.Tests reference OrderService.Api

# Package utili
dotnet add OrderService.Tests package Moq
dotnet add OrderService.Tests package FluentAssertions
```

```
OrderService/
├── OrderService.sln
├── OrderService.Api/
├── OrderService.Core/
├── OrderService.Infrastructure/
└── OrderService.Tests/          ← nuovo
    ├── OrderService.Tests.csproj
    ├── Unit/
    ├── Integration/
    └── Fixtures/
```

## Eseguire i test

```bash
dotnet test

# Con output verboso
dotnet test --verbosity normal

# Filtra per nome
dotnet test --filter "CreateOrder"

# Con coverage
dotnet test --collect:"XPlat Code Coverage"
```

## Confronto con Java

| JUnit 5 | xUnit | NUnit | MSTest |
|---|---|---|---|
| `@Test` | `[Fact]` | `[Test]` | `[TestMethod]` |
| `@ParameterizedTest` + `@ValueSource` | `[Theory]` + `[InlineData]` | `[TestCase]` | `[DataTestMethod]` + `[DataRow]` |
| `@BeforeEach` | Costruttore | `[SetUp]` | `[TestInitialize]` |
| `@AfterEach` | `IDisposable.Dispose` | `[TearDown]` | `[TestCleanup]` |
| `@BeforeAll` | `IClassFixture<T>` | `[OneTimeSetUp]` | `[ClassInitialize]` |
| `@ExtendWith` | `IClassFixture<T>` | `[TestFixture]` attributes | `[TestClass]` |
| `assertThat(x).isEqualTo(y)` (AssertJ) | `x.Should().Be(y)` (FluentAssertions) | `Assert.That(x, Is.EqualTo(y))` | `Assert.AreEqual(y, x)` |
