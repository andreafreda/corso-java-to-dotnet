# 4.5 OpenAPI, Scalar e file `.http`: testa subito la tua API

---

## Introduzione

Il microservizio gira. Prima di aggiungere altri layer, fermiamoci a testarlo — è il momento giusto, finché il codice è ancora semplice e il feedback è immediato. Due strumenti: **Scalar** per esplorare e provare gli endpoint dal browser, e il **file `.http`** per i test ripetibili che puoi committare nel repository.

## Scalar in ASP.NET Core

> 📝 **Code Step 3** — Installa Scalar, aggiorna `Program.cs` e crea `OrderService.http`

In .NET 10 il template `webapi` include già `Microsoft.AspNetCore.OpenApi` che espone la specifica OpenAPI in formato JSON — ma non ha un'interfaccia interattiva. Per quella hai due opzioni principali:

- **Swashbuckle.AspNetCore** — il package classico che porta Swagger UI. Lo conosci se hai lavorato con Spring Boot + SpringDoc. È ancora usato moltissimo, ma il suo mantenimento ha rallentato e con .NET 9/10 richiede qualche configurazione extra.
- **Scalar.AspNetCore** — nata esattamente per .NET 9/10, si integra in due righe, UI più moderna.

Nel corso usiamo **Scalar** — è la scelta più naturale su .NET 10 e non richiede workaround. Se nel tuo progetto hai già Swashbuckle, continua pure a usarlo: il concetto è identico.

```bash
dotnet add OrderService.Api package Scalar.AspNetCore
```

```csharp
// Program.cs
using Scalar.AspNetCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddOpenApi();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();
    app.MapScalarApiReference();
}

app.MapControllers();
app.Run();
```

```bash
dotnet run --project OrderService.Api
# http://localhost:5000/scalar  →  Scalar UI
```

Scalar mostra tutti gli endpoint, i tipi di request e response, e ti permette di eseguire chiamate direttamente dal browser — senza Postman, senza curl.

## Il file `.http`

Il file `.http` ti permette di scrivere le chiamate una volta e rieseguirle sempre, direttamente da VS Code con l'estensione REST Client. Si committa nel repository — il tuo team ha subito un modo per testare gli endpoint.

```
OrderService.Api/
└── OrderService.http
```

```http
@baseUrl = http://localhost:5000

### Ottieni tutti gli ordini
GET {{baseUrl}}/orders
Accept: application/json

### Ottieni un ordine specifico
GET {{baseUrl}}/orders/1
Accept: application/json

### Ordine non trovato
GET {{baseUrl}}/orders/99
Accept: application/json

### Crea un nuovo ordine
POST {{baseUrl}}/orders
Content-Type: application/json

{
  "customer": "Andrea Fred",
  "total": 99.99
}
```

In VS Code con l'estensione REST Client installata, sopra ogni `###` appare il link **Send Request** — clicca e vedi la risposta inline, accanto al codice.

## Spiegazione

**`app.MapOpenApi()`** — espone la specifica OpenAPI in formato JSON su `/openapi/v1.json`. È il documento che Scalar legge per generare l'interfaccia.

**`MapScalarApiReference` solo in Development** — in produzione non vuoi esporre la documentazione dell'API pubblicamente. Il blocco `if (app.Environment.IsDevelopment())` garantisce che Scalar sia attivo solo in locale.

**`@baseUrl` nel file `.http`** — le variabili nel file `.http` ti permettono di cambiare l'URL base in un solo posto. Quando passi da locale a staging basta cambiare quella riga.

**`.http` vs Postman** — il file `.http` è testo, si committa, non richiede account, funziona offline. Postman ha più funzionalità (environment multipli, test automatici, mock server) ed è più comodo per API complesse. Per il corso usiamo il file `.http` — zero setup, tutto nel repository.

## Confronto con Java

| Spring Boot | ASP.NET Core |
|---|---|
| `springdoc-openapi` (dipendenza esterna) | `Microsoft.AspNetCore.OpenApi` (incluso nel template) + `Scalar.AspNetCore` |
| `http://localhost:8080/swagger-ui.html` | `http://localhost:5000/scalar` |
| File `.http` con REST Client VS Code | Stesso — identico strumento |
| `@Operation`, `@Schema` per annotare | Attributi opzionali, Scalar funziona senza |
