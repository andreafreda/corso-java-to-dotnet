# 4.8 Wiring: Dependency Injection e Program.cs

---

## Introduzione

Il progetto compila e parte, ma la prima chiamata all'API crasha a runtime — il DI container non sa come risolvere `IOrderService` e `IOrderRepository` perché non ha ancora nessuna implementazione registrata. Questo paragrafo sistema entrambe le cose.

Implementiamo un repository in-memory in `Infrastructure` — una `List<Order>` che simula il database. È temporaneo e lo sappiamo: nel capitolo 5 lo sostituiamo con EF Core e PostgreSQL senza toccare nulla in `Core` o `Api`. È esattamente la prova che l'architettura funziona.

## Il repository in-memory in `Infrastructure`

> 📝 **Code Step 6** — Crea `Infrastructure/Repositories/InMemoryOrderRepository.cs`, `Core/DependencyInjection.cs`, `Infrastructure/DependencyInjection.cs` e aggiorna `Program.cs`

```
OrderService.Infrastructure/
└── Repositories/
    └── InMemoryOrderRepository.cs
```

```csharp
// Infrastructure/Repositories/InMemoryOrderRepository.cs
using OrderService.Core.Entities;
using OrderService.Core.Interfaces.Repositories;

namespace OrderService.Infrastructure.Repositories;

// Implementazione temporanea — verrà sostituita con EF Core nel capitolo 5
public class InMemoryOrderRepository : IOrderRepository
{
    private readonly List<Order> _store =
    [
        new() { Id = 1, Customer = "Mario Rossi",    Total = 150.00m, Status = OrderStatus.Confirmed },
        new() { Id = 2, Customer = "Giulia Bianchi", Total =  89.50m, Status = OrderStatus.Pending   },
    ];

    private int _nextId = 3;

    public Task<IEnumerable<Order>> GetAllAsync(CancellationToken ct = default) =>
        Task.FromResult<IEnumerable<Order>>(_store);

    public Task<Order?> GetByIdAsync(int id, CancellationToken ct = default) =>
        Task.FromResult(_store.FirstOrDefault(o => o.Id == id));

    public Task<Order> AddAsync(Order order, CancellationToken ct = default)
    {
        order.Id = _nextId++;
        _store.Add(order);
        return Task.FromResult(order);
    }
}
```

## Extension methods per la registrazione DI

Come visto in 3.4, il pattern idiomatico in .NET è raccogliere le registrazioni in extension methods — uno per layer — e richiamarli in `Program.cs`.

`IServiceCollection` viene da `Microsoft.Extensions.DependencyInjection.Abstractions` — non è incluso automaticamente in un `classlib`. Aggiungilo a `Core` e `Infrastructure`:

```bash
dotnet add OrderService.Core package Microsoft.Extensions.DependencyInjection.Abstractions
dotnet add OrderService.Infrastructure package Microsoft.Extensions.DependencyInjection.Abstractions
```

```csharp
// Core/DependencyInjection.cs
using Microsoft.Extensions.DependencyInjection;
using OrderService.Core.Interfaces.Services;
using OrderService.Core.Services;

namespace OrderService.Core;

public static class DependencyInjection
{
    public static IServiceCollection AddCore(this IServiceCollection services)
    {
        services.AddScoped<IOrderService, OrderService>();
        return services;
    }
}
```

```csharp
// Infrastructure/DependencyInjection.cs
using Microsoft.Extensions.DependencyInjection;
using OrderService.Core.Interfaces.Repositories;
using OrderService.Infrastructure.Repositories;

namespace OrderService.Infrastructure;

public static class DependencyInjection
{
    public static IServiceCollection AddInfrastructure(this IServiceCollection services)
    {
        // Nel capitolo 5 questa riga diventa: services.AddDbContext + AddScoped<IOrderRepository, OrderRepository>
        services.AddSingleton<IOrderRepository, InMemoryOrderRepository>();
        return services;
    }
}
```

## `Program.cs`

```csharp
// Api/Program.cs
using OrderService.Core;
using OrderService.Infrastructure;
using Scalar.AspNetCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddOpenApi();

builder.Services.AddCore();
builder.Services.AddInfrastructure();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();
    app.MapScalarApiReference();
}

app.MapControllers();
app.Run();
```

```bash
dotnet run --project OrderService.Api

# GET  http://localhost:5000/orders    →  200 [ {id:1, ...}, {id:2, ...} ]
# GET  http://localhost:5000/orders/1  →  200 { id:1, customer: "Mario Rossi", ... }
# GET  http://localhost:5000/orders/99 →  404
# POST http://localhost:5000/orders    →  201 { id:3, ... }
```

Il microservizio è funzionante end-to-end. L'architettura è quella definitiva — controller → service → repository. Nel capitolo 5 `InMemoryOrderRepository` diventa un repository EF Core reale, e nient'altro cambia.

## Spiegazione

**`AddSingleton` per il repository in-memory** — ha senso qui perché la `List<Order>` deve sopravvivere tra le richieste. Con EF Core useremo `AddScoped` — una istanza per richiesta HTTP, che è il comportamento corretto per un `DbContext`.

**`AddScoped` per il service** — ogni richiesta HTTP ottiene la propria istanza di `OrderService`. È il lifetime corretto per i service che dipendono da risorse scoped come il `DbContext`.

**Extension methods `AddCore` e `AddInfrastructure`** — `Program.cs` rimane pulito e leggibile. Ogni layer si auto-registra. Quando in cap 5 aggiungiamo EF Core, cambia solo `AddInfrastructure` — `Program.cs` non si tocca.

## Confronto con Java

| Spring Boot | ASP.NET Core |
|---|---|
| `@Service`, `@Repository` — annotazioni | Registrazione esplicita nel DI container |
| Nessun `main` da modificare per aggiungere bean | `Program.cs` + extension methods per layer |
| `@ComponentScan` trova tutto automaticamente | Registrazione esplicita — sai sempre cosa è registrato |
| `@Scope("prototype")` / `@Scope("singleton")` | `AddTransient` / `AddScoped` / `AddSingleton` |
