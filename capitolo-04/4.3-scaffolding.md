# 4.3 `dotnet new webapi` e scaffolding della struttura multi-progetto

---

## Introduzione

In 4.2 abbiamo visto la struttura che vogliamo ottenere e capito perché. Adesso la costruiamo — un comando alla volta. Se hai mai usato `mvn archetype:generate` o Spring Initializr da terminale, il processo ti sembrerà familiare. Se non l'hai mai fatto, è comunque più semplice di quanto sembri.

---

## Come si faceva in Java

```bash
# Opzione 1 — Spring Initializr da curl
curl https://start.spring.io/starter.zip \
  -d type=maven-project \
  -d dependencies=web \
  -d groupId=com.example \
  -d artifactId=order-service \
  -o order-service.zip && unzip order-service.zip

# Opzione 2 — Maven archetype
mvn archetype:generate \
  -DgroupId=com.example \
  -DartifactId=order-service \
  -DarchetypeArtifactId=maven-archetype-quickstart

# Per il multi-module, poi si modifica manualmente il pom.xml padre
# aggiungendo <modules> e <packaging>pom</packaging>
```

Creare un multi-module Maven a mano richiede di editare il `pom.xml` padre e i moduli figlio — non c'è un comando unico. Con .NET bastano pochi comandi CLI.

---

## Scaffolding con `dotnet` CLI

> 📝 **Code Step 1** — Crea la solution e i tre progetti

```bash
# 1. Crea la cartella radice e la solution
mkdir OrderService && cd OrderService
dotnet new sln -n OrderService

# 2. Crea i tre progetti
dotnet new webapi   -n OrderService.Api            -o OrderService.Api
dotnet new classlib -n OrderService.Core           -o OrderService.Core
dotnet new classlib -n OrderService.Infrastructure -o OrderService.Infrastructure

# 3. Aggiunge i progetti alla solution
dotnet sln add OrderService.Api/OrderService.Api.csproj
dotnet sln add OrderService.Core/OrderService.Core.csproj
dotnet sln add OrderService.Infrastructure/OrderService.Infrastructure.csproj

# 4. Dichiara le dipendenze tra progetti
dotnet add OrderService.Api/OrderService.Api.csproj \
    reference OrderService.Core/OrderService.Core.csproj
dotnet add OrderService.Api/OrderService.Api.csproj \
    reference OrderService.Infrastructure/OrderService.Infrastructure.csproj
dotnet add OrderService.Infrastructure/OrderService.Infrastructure.csproj \
    reference OrderService.Core/OrderService.Core.csproj

# 5. Ripristina le dipendenze
dotnet restore

# 6. Verifica che tutto compili
dotnet build
```

---

## Spiegazione

**`dotnet new sln`** crea il file `.sln` — è solo un registro che tiene traccia di quali progetti fanno parte della solution. Non contiene codice, non contiene dipendenze.

**`dotnet new webapi`** genera il progetto API: `Program.cs`, `appsettings.json`, `appsettings.Development.json`, il `.csproj` con SDK `Microsoft.NET.Sdk.Web`. In .NET 10 il template è già minimale — niente `WeatherForecastController` di default.

**`dotnet new classlib`** genera una libreria di classi: solo il `.csproj` con SDK `Microsoft.NET.Sdk`. È il template giusto per `Core` e `Infrastructure` — non hanno bisogno del web stack.

**`dotnet sln add`** registra un progetto nella solution. In Visual Studio e VS Code la solution è quello che vedi nell'explorer — senza questo comando i progetti esistono ma sono invisibili all'IDE.

**`dotnet add reference`** aggiunge un `<ProjectReference>` nel `.csproj`. È l'equivalente di una `<dependency>` su un modulo interno in Maven. Dopo questo comando, `OrderService.Api` può usare i tipi definiti in `Core` e in `Infrastructure`.

**Perché `Core` non dipende da niente?** Nessun `dotnet add reference` coinvolge `Core` come destinazione — solo come sorgente. `Core` non importa né `Api` né `Infrastructure`. Questo è il vincolo architetturale della Clean Architecture: il dominio non sa nulla del web framework né del database.

## Il risultato

```
OrderService/
├── OrderService.sln
├── OrderService.Api/
│   ├── Program.cs
│   ├── appsettings.json
│   ├── appsettings.Development.json
│   └── OrderService.Api.csproj          ← Sdk.Web, ref a Core e Infrastructure
├── OrderService.Core/
│   └── OrderService.Core.csproj         ← Sdk base, nessuna dipendenza
└── OrderService.Infrastructure/
    └── OrderService.Infrastructure.csproj  ← Sdk base, ref a Core
```

## Confronto con Java

| Maven multi-module | .NET CLI |
|---|---|
| `mvn archetype:generate` | `dotnet new webapi / classlib` |
| Modifica manuale `pom.xml` padre con `<modules>` | `dotnet sln add` |
| `<dependency>` su modulo interno nel `pom.xml` | `dotnet add reference` |
| `mvn install` per rendere il modulo disponibile | Non serve — i `<ProjectReference>` si risolvono in build |
| `mvn clean package` | `dotnet build` |
| `./mvnw spring-boot:run` | `dotnet run --project OrderService.Api` |

La differenza più pratica: in Maven creare un multi-module richiede di scrivere XML a mano. Con `dotnet` CLI ci vogliono sei comandi e non si tocca un file.
