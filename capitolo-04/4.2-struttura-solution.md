# 4.2 Struttura della Solution: singolo progetto vs multi-progetto

---

## Introduzione

Abbiamo già creato lo scheletro nel 4.1. Prima di scrivere codice, vale la pena fermarsi un momento sulla scelta architetturale che abbiamo fatto — e capire perché. La struttura del progetto condiziona tutto quello che verrà dopo.

---

## Come si faceva in Java

In un progetto Spring Boot multi-module Maven la struttura è questa:

```
order-service/
├── pom.xml                          ← parent POM
├── order-service-api/               ← layer web
│   ├── pom.xml
│   └── src/main/java/...
├── order-service-core/              ← dominio e logica
│   ├── pom.xml
│   └── src/main/java/...
└── order-service-infrastructure/    ← database, client HTTP
    ├── pom.xml
    └── src/main/java/...
```

Le dipendenze tra moduli si dichiarano nel `pom.xml` di ciascuno. Il parent POM gestisce le versioni condivise.

## La struttura che useremo

```
OrderService/
├── OrderService.sln
│
├── OrderService.Api/
│   ├── Controllers/
│   ├── appsettings.json
│   ├── appsettings.Development.json
│   ├── Program.cs
│   └── OrderService.Api.csproj
│
├── OrderService.Core/
│   ├── Entities/
│   ├── Interfaces/
│   │   ├── Repositories/
│   │   └── Services/
│   ├── Services/
│   └── OrderService.Core.csproj
│
├── OrderService.Infrastructure/
│   ├── Persistence/
│   │   └── AppDbContext.cs     ← creato nel Capitolo 5
│   ├── Repositories/
│   └── OrderService.Infrastructure.csproj
│
└── OrderService.Tests/         ← creato nel Capitolo 9
    └── OrderService.Tests.csproj
```

Le dipendenze:

```
Api  ──→  Core  ←──  Infrastructure
              ↑
           Tests
```

`Core` non importa nulla di esterno. `Infrastructure` implementa le interfacce definite in `Core`. `Api` orchestra tutto.

---

## Spiegazione

**Perché separare Core da Infrastructure?**

Le entità di dominio e le interfacce (`IOrdineRepository`, `IOrdineService`) vivono in `Core`. Le implementazioni concrete (`OrdineRepository` con EF Core, `OrdineService`) vivono in `Infrastructure` e `Core`. Questo significa che puoi:

- Cambiare database senza toccare il dominio
- Testare la logica di business con implementazioni in-memory, senza un database reale
- Leggere `Core` e capire cosa fa il microservizio senza preoccuparti dei dettagli tecnici

**Singolo progetto — quando ha senso**

Se stai costruendo un microservizio molto piccolo, con 3-4 endpoint e nessuna logica complessa, un singolo progetto flat è più che sufficiente:

```bash
dotnet new webapi -n OrderService
```

Niente solution, niente layer. Aggiungi struttura quando il progetto cresce — non in anticipo.

Per `OrderService` nel corso andiamo con la struttura multi-progetto perché vogliamo mostrare i pattern che trovi nei progetti reali.

## I file `.csproj` dei tre progetti

```xml
<!-- OrderService.Core.csproj — nessuna dipendenza esterna -->
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>
</Project>
```

```xml
<!-- OrderService.Infrastructure.csproj — dipende da Core -->
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>
  <ItemGroup>
    <ProjectReference Include="..\OrderService.Core\OrderService.Core.csproj" />
  </ItemGroup>
</Project>
```

```xml
<!-- OrderService.Api.csproj — dipende da Core e Infrastructure -->
<Project Sdk="Microsoft.NET.Sdk.Web">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>
  <ItemGroup>
    <ProjectReference Include="..\OrderService.Core\OrderService.Core.csproj" />
    <ProjectReference Include="..\OrderService.Infrastructure\OrderService.Infrastructure.csproj" />
  </ItemGroup>
</Project>
```

Nota: `OrderService.Api` usa `Microsoft.NET.Sdk.Web` — include tutto il necessario per una Web API. Gli altri usano `Microsoft.NET.Sdk` — il SDK base per librerie di classi.

## Confronto con Java

| Maven multi-module | .NET Solution |
|---|---|
| `pom.xml` padre | `OrderService.sln` |
| `pom.xml` per modulo | `.csproj` per progetto |
| `<modules>` nel parent | `dotnet sln add` |
| `<dependency>` su modulo interno | `<ProjectReference>` nel `.csproj` |
| `src/main/java/...` obbligatorio | Struttura cartelle libera |
| `<dependencyManagement>` | `Directory.Build.props` |
