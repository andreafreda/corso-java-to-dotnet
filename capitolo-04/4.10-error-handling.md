# 4.10 Gestione degli errori e Problem Details (RFC 7807)

> **Analogia Java**: equivalente di `@ControllerAdvice` + `@ExceptionHandler`

---

## Introduzione

Il microservizio risponde correttamente ai casi felici. Ma cosa succede quando qualcosa va storto — un ordine non trovato, un errore inatteso, un'eccezione nel service? Senza gestione centralizzata degli errori, ASP.NET Core restituisce stack trace in chiaro in development e risposte vuote in production. Non va bene.

Lo standard per le risposte di errore nelle API REST è **RFC 7807 — Problem Details**. ASP.NET Core lo supporta nativamente da .NET 7.

---

## Come si faceva in Java

```java
// @ControllerAdvice — intercetta le eccezioni da tutti i controller
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ProblemDetail> handleNotFound(EntityNotFoundException ex) {
        var problem = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        problem.setTitle("Resource not found");
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(problem);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ProblemDetail> handleGeneric(Exception ex) {
        var problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.INTERNAL_SERVER_ERROR, "An unexpected error occurred");
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(problem);
    }
}
```

## Problem Details in ASP.NET Core

> 📝 **Code Step 8** — Crea `Core/Exceptions/DomainExceptions.cs` e `Api/Middleware/ExceptionHandler.cs`, aggiorna `Program.cs` e `Api/Filters/ValidationFilter.cs`

Il formato Problem Details è questo:

```json
{
  "type": "https://tools.ietf.org/html/rfc7807",
  "title": "Order not found",
  "status": 404,
  "detail": "Order with id 99 was not found.",
  "instance": "/orders/99"
}
```

ASP.NET Core gestisce gli errori tramite un exception handler middleware. Creiamo eccezioni di dominio e un handler globale:

```csharp
// Core/Exceptions/DomainExceptions.cs
namespace OrderService.Core.Exceptions;

public class NotFoundException : Exception
{
    public NotFoundException(string message) : base(message) { }
}
```

```csharp
// Api/Middleware/ExceptionHandler.cs
using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Mvc;
using OrderService.Core.Exceptions;

namespace OrderService.Api.Middleware;

public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
    {
        _logger = logger;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext context,
        Exception exception,
        CancellationToken ct)
    {
        var (statusCode, title) = exception switch
        {
            NotFoundException => (StatusCodes.Status404NotFound, "Resource not found"),
            ArgumentException => (StatusCodes.Status400BadRequest, "Invalid request"),
            _ => (StatusCodes.Status500InternalServerError, "An unexpected error occurred")
        };

        if (statusCode == StatusCodes.Status500InternalServerError)
            _logger.LogError(exception, "Unhandled exception");

        var problem = new ProblemDetails
        {
            Status   = statusCode,
            Title    = title,
            Detail   = exception.Message,
            Instance = context.Request.Path
        };

        context.Response.StatusCode = statusCode;
        await context.Response.WriteAsJsonAsync(problem, ct);
        return true;
    }
}
```

```csharp
// Program.cs — aggiungi prima di app.MapControllers()
using OrderService.Api.Middleware;

// ...

builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
builder.Services.AddProblemDetails();

// ...

app.UseExceptionHandler();
app.MapControllers();
```

**Il service non cambia** — `GetByIdAsync` in `OrderService.cs` rimane identico a come lo hai scritto in 4.7. Il 404 è gestito dal controller con `NotFound()`, che è il posto giusto per questa logica quando hai accesso a `IActionResult`.

Il pattern con eccezioni diventa utile nei layer interni — per esempio quando un'operazione in cascata fallisce e non puoi restituire `IActionResult`. In quel caso lanceresti `NotFoundException` e l'handler la intercetterebbe automaticamente:

```csharp
// Esempio alternativo — quando il 404 emerge da un layer interno
// NON serve modificare OrderService.cs adesso, è solo per riferimento
public async Task<OrderDto> GetByIdOrThrowAsync(int id, CancellationToken ct = default)
{
    var order = await _repository.GetByIdAsync(id, ct);
    return order is null
        ? throw new NotFoundException($"Order with id {id} was not found.")
        : ToDto(order);
}
```

## Standardizzare la risposta di validazione

In 4.9 il `ValidationFilter` restituiva un formato custom. Ora che abbiamo Problem Details, allineiamo il formato:

```csharp
// Api/Filters/ValidationFilter.cs — aggiorna il risultato
if (!result.IsValid)
{
    var errors = result.Errors
        .GroupBy(e => e.PropertyName)
        .ToDictionary(g => g.Key, g => g.Select(e => e.ErrorMessage).ToArray());

    var problem = new ValidationProblemDetails(errors)
    {
        Status = StatusCodes.Status400BadRequest,
        Title  = "Validation failed"
    };

    context.Result = new BadRequestObjectResult(problem);
    return;
}
```

Ora tutti gli errori — validazione, not found, errori inattesi — usano lo stesso formato Problem Details. Il client sa sempre cosa aspettarsi.

## Confronto con Java

| Spring Boot | ASP.NET Core |
|---|---|
| `@RestControllerAdvice` + `@ExceptionHandler` | `IExceptionHandler` + `UseExceptionHandler()` |
| `ProblemDetail` (Spring 6+) | `ProblemDetails` — nativo da .NET 7 |
| `ResponseEntityExceptionHandler` come base | `IExceptionHandler` — interfaccia, nessuna classe base |
| Eccezioni di dominio custom | Stesso pattern — `NotFoundException : Exception` |
| `@Valid` + `BindingResult` per validazione | `ValidationProblemDetails` nel filtro |
