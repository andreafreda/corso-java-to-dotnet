# 4.9 Validazione degli input con FluentValidation

> **Analogia Java**: equivalente di Bean Validation / Hibernate Validator

---

## Introduzione

Il microservizio gira e risponde. Ma cosa succede se qualcuno invia un ordine con `Customer` vuoto o `Total` negativo? Per ora non succede nulla di buono — il dato entra nel sistema e finisce nel repository senza controlli. Aggiungiamo la validazione.

In Java si usava Bean Validation con annotazioni sull'entità o sul DTO. In .NET l'equivalente più diffuso è **FluentValidation** — le regole di validazione stanno in una classe separata, non sull'oggetto validato.

---

## Come si faceva in Java

```java
// Annotazioni sul DTO — validazione dichiarativa
public record CreateOrderRequest(
    @NotBlank(message = "Il cliente è obbligatorio")
    String customer,

    @DecimalMin(value = "0.01", message = "Il totale deve essere positivo")
    BigDecimal total
) {}

// Nel controller — @Valid attiva la validazione
@PostMapping
public ResponseEntity<OrderDto> create(@RequestBody @Valid CreateOrderRequest request) {
    // se la validazione fallisce, Spring risponde automaticamente con 400
}
```

## FluentValidation

> 📝 **Code Step 7** — Installa il package, crea `Api/Validators/CreateOrderRequestValidator.cs` e `Api/Filters/ValidationFilter.cs`, aggiorna `Program.cs`

```bash
dotnet add OrderService.Api package FluentValidation
dotnet add OrderService.Api package FluentValidation.DependencyInjectionExtensions
```

```csharp
// Api/Validators/CreateOrderRequestValidator.cs
using FluentValidation;
using OrderService.Core.DTOs;

namespace OrderService.Api.Validators;

public class CreateOrderRequestValidator : AbstractValidator<CreateOrderRequest>
{
    public CreateOrderRequestValidator()
    {
        RuleFor(x => x.Customer)
            .NotEmpty().WithMessage("Customer is required.")
            .MaximumLength(100).WithMessage("Customer must not exceed 100 characters.");

        RuleFor(x => x.Total)
            .GreaterThan(0).WithMessage("Total must be greater than zero.");
    }
}
```

```csharp
// Program.cs — aggiungi dopo AddCore() e AddInfrastructure()
using FluentValidation;
using OrderService.Api.Validators;

// ...

builder.Services.AddValidatorsFromAssemblyContaining<CreateOrderRequestValidator>();
```

FluentValidation non si integra automaticamente con `[ApiController]` — va invocato esplicitamente nel controller o tramite un filtro. Il modo più pulito è un action filter globale:

```csharp
// Program.cs
using OrderService.Api.Filters;

// ...

builder.Services.AddControllers(options =>
    options.Filters.Add<ValidationFilter>());
```

```csharp
// Api/Filters/ValidationFilter.cs
using FluentValidation;
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;

namespace OrderService.Api.Filters;

public class ValidationFilter : IAsyncActionFilter
{
    private readonly IServiceProvider _serviceProvider;

    public ValidationFilter(IServiceProvider serviceProvider)
    {
        _serviceProvider = serviceProvider;
    }

    public async Task OnActionExecutionAsync(ActionExecutingContext context, ActionExecutionDelegate next)
    {
        foreach (var argument in context.ActionArguments.Values)
        {
            if (argument is null) continue;

            var validatorType = typeof(IValidator<>).MakeGenericType(argument.GetType());
            var validator = _serviceProvider.GetService(validatorType) as IValidator;
            if (validator is null) continue;

            var result = await validator.ValidateAsync(new ValidationContext<object>(argument));
            if (!result.IsValid)
            {
                var errors = result.Errors
                    .GroupBy(e => e.PropertyName)
                    .ToDictionary(g => g.Key, g => g.Select(e => e.ErrorMessage).ToArray());

                context.Result = new BadRequestObjectResult(new { errors });
                return;
            }
        }

        await next();
    }
}
```

## Spiegazione

**Regole in una classe separata** — in Bean Validation le regole stanno sull'oggetto validato con annotazioni. In FluentValidation stanno in `AbstractValidator<T>`, una classe dedicata. Il vantaggio: le regole sono testabili in isolamento, e `CreateOrderRequest` rimane un `record` pulito senza annotazioni.

**`AbstractValidator<T>` con costruttore** — tutte le regole si definiscono nel costruttore con `RuleFor`. Puoi avere regole condizionali, regole che dipendono da altri campi, validazioni asincrone — tutto con la stessa sintassi fluente.

**`ValidationFilter` come filtro globale** — intercetta ogni richiesta prima che arrivi al metodo del controller. Se la validazione fallisce, risponde con 400 senza neanche entrare nell'action. Nel capitolo 4.10 vedremo come standardizzare il formato della risposta di errore con Problem Details.

**`AddValidatorsFromAssemblyContaining`** — registra automaticamente tutti i validator trovati nell'assembly. Aggiungi un validator, viene registrato automaticamente — niente da aggiornare in `Program.cs`.

---

## Confronto con Java

| Bean Validation / Spring | FluentValidation |
|---|---|
| Annotazioni sul DTO (`@NotBlank`, `@Min`) | Regole in `AbstractValidator<T>` separato |
| `@Valid` nel controller attiva la validazione | `ValidationFilter` globale — automatico su tutti gli endpoint |
| Risposta 400 automatica con `@ControllerAdvice` | Risposta 400 nel filtro, formato personalizzabile |
| Regole testabili solo con il contesto Spring | `AbstractValidator<T>` testabile in isolamento con xUnit |
| `@NotBlank`, `@Size`, `@DecimalMin` | `.NotEmpty()`, `.MaximumLength()`, `.GreaterThan()` |
