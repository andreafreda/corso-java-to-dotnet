## Introduzione

Prima di scrivere una riga di codice, mettiamo in piedi l'ambiente. Servono tre cose: il runtime, l'editor, e un modo per testare gli endpoint. Niente di elaborato — se hai già lavorato con Spring Boot da terminale, il processo ti sembrerà familiare.

## Come si faceva in Java

```bash
# Verifica Java installato
java -version
mvn -version

# Nuovo progetto Spring Boot da terminale
curl https://start.spring.io/starter.zip \
  -d dependencies=web \
  -d name=order-service \
  -o order-service.zip && unzip order-service.zip

# Avvio
cd order-service
mvn spring-boot:run
```

Per testare gli endpoint in VS Code si usava il plugin **REST Client** con file `.http` — esattamente lo stesso strumento che useremo qui.

## Installazione

### 1. .NET 10 SDK

Scarica e installa da [dot.net](https://dot.net). Verifica l'installazione:

```bash
dotnet --version
# 10.0.xxx
```

### 2. VS Code

Installa le estensioni:

```bash
# Da terminale
code --install-extension ms-dotnettools.csdevkit
code --install-extension humao.rest-client
```

Oppure cerca nell'estensioni di VS Code:
- **C# Dev Kit** (Microsoft) — IntelliSense, debug, navigazione
- **REST Client** — per testare gli endpoint con file `.http`

### 3. Verifica finale

```bash
dotnet --info
```

Deve mostrare SDK 10.x e runtime installati. Se vedi versioni precedenti, controlla che il PATH punti alla versione giusta o aggiungi un `global.json` (vedi 1.2).

## Creazione del progetto

```bash
# Crea la cartella e la solution
mkdir OrderService && cd OrderService
dotnet new sln -n OrderService

# Crea i tre progetti
dotnet new webapi -n OrderService.Api -o OrderService.Api
dotnet new classlib -n OrderService.Core -o OrderService.Core
dotnet new classlib -n OrderService.Infrastructure -o OrderService.Infrastructure

# Aggiunge i progetti alla solution
dotnet sln add OrderService.Api/OrderService.Api.csproj
dotnet sln add OrderService.Core/OrderService.Core.csproj
dotnet sln add OrderService.Infrastructure/OrderService.Infrastructure.csproj

# Collega le dipendenze tra progetti
dotnet add OrderService.Api/OrderService.Api.csproj reference OrderService.Core/OrderService.Core.csproj
dotnet add OrderService.Api/OrderService.Api.csproj reference OrderService.Infrastructure/OrderService.Infrastructure.csproj
dotnet add OrderService.Infrastructure/OrderService.Infrastructure.csproj reference OrderService.Core/OrderService.Core.csproj

# Ripristina le dipendenze
dotnet restore

# Primo avvio
dotnet run --project OrderService.Api
```

## Spiegazione

Il template `webapi` in .NET 10 genera un progetto già minimale — niente `WeatherForecastController`, niente codice di esempio da eliminare. `Program.cs` è pronto per essere usato così com'è.

La struttura che abbiamo creato rispecchia la **Clean Architecture** che abbiamo anticipato nel 3.1:

```
OrderService/
├── OrderService.sln
├── OrderService.Api/           ← entry point, controller, Program.cs
├── OrderService.Core/          ← dominio: entità, interfacce, servizi
└── OrderService.Infrastructure/← implementazioni: repository, EF Core
```

`Core` non dipende da nulla — né da `Api` né da `Infrastructure`. È il vincolo architetturale più importante: il dominio non sa nulla del database né del web framework.

## Strumenti per testare gli endpoint

Hai diverse opzioni, tutte valide:

**REST Client (VS Code)** — estensione che legge file `.http` direttamente nell'editor. Zero setup, risultati inline accanto al codice. I file `.http` possono essere committati nel repository e condivisi col team.

**Bruno** — client API open source e gratuito, alternativa moderna a Postman. Le collezioni sono salvate come file su disco (non nel cloud), quindi si commitano in git senza account. Interfaccia grafica, ottimo per chi preferisce non stare nel terminale.

**Postman** — il classico. Probabilmente Lo conosci già se vieni da Java. Interfaccia grafica, environment variables, collezioni condivisibili — tutto quello che serve per testare una API in modo organizzato.

## Il file `.http` per testare gli endpoint

Il file `.http` si scrive una volta e si esegue ogni volta che vuoi — direttamente da VS Code con l'estensione REST Client. Questo è il formato:

```http
### Ottieni tutti gli ordini
GET http://localhost:5000/orders
Accept: application/json

### Crea un ordine
POST http://localhost:5000/orders
Content-Type: application/json

{
  "customer": "Andrea Fred",
  "total": 99.99
}

### Ottieni un ordine specifico
GET http://localhost:5000/orders/1
Accept: application/json
```

Con l'estensione REST Client installata, apparirà il link **Send Request** sopra ogni chiamata — clicca e vedi la risposta direttamente in VS Code. Il file definitivo del progetto viene creato nel Code Step 3 (paragrafo 4.5).
