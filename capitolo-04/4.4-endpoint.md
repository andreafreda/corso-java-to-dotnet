# 4.4 Endpoint REST: definiamo il contratto dell'API

---

## Una scelta didattica

Prima di scrivere la prima riga di codice, una nota su come procederemo in questo capitolo.

Nei corsi tradizionali si parte dalle fondamenta: prima le entità, poi il repository, poi il service, e solo alla fine gli endpoint. Il problema è che arrivi a vedere qualcosa di funzionante dopo aver scritto tre layer di codice che non hai ancora potuto testare. Se hai sbagliato qualcosa all'inizio, lo scopri solo alla fine.

Noi partiamo dall'API — dal contratto esterno — e scendiamo verso la persistenza. 

UN ERRORE VOLUTO: In questo paragrafo gli endpoint restituiscono dati hardcoded con modelli definiti temporaneamente in `Api`: il microservizio gira e risponde già alla fine di 4.4. Nei paragrafi successivi spostiamo i modelli in `Core`, introduciamo il service layer e infine la persistenza reale. Ad ogni step il progetto rimane funzionante.

---

## Introduzione

Gli endpoint definiscono il contratto dell'API: cosa accetta, cosa restituisce, quali URL espone. Scriverli per primi ha un vantaggio preciso — puoi discutere il contratto con il tuo team, testarlo con Swagger o un file `.http`, e tutto il codice che scriverai dopo servirà a soddisfare quegli endpoint.

Useremo i **controller** — come visto in 3.3, la struttura è quasi identica a Spring MVC e non richiede dipendenze aggiuntive.

## Come si faceva in Java

```java
@RestController
@RequestMapping("/orders")
public class OrderController {

    @GetMapping
    public List<OrderDto> getAll() {
        return List.of(
            new OrderDto(1, "Mario Rossi",    new BigDecimal("150.00"), "CONFIRMED"),
            new OrderDto(2, "Giulia Bianchi", new BigDecimal("89.50"),  "PENDING")
        );
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrderDto> getById(@PathVariable int id) {
        if (id != 1) return ResponseEntity.notFound().build();
        return ResponseEntity.ok(new OrderDto(1, "Mario Rossi", new BigDecimal("150.00"), "CONFIRMED"));
    }

    @PostMapping
    public ResponseEntity<OrderDto> create(@RequestBody CreateOrderRequest request) {
        var order = new OrderDto(3, request.customer(), request.total(), "PENDING");
        return ResponseEntity.created(URI.create("/orders/3")).body(order);
    }
}
```

## Il controller in `OrderService.Api`

> 📝 **Code Step 2** — Crea `Api/Models/OrderModels.cs` e `Api/Controllers/OrdersController.cs`

I modelli li definiamo per ora direttamente in `Api` — li sposteremo in `Core` nel prossimo paragrafo.

```
OrderService.Api/
├── Controllers/
│   └── OrdersController.cs
└── Models/
    └── OrderModels.cs     ← temporaneo, si sposta in Core nel 4.6
```

```csharp
// Models/OrderModels.cs — temporanei, verranno spostati in Core
namespace OrderService.Api.Models;

public record OrderDto(int Id, string Customer, decimal Total, string Status, DateTime CreatedAt);
public record CreateOrderRequest(string Customer, decimal Total);
```

```csharp
// Controllers/OrdersController.cs
using Microsoft.AspNetCore.Mvc;
using OrderService.Api.Models;

namespace OrderService.Api.Controllers;

[ApiController]
[Route("orders")]
public class OrdersController : ControllerBase
{
    [HttpGet]
    public IActionResult GetAll()
    {
        var orders = new[]
        {
            new OrderDto(1, "Mario Rossi",    150.00m, "Confirmed", DateTime.UtcNow),
            new OrderDto(2, "Giulia Bianchi",  89.50m, "Pending",   DateTime.UtcNow),
        };
        return Ok(orders);
    }

    [HttpGet("{id:int}")]
    public IActionResult GetById(int id)
    {
        if (id != 1) return NotFound();
        return Ok(new OrderDto(1, "Mario Rossi", 150.00m, "Confirmed", DateTime.UtcNow));
    }

    [HttpPost]
    public IActionResult Create([FromBody] CreateOrderRequest request)
    {
        var order = new OrderDto(3, request.Customer, request.Total, "Pending", DateTime.UtcNow);
        return CreatedAtAction(nameof(GetById), new { id = order.Id }, order);
    }
}
```

Il template `dotnet new webapi` in .NET 10 genera un `Program.cs` con Minimal API. Per usare i controller dobbiamo aggiornarlo:

```csharp
// Api/Program.cs — sostituisci il contenuto generato dal template
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();

var app = builder.Build();

app.MapControllers();
app.Run();
```

```bash
dotnet run --project OrderService.Api
# GET  http://localhost:5000/orders    →  200 con la lista
# GET  http://localhost:5000/orders/1  →  200
# GET  http://localhost:5000/orders/99 →  404
# POST http://localhost:5000/orders    →  201
```

Il microservizio gira. I dati sono hardcoded e i modelli sono nel posto sbagliato — lo sappiamo, è intenzionale. Dal prossimo paragrafo iniziamo a mettere tutto al posto giusto. `Program.cs` verrà completato con Scalar e il DI nei prossimi step.

## Spiegazione

**`record` come modelli temporanei** — `OrderDto` e `CreateOrderRequest` sono già nella forma definitiva. Vivono in `Api/Models` per ora solo perché `Core` non esiste ancora come layer strutturato. In 4.6 li spostiamo senza modificarli.

**`[ApiController]`** — abilita la validazione automatica del model e l'inferenza delle sorgenti dei parametri. Con `[ApiController]` attivo, ASP.NET Core capisce da solo che `CreateOrderRequest` viene dal body e `id` dalla route.

**`CreatedAtAction`** — restituisce HTTP 201 con l'header `Location: /orders/3` generato automaticamente da `nameof(GetById)`. Equivalente di `ResponseEntity.created(URI.create(...)).body(dto)` in Spring.

---

## Confronto con Java

| Spring MVC | ASP.NET Core |
|---|---|
| `@RestController` | `[ApiController]` + `: ControllerBase` |
| `@RequestMapping("/orders")` | `[Route("orders")]` |
| `@GetMapping("/{id}")` | `[HttpGet("{id:int}")]` |
| `@PathVariable int id` | `int id` (inferito dalla route) |
| `@RequestBody` | `[FromBody]` (o inferito con `[ApiController]`) |
| `ResponseEntity.ok(body)` | `Ok(body)` |
| `ResponseEntity.notFound().build()` | `NotFound()` |
| `ResponseEntity.created(uri).body(dto)` | `CreatedAtAction(nameof(GetById), ...)` |
