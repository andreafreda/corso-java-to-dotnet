# 4.6 DTO e dominio: entità e modelli nel progetto Core

---

## Introduzione

In 4.4 abbiamo definito `OrderDto` e `CreateOrderRequest` direttamente in `Api/Models` — il posto sbagliato, lo sapevamo. Ora li spostiamo in `Core`, il layer che non dipende da nessuno e che contiene tutto ciò che definisce il dominio del microservizio: entità, DTO e interfacce.

Nessun codice nuovo da scrivere da zero — si tratta di spostare e completare ciò che esiste già.

---

## Come si faceva in Java

```java
// In Spring Boot il confine era spesso sfumato — entità JPA con annotazioni di persistenza
// mescolate al dominio, DTO sparsi tra i package senza una regola precisa.

// package com.example.order.model
@Entity
public class Order { ... }  // dominio + persistenza insieme

// package com.example.order.dto
public record OrderDto(...) {}  // almeno i DTO erano separati
```

In Clean Architecture il contratto è preciso: `Core` contiene il dominio puro, senza annotazioni di framework. Le annotazioni EF Core arriveranno in `Infrastructure` nel capitolo 5.

## Il codice

> 📝 **Code Step 4** — Cancella `Api/Models/OrderModels.cs`, crea la struttura `Core/` con entità, DTO e interfacce, aggiorna i `using` nel controller

Inizia cancellando `OrderService.Api/Models/OrderModels.cs` e ricreandolo in `Core`:

```csharp
// Core/DTOs/OrderModels.cs — spostato da Api/Models, invariato
namespace OrderService.Core.DTOs;

public record OrderDto(int Id, string Customer, decimal Total, string Status, DateTime CreatedAt);
public record CreateOrderRequest(string Customer, decimal Total);
```

```csharp
// Core/Entities/Order.cs — nuovo
namespace OrderService.Core.Entities;

public class Order
{
    public int Id { get; set; }
    public string Customer { get; set; } = string.Empty;
    public decimal Total { get; set; }
    public OrderStatus Status { get; set; } = OrderStatus.Pending;
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
}

public enum OrderStatus
{
    Pending,
    Confirmed,
    Cancelled
}
```

```csharp
// Core/Interfaces/Repositories/IOrderRepository.cs — nuovo
using OrderService.Core.Entities;

namespace OrderService.Core.Interfaces.Repositories;

public interface IOrderRepository
{
    Task<IEnumerable<Order>> GetAllAsync(CancellationToken ct = default);
    Task<Order?> GetByIdAsync(int id, CancellationToken ct = default);
    Task<Order> AddAsync(Order order, CancellationToken ct = default);
}
```

```csharp
// Core/Interfaces/Services/IOrderService.cs — nuovo
using OrderService.Core.DTOs;

namespace OrderService.Core.Interfaces.Services;

public interface IOrderService
{
    Task<IEnumerable<OrderDto>> GetAllAsync(CancellationToken ct = default);
    Task<OrderDto?> GetByIdAsync(int id, CancellationToken ct = default);
    Task<OrderDto> CreateAsync(CreateOrderRequest request, CancellationToken ct = default);
}
```

Ora aggiorna il controller per usare i tipi da `Core` invece che da `Api.Models`:

```csharp
// Controllers/OrdersController.cs — solo i using cambiano, il corpo è invariato
using Microsoft.AspNetCore.Mvc;
using OrderService.Core.DTOs;           // ← era OrderService.Api.Models
// OrderService.Core.Interfaces.Services non serve ancora — arriva in 4.7

namespace OrderService.Api.Controllers;

[ApiController]
[Route("orders")]
public class OrdersController : ControllerBase
{
    [HttpGet]
    public IActionResult GetAll()
    {
        var orders = new[]
        {
            new OrderDto(1, "Mario Rossi",    150.00m, "Confirmed", DateTime.UtcNow),
            new OrderDto(2, "Giulia Bianchi",  89.50m, "Pending",   DateTime.UtcNow),
        };
        return Ok(orders);
    }

    [HttpGet("{id:int}")]
    public IActionResult GetById(int id)
    {
        if (id != 1) return NotFound();
        return Ok(new OrderDto(1, "Mario Rossi", 150.00m, "Confirmed", DateTime.UtcNow));
    }

    [HttpPost]
    public IActionResult Create([FromBody] CreateOrderRequest request)
    {
        var order = new OrderDto(3, request.Customer, request.Total, "Pending", DateTime.UtcNow);
        return CreatedAtAction(nameof(GetById), new { id = order.Id }, order);
    }
}
```

```bash
dotnet build   # verde
dotnet run --project OrderService.Api
# gli endpoint funzionano esattamente come prima
```

## Spiegazione

**`Core` non dipende da nessuno** — `OrderService.Core.csproj` non ha `<PackageReference>` né `<ProjectReference>`. È il layer più interno dell'architettura: tutto dipende da lui, lui non dipende da niente.

**Entità senza annotazioni** — `Order` è una classe C# normale. Nessun `[Table]`, nessun `[Key]`. Le annotazioni EF Core le aggiungeremo in `Infrastructure` nel capitolo 5 — il dominio non sa nulla del database.

**`IOrderRepository` in `Core`** — l'interfaccia definisce il contratto ("voglio poter recuperare e salvare ordini") senza dire come. L'implementazione concreta con EF Core arriverà in `Infrastructure`. Questo è il principio di inversione delle dipendenze in pratica.

**`decimal` per i valori monetari** — tipo primitivo del CLR a precisione fissa. Niente boxing, niente `new BigDecimal("99.99")` — scrivi `99.99m` e il compilatore sa già di cosa si tratta.

---

## Confronto con Java

| Java / Spring Boot | .NET / Clean Architecture |
|---|---|
| Entità `@Entity` con annotazioni JPA | Entità POCO senza annotazioni — il mapping EF Core va in `Infrastructure` |
| DTO sparsi nei package | DTO in `Core/DTOs`, sempre separati dall'entità |
| `BigDecimal` per valori monetari | `decimal` — tipo primitivo, niente boxing |
| `JpaRepository<T, ID>` come interfaccia base | `IOrderRepository` custom definita nel dominio |
| `Long` come tipo ID | `int` (o `Guid` per sistemi distribuiti) |
