# 6.4 Outbox pattern: garantire consistenza distribuita

> **Analogia Java**: pattern architetturale indipendente dal linguaggio — applicabile con Spring + Kafka/Service Bus

---

## Introduzione

In 6.3 il producer pubblica l'evento su Service Bus dopo aver salvato l'ordine. C'è un problema silenzioso:

```csharp
await _repository.AddAsync(order, ct);       // 1. salva su DB ✓
await _publisher.PublishOrderCreatedAsync(); // 2. pubblica su Service Bus — può fallire
```

Se il publish fallisce (network timeout, Service Bus down, crash del processo), l'ordine è nel database ma l'evento non è mai stato inviato. `InventoryService` non scalarà il magazzino. La consistenza è rotta.

L'**Outbox pattern** risolve questo: salvi l'evento nel database nella stessa transazione dell'ordine, poi un processo separato lo legge e lo pubblica su Service Bus. O entrambi i dati vengono salvati, o nessuno.

---

## Il problema senza Outbox

```
Client → OrderService
              ↓
         [DB Transaction]
              ↓
         SaveOrder ✓
              ↓
         [Transaction committed]
              ↓
         PublishEvent ← crash qui → evento perso per sempre
```

---

## Come si faceva in Java

```java
// Debezium — Change Data Capture, legge il transaction log del DB
// Non richiede codice applicativo — legge direttamente il WAL di PostgreSQL
// e pubblica su Kafka automaticamente

// Oppure: Outbox table con Spring
@Transactional
public Order createOrder(CreateOrderRequest request) {
    Order order = orderRepository.save(new Order(request));

    // Salva l'evento nella stessa transazione
    OutboxEvent event = new OutboxEvent(
        "OrderCreated",
        objectMapper.writeValueAsString(new OrderCreatedEvent(order)));
    outboxRepository.save(event);

    return order;  // Spring commita entrambi
}

// Scheduler separato che pubblica gli eventi pendenti
@Scheduled(fixedDelay = 1000)
@Transactional
public void processOutbox() {
    List<OutboxEvent> pending = outboxRepository.findByPublishedFalse();
    for (OutboxEvent event : pending) {
        messagingTemplate.send(event.getTopic(), event.getPayload());
        event.setPublished(true);
    }
}
```

### 1. Entità Outbox

```csharp
namespace OrderService.Core.Entities;

public class OutboxMessage
{
    public Guid      Id          { get; set; } = Guid.NewGuid();
    public string    Type        { get; set; } = string.Empty;  // es. "OrderCreated"
    public string    Payload     { get; set; } = string.Empty;  // JSON dell'evento
    public DateTime  CreatedAt   { get; set; } = DateTime.UtcNow;
    public DateTime? ProcessedAt { get; set; }                  // null = non ancora pubblicato
```

### 2. Aggiungi `OutboxMessage` al DbContext

```csharp
// Infrastructure/Persistence/AppDbContext.cs
public class AppDbContext : DbContext

    public DbSet<Order>         Orders  { get; set; }
    public DbSet<OutboxMessage> Outbox  { get; set; }  // ← aggiunto

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(entity =>
        {
            entity.ToTable("Orders");
            entity.HasKey(o => o.Id);

            entity.ToTable("Outbox");
            entity.HasKey(o => o.Id);
            entity.Property(o => o.Payload).HasColumnType("nvarchar(max)");
        });
    }
}
```

### 4. Salva ordine ed evento nella stessa transazione

Il Core costruisce l'evento e lo passa al repository — la serializzazione resta in Infrastructure.

```csharp
// Core/Services/OrderService.cs
public async Task<Order> CreateAsync(CreateOrderRequest request, CancellationToken ct = default)
{
    var order = new Order
    {
        Customer  = request.Customer,
        Total     = request.Total,
        Status    = OrderStatus.Pending,
        CreatedAt = DateTime.UtcNow
    };

    // Il Core costruisce l'evento ma NON lo serializza — è responsabilità di Infrastructure
    var eventData = new OrderCreatedEvent(
        OrderId:  0,                 // Id non ancora assegnato — il repository lo sostituirà
        Customer: order.Customer,
        Total:    order.Total,
        CreatedAt: order.CreatedAt);

    await _repository.AddWithOutboxAsync(order, eventData, ct);

    return order;
}
```

```csharp
// Infrastructure/Repositories/EfCoreOrderRepository.cs — aggiunto
public async Task<Order> AddWithOutboxAsync(
    Order order, OrderCreatedEvent eventData, CancellationToken ct = default)
{
    await using var tx = await _db.Database.BeginTransactionAsync(ct);

    _db.Orders.Add(order);
    await _db.SaveChangesAsync(ct);  // order.Id ora disponibile (generato dal DB)

    // Serializzazione in Infrastructure — il Core non tocca JsonSerializer
    _db.Outbox.Add(new OutboxMessage
    {
        Type    = "OrderCreated",
        Payload = JsonSerializer.Serialize(eventData with { OrderId = order.Id })
    });
    await _db.SaveChangesAsync(ct);

    await tx.CommitAsync(ct);  // unico commit atomico

    return order;
}
```

### 5. Outbox processor — pubblica i messaggi pendenti

```csharp
// Infrastructure/Messaging/OutboxProcessor.cs
using System.Text.Json;
using Azure.Messaging.ServiceBus;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using OrderService.Core.Entities;
using OrderService.Infrastructure.Persistence;

namespace OrderService.Infrastructure.Messaging;

public class OutboxProcessor : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ServiceBusSender     _sender;       // creato una volta sola
    private readonly ILogger<OutboxProcessor> _logger;

    public OutboxProcessor(
        IServiceScopeFactory scopeFactory,
        ServiceBusClient serviceBusClient,
        ILogger<OutboxProcessor> logger)
    {
        _scopeFactory = scopeFactory;
        _sender       = serviceBusClient.CreateSender("order-created");
        _logger       = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            await ProcessPendingMessagesAsync(ct);
            await Task.Delay(TimeSpan.FromSeconds(5), ct).ConfigureAwait(ConfigureAwaitOptions.SuppressThrowing);
        }
    }

    private async Task ProcessPendingMessagesAsync(CancellationToken ct)
    {
        using var scope = _scopeFactory.CreateScope();
        var db          = scope.ServiceProvider.GetRequiredService<AppDbContext>();

        var pending = await db.Outbox
            .Where(o => o.ProcessedAt == null)
            .OrderBy(o => o.CreatedAt)
            .Take(10)
            .ToListAsync(ct);

        if (pending.Count == 0) return;

        foreach (var outbox in pending)
        {
            try
            {
                var message = new ServiceBusMessage(outbox.Payload)
                {
                    ContentType = "application/json",
                    Subject     = outbox.Type,
                    MessageId   = outbox.Id.ToString()
                };

                await _sender.SendMessageAsync(message, ct);

                outbox.ProcessedAt = DateTime.UtcNow;
                _logger.LogInformation(
                    "Outbox message {Id} ({Type}) pubblicato su Service Bus", outbox.Id, outbox.Type);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Errore pubblicazione outbox message {Id}", outbox.Id);
                // non marcare come processed — verrà riprovato al prossimo ciclo
            }
        }

        await db.SaveChangesAsync(ct);
    }
}
```

## Il flusso completo

```
POST /orders
     ↓
[DB Transaction]
     ├── INSERT Orders (order)
     └── INSERT Outbox (evento)
     ↓
[Transaction committed] ← unico punto di fallimento atomico
     ↓
[OutboxProcessor — ogni 5 secondi]
     ↓
     ├── SELECT Outbox WHERE ProcessedAt IS NULL
     ├── SendMessageAsync → Service Bus ✓
     └── UPDATE Outbox SET ProcessedAt = NOW()
```

Se il processo crasha tra il commit e la pubblicazione, al riavvio l'`OutboxProcessor` ritrova i messaggi con `ProcessedAt = null` e li pubblica. L'evento viene consegnato **at-least-once** — potrebbe essere pubblicato più di una volta, quindi il consumer deve essere **idempotente**.

---

## Spiegazione

**At-least-once delivery** — l'Outbox garantisce che l'evento venga pubblicato almeno una volta, non esattamente una volta. In caso di crash tra `SendMessageAsync` e l'aggiornamento di `ProcessedAt`, il messaggio viene inviato di nuovo. Il consumer deve gestire i duplicati — tipicamente con un campo `MessageId` e una tabella di eventi già processati.

**`MessageId = outbox.Id.ToString()`** — usare l'Id dell'Outbox come `MessageId` permette a Service Bus di deduplicare i messaggi duplicati se hai abilitato la duplicate detection sul topic (finestra di deduplication configurabile).

**Polling vs Change Data Capture** — il nostro `OutboxProcessor` fa polling ogni 5 secondi. È semplice ma introduce una latenza minima. L'alternativa è Debezium: legge il transaction log del database (WAL) e pubblica in tempo reale senza polling. Più complesso, più potente — vale per volumi alti.

**Transazione esplicita e doppio `SaveChangesAsync`** — il metodo fa due `SaveChangesAsync` dentro la stessa transazione: il primo genera `order.Id` (assegnato dal DB come `IDENTITY`), il secondo inserisce l'`OutboxMessage` con l'Id corretto. Entrambi i save fanno parte della stessa transazione — se `CommitAsync` non viene chiamato, entrambe le operazioni vengono rollbackate. Non è un bug: è atomico. Il doppio save è necessario perché `int IDENTITY` non è noto prima dell'insert. L'alternativa è usare `Guid` come tipo di `Order.Id`: il Guid viene assegnato in memoria prima dell'insert (`Guid.NewGuid()`), il payload può essere serializzato subito, e basta un unico `SaveChangesAsync` senza transazione esplicita. Il trade-off è che gli URL diventano `/orders/{guid}` — cambiamento che si propaga a controller, route constraints e test.

**Serializzazione in Infrastructure, non in Core** — `Core/Services/OrderService.cs` non usa `JsonSerializer`. Costruisce l'evento come oggetto tipizzato (`OrderCreatedEvent`) e lo passa al repository. La serializzazione a JSON avviene in `EfCoreOrderRepository`, che fa parte di Infrastructure. Questo rispetta la Dependency Inversion: il Core dipende da astrazioni, non da dettagli di implementazione (il formato di serializzazione è un dettaglio di Infrastructure).

**Perché non `IDbContextFactory`** — `AppDbContext` è `Scoped`, `OutboxProcessor` è `Singleton`. Usiamo `IServiceScopeFactory` come in 6.3 per creare uno scope per ogni ciclo di elaborazione. In alternativa si può usare `IDbContextFactory<AppDbContext>` registrato come `Singleton` — ma `IServiceScopeFactory` è il pattern più generale.

## Confronto con Java

| Java | .NET |
|---|---|
| Tabella Outbox con Spring Data + `@Transactional` | `OutboxMessage` in EF Core + `SaveChangesAsync` atomico |
| `@Scheduled(fixedDelay = 1000)` per il polling | `BackgroundService` con `Task.Delay` |
| Debezium per Change Data Capture | Debezium (disponibile anche per SQL Server) |
| Idempotency key sul consumer | `MessageId` su `ServiceBusMessage` + duplicate detection |
| At-least-once con Spring JMS | At-least-once con `CompleteMessageAsync` esplicito |
