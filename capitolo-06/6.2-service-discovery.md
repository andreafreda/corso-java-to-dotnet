# 6.2 Service discovery e pattern client-side

> **Analogia Java**: equivalente di Eureka + Ribbon / Spring Cloud LoadBalancer

---

## Introduzione

In 6.1 l'URL di `InventoryService` è hardcoded in `appsettings.Development.json`. In un ambiente reale i microservizi scalano orizzontalmente — più istanze, IP dinamici, deployment rolling. Serve un meccanismo per trovare le istanze disponibili a runtime: il **service discovery**.

In Java usavi Eureka (registry) + Ribbon o Spring Cloud LoadBalancer (client-side load balancing). In .NET l'ecosistema è più eterogeneo: in produzione su Kubernetes il discovery è integrato nella piattaforma, per ambienti più semplici si usano soluzioni come Consul o il load balancing nativo di `IHttpClientFactory`.

---

## Come si faceva in Java

```java
// Eureka — il servizio si registra all'avvio
// application.yml
eureka:
  client:
    serviceUrl:
      defaultZone: http://eureka-server:8761/eureka/
  instance:
    preferIpAddress: true
```

```java
// Ribbon / Spring Cloud LoadBalancer — client-side load balancing
@LoadBalanced
@Bean
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// Usa il nome del servizio invece dell'IP — Ribbon risolve
restTemplate.getForObject("http://inventory-service/inventory/1", InventoryResponse.class);
```

```java
// Spring Cloud LoadBalancer moderno
@Bean
@LoadBalanced
public WebClient.Builder webClientBuilder() {
    return WebClient.builder();
}
```

## Service discovery in .NET: le opzioni

### Opzione 1 — Kubernetes (produzione)

In Kubernetes il discovery è integrato: ogni `Service` ha un DNS stabile (`inventory-service.default.svc.cluster.local`). Non serve nessuna libreria — basta configurare l'URL nel `ConfigMap`:

```yaml
# k8s/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: order-service-config
data:
  Services__InventoryUrl: "http://inventory-service:80"
```

L'URL non cambia mai — Kubernetes fa il load balancing a livello di `Service`. È il pattern più semplice e il più usato in produzione.

### Opzione 2 — Consul (ambienti ibridi)

```bash
dotnet add OrderService.Infrastructure package Winton.Extensions.Configuration.Consul
```

```csharp
// Program.cs
builder.Configuration.AddConsul("order-service/config");

// appsettings.json punta al nome del servizio Consul
// "Services:InventoryUrl": "http://inventory-service"
```

### Opzione 3 — Client-side load balancing con `IHttpClientFactory`

Per ambienti senza orchestratore, puoi implementare un load balancer client-side semplice direttamente in `IHttpClientFactory`:

```csharp
// Infrastructure/Http/RoundRobinHandler.cs
public class RoundRobinHandler : DelegatingHandler
{
    private readonly string[] _urls;
    private int _index;

    public RoundRobinHandler(string[] urls)
    {
        _urls = urls;
    }

    protected override Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken ct)
    {
        var url = _urls[Interlocked.Increment(ref _index) % _urls.Length];
        request.RequestUri = new Uri(url + request.RequestUri!.PathAndQuery);
        return base.SendAsync(request, ct);
    }
}
```

```csharp
// Infrastructure/DependencyInjection.cs
var inventoryUrls = configuration
    .GetSection("Services:InventoryUrls")
    .Get<string[]>()!;

// AddHttpMessageHandler<T> richiede che il handler sia Transient — Singleton causa eccezione a runtime
services.AddTransient(_ => new RoundRobinHandler(inventoryUrls));
services.AddHttpClient("inventory")
    .AddHttpMessageHandler<RoundRobinHandler>();
```

```json
// appsettings.Development.json
{
  "Services": {
    "InventoryUrls": [
      "http://localhost:5001",
      "http://localhost:5002"
    ]
  }
}
```

## Pattern per il corso: configurazione per ambiente

Nel corso non installiamo un registry — non è necessario per capire il pattern. Usiamo la configurazione per ambiente come livello di astrazione: l'URL cambia tra Development, Staging e Production senza toccare il codice.

```json
// appsettings.Development.json — istanza locale
{
  "Services": {
    "InventoryUrl": "http://localhost:5001"
  }
}
```

```json
// appsettings.Production.json — nome DNS Kubernetes
{
  "Services": {
    "InventoryUrl": "http://inventory-service"
  }
}
```

```csharp
// In Infrastructure/DependencyInjection.cs — invariato
services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    client.BaseAddress = new Uri(configuration["Services:InventoryUrl"]!);
});
```

In Development punti all'istanza locale. In produzione su Kubernetes l'URL è il nome del `Service` Kubernetes — il DNS del cluster risolve automaticamente verso le istanze disponibili con load balancing round-robin.

## Spiegazione

**Perché non Eureka in .NET** — Eureka è un progetto Spring Cloud. Esiste un client .NET (`Steeltoe`) ma non è lo standard dell'ecosistema. In .NET su Azure il discovery è delegato a Kubernetes, App Service o Azure Container Apps — tutti forniscono DNS stabile per i servizi.

**Client-side vs server-side load balancing** — Ribbon e Spring Cloud LoadBalancer sono client-side: ogni client conosce le istanze e sceglie a quale parlare. In Kubernetes il load balancing è server-side (o meglio, infrastrutturale): il client parla con un singolo IP virtuale (`ClusterIP`) e Kubernetes distribuisce il traffico. Più semplice, meno codice.

**`ASPNETCORE_ENVIRONMENT`** — la variabile d'ambiente che seleziona quale `appsettings.{Environment}.json` viene caricato. In Development è impostata automaticamente da `dotnet run`. In produzione la imposti nel `Deployment` Kubernetes o nella configurazione del container.

## Confronto con Java

| Spring Cloud | ASP.NET Core |
|---|---|
| Eureka Server — registry centralizzato | Kubernetes DNS / Consul |
| `@EnableEurekaClient` — auto-registrazione | Nessuna registrazione — Kubernetes gestisce |
| `@LoadBalanced RestTemplate` — client-side LB | `DelegatingHandler` custom / Kubernetes Service |
| `spring.application.name` come nome servizio | Nome del `Service` Kubernetes |
| `application-dev.yml` per URL locali | `appsettings.Development.json` per URL locali |
| Ribbon — round robin client-side | `DelegatingHandler` custom (raro in produzione) |
