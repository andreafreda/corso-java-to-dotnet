# 1.2 L'SDK e la CLI (`dotnet` CLI)

> **Analogia Java**: equivalente di `mvn`/`gradle` da terminale — un unico strumento per creare, compilare, eseguire, testare e pubblicare progetti .NET

---

## Il concetto che già conosci

Se vieni da Maven o Gradle, sai già cosa significa avere una CLI che gestisce l'intero ciclo di vita del progetto da terminale: crei lo scaffolding, aggiungi dipendenze, compili, esegui i test, crei l'artefatto da distribuire. Tutto senza aprire un IDE.

In .NET questo strumento si chiama semplicemente **`dotnet`** ed è incluso nell'SDK. Un solo eseguibile copre tutto: non c'è un `dotnet-compiler` separato da un `dotnet-test-runner`, come non c'è un `mvn` separato da un `gradle`.

---

## SDK vs Runtime: cosa installi

Il ragionamento è lo stesso che conosci da Java: da una parte hai il **JDK**, che include il compilatore, gli strumenti da riga di comando e il runtime; dall'altra il **JRE**, che contiene solo il runtime per eseguire applicazioni già compilate.

In .NET funziona allo stesso modo:

| Java | .NET | Contiene | Quando serve |
|---|---|---|---|
| **JDK** | **.NET SDK** | Compiler (Roslyn) + runtime + CLI (`dotnet`) + template | Sviluppo, build, CI/CD |
| **JRE** | **.NET Runtime** | Solo il CLR per eseguire app già compilate | Deploy su server di produzione |

In sviluppo installi sempre l'**SDK**. Sul server puoi installare solo il runtime (immagine Docker più leggera), oppure usare la **self-contained publish** — il runtime viene incluso direttamente nell'artefatto e sul server non serve installare nulla.

Verifica cosa hai installato:

```bash
dotnet --info
```

Output tipico (semplificato):
```
.NET SDK:
 Version:   10.0.100
 Commit:    abc123

Runtime Environment:
 OS Name:   Windows / Linux / macOS
 ...

Installed SDKs:
  10.0.100  [C:\Program Files\dotnet\sdk]

Installed runtimes:
  Microsoft.NETCore.App 10.0.0
  Microsoft.AspNetCore.App 10.0.0
```

## I comandi principali: confronto con Maven/Gradle

| Operazione | Maven | Gradle | `dotnet` CLI |
|---|---|---|---|
| Crea nuovo progetto | `mvn archetype:generate` | `gradle init` | `dotnet new <template>` |
| Compila | `mvn compile` | `gradle compileJava` | `dotnet build` |
| Esegui i test | `mvn test` | `gradle test` | `dotnet test` |
| Esegui l'applicazione | `mvn spring-boot:run` | `gradle bootRun` | `dotnet run` |
| Ripristina dipendenze | `mvn dependency:resolve` | `gradle dependencies` | `dotnet restore` |
| Crea l'artefatto | `mvn package` | `gradle build` | `dotnet publish` |
| Aggiungi una dipendenza | (modifica `pom.xml`) | (modifica `build.gradle`) | `dotnet add package <nome>` |
| Rimuovi una dipendenza | (modifica `pom.xml`) | (modifica `build.gradle`) | `dotnet remove package <nome>` |
| Lista dipendenze | `mvn dependency:tree` | `gradle dependencies` | `dotnet list package` |

## Creare un progetto: i template

Il comando `dotnet new` usa **template** predefiniti, analoghi agli archetipi Maven. I più usati:

```bash
# Web API (quello che useremo nel corso)
dotnet new webapi -n OrderService

# Libreria di classi
dotnet new classlib -n OrderService.Core

# Progetto di test xUnit
dotnet new xunit -n OrderService.Tests

# Console application
dotnet new console -n MyTool

# Solution file (equivalente del pom.xml padre nei multi-module Maven)
dotnet new sln -n OrderService
```

Vedi tutti i template disponibili:

```bash
dotnet new list
```

---

## Il ciclo di vita di un progetto tipico

Partendo da zero, il flusso è questo (lo ritroverai identico nel capitolo 4):

```bash
# 1. Creo la solution e i progetti
dotnet new sln -n OrderService
dotnet new webapi -n OrderService.Api
dotnet new classlib -n OrderService.Core

# 2. Aggiungo i progetti alla solution
dotnet sln add OrderService.Api/OrderService.Api.csproj
dotnet sln add OrderService.Core/OrderService.Core.csproj

# 3. Aggiungo un riferimento tra progetti (Api dipende da Core)
dotnet add OrderService.Api/OrderService.Api.csproj reference OrderService.Core/OrderService.Core.csproj

# 4. Aggiungo un pacchetto NuGet
dotnet add OrderService.Api package Microsoft.EntityFrameworkCore

# 5. Ripristino le dipendenze (come `mvn dependency:resolve`)
dotnet restore

# 6. Compilo
dotnet build

# 7. Eseguo in modalità sviluppo
dotnet run --project OrderService.Api

# 8. Eseguo i test
dotnet test
```

## `dotnet build` vs `dotnet publish`

Una distinzione che in Maven non esiste in modo così netto:

- **`dotnet build`**: compila il codice, produce la `.dll` nella cartella `bin/Debug`. Serve per lo sviluppo locale.
- **`dotnet publish`**: prepara l'output per la distribuzione, ottimizza, risolve tutte le dipendenze runtime. Equivale a `mvn package` ma con più opzioni.

Esempi di `dotnet publish`:

```bash
# Pubblicazione standard (richiede il runtime installato sul server)
dotnet publish -c Release -o ./output

# Self-contained: il runtime è incluso, nessuna installazione sul server
dotnet publish -c Release -r linux-x64 --self-contained -o ./output

# Single-file: tutto in un unico eseguibile
dotnet publish -c Release -r linux-x64 --self-contained -p:PublishSingleFile=true -o ./output
```

L'opzione **self-contained** non ha un equivalente diretto in Maven/Gradle (ci si avvicina con `spring-boot:build-image` o uber-jar, ma non è la stessa cosa).

## Gestione di più SDK installati: `global.json`

Hai installato .NET 8 e .NET 10? Puoi forzare la versione usata in una directory con un file `global.json`:

```json
{
  "sdk": {
    "version": "10.0.100",
    "rollForward": "latestMinor"
  }
}
```

Equivale al `<java.version>` in Maven o al `sourceCompatibility` in Gradle, ma a livello di SDK intero. Utile in CI/CD per garantire che tutti usino esattamente la stessa versione.

## Global tools: strumenti installati a livello di sistema

`dotnet` supporta i **global tools**, pacchetti NuGet eseguibili installati a livello di utente — come i plugin Maven installati globalmente o i task Gradle globali.

```bash
# Installa uno strumento globale (es. EF Core CLI)
dotnet tool install --global dotnet-ef

# Aggiorna
dotnet tool update --global dotnet-ef

# Lista strumenti installati
dotnet tool list --global

# Usa lo strumento appena installato
dotnet ef migrations add InitialCreate
```

I global tools vengono installati in `~/.dotnet/tools` e aggiunti al PATH automaticamente.

---

## In sintesi

- **`dotnet`** è il tuo nuovo `mvn`/`gradle`: un unico comando per tutto il ciclo di vita.
- Installa l'**SDK** in sviluppo, il solo **runtime** (o niente, con self-contained) in produzione.
- I **template** (`dotnet new`) sostituiscono gli archetipi Maven.
- `dotnet build` è per lo sviluppo locale; `dotnet publish` prepara l'artefatto per la distribuzione.
- Il file **`global.json`** fissa la versione dell'SDK, come `<java.version>` in Maven.
- I **global tools** sono l'equivalente dei plugin Maven installati a livello di utente.
