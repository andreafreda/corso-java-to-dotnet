# 1.7 [Novità] .NET 10: file-based programs e `#:package`

> **Analogia Java**: nessun equivalente diretto — è una di quelle cose che in Java semplicemente non esistono (ancora)

---

## Un po' di contesto

Ogni anno esce una nuova versione di .NET, e ogni anno porta qualcosa di nuovo. La maggior parte delle novità sono miglioramenti incrementali — performance, API, fix. Ma ogni tanto arriva qualcosa che cambia il modo di scrivere codice in modo più visibile.

**.NET 10** ha introdotto i **file-based programs**, una feature pensata per ridurre al minimo il codice da scrivere per avviare un progetto o uno script. Se hai mai provato Python o Node.js e hai pensato "perché in Java devo scrivere tre file e una classe prima di stampare 'hello world'?", questa feature è la risposta di .NET a quella domanda.

---

## Top-level statements: il passo precedente

Prima di arrivare ai file-based programs, vale la pena ricordare che già da .NET 6 esiste una feature chiamata **top-level statements**: puoi scrivere codice direttamente nel file `Program.cs` senza dichiarare una classe o un metodo `Main`.

```csharp
// Program.cs — valido da .NET 6 in poi
Console.WriteLine("Hello, world!");
```

Niente `class Program`, niente `static void Main(string[] args)`. Il compilatore lo aggiunge per te. Se vieni da Java, probabilmente hai già avuto un momento di incredulità la prima volta che hai visto questo.

---

## File-based programs: lo step successivo

Con .NET 10 si fa un passo in più. Un **file-based program** è un singolo file `.cs` che puoi eseguire direttamente con `dotnet run`, **senza creare un progetto**, senza `.csproj`, senza cartelle, senza nulla.

Crei un file `hello.cs`:

```csharp
Console.WriteLine("Hello from a single file!");
```

E lo esegui:

```bash
dotnet run hello.cs
```

Fatto. Nessun `dotnet new`, nessun `dotnet build`, nessuna struttura di progetto. È pensato per script veloci, prototipazione, esempi didattici — esattamente come eseguiresti uno script Python con `python hello.py`.

---

## `#:package` — importare pacchetti NuGet senza `.csproj`

La parte più interessante arriva qui. Nei file-based programs puoi importare pacchetti NuGet direttamente nel codice, con una direttiva speciale in cima al file:

```csharp
#:package Humanizer@2.14.1

using Humanizer;

var data = DateTime.UtcNow.AddDays(-3);
Console.WriteLine(data.Humanize()); // "3 days ago"
```

```bash
dotnet run script.cs
```

.NET scarica automaticamente il pacchetto NuGet, lo compila e lo esegue. Niente `dotnet add package`, niente `dotnet restore`, niente `.csproj`. È l'equivalente di `pip install humanizer && python script.py` condensato in un unico comando.

Puoi aggiungere più pacchetti:

```csharp
#:package Humanizer@2.14.1
#:package Spectre.Console@0.49.0

using Humanizer;
using Spectre.Console;

AnsiConsole.MarkupLine("[green]Hello[/] from .NET 10!");
Console.WriteLine(DateTime.UtcNow.AddHours(-2).Humanize());
```

---

## Quando usarli (e quando no)

I file-based programs sono ottimi per:
- Script di automazione veloci
- Prototipi e proof of concept
- Esempi didattici (come quelli di questo corso)
- Tool interni usa-e-getta

**Non sono pensati per microservizi o applicazioni in produzione.** Appena il tuo progetto cresce oltre uno script, torni alla struttura classica con `.csproj` e solution. La feature non sostituisce nulla — aggiunge uno strumento comodo per i casi d'uso giusti.

In Java hai probabilmente incontrato **JShell** (introdotto con Java 9): una REPL interattiva da terminale dove puoi eseguire istruzioni Java al volo, senza creare una classe. È comoda per sperimentare, ma non ti permette di darle un file da eseguire come script. Per quello esiste **JBang**, un tool di terze parti che si avvicina di più all'esperienza dei file-based programs.

| | Java | .NET 10 |
|---|---|---|
| REPL interattiva | JShell (Java 9+, nativo) | `dotnet-repl` (tool esterno) |
| Esegui un file come script | JBang (tool esterno) | `dotnet run file.cs` (nativo) |

Su questo fronte .NET 10 fa un passo in più rispetto a Java: l'esecuzione di file singoli è nativa, senza installare nulla di extra.

---

## In sintesi

- I **file-based programs** ti permettono di eseguire un file `.cs` direttamente con `dotnet run file.cs`, senza creare un progetto.
- La direttiva **`#:package`** importa pacchetti NuGet al volo, senza `.csproj` né `dotnet restore`.
- Sono pensati per script, prototipi e didattica — non per applicazioni in produzione.
- Non hanno un equivalente diretto in Java nativo; il tool più simile nell'ecosistema Java è JBang.
