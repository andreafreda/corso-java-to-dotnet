# 1.5 La Solution (`.sln`) e i progetti multipli

> **Analogia Java**: equivalente del multi-module Maven project — un contenitore che raggruppa più progetti correlati

---

## Il concetto che già conosci

Se hai lavorato su un progetto Java di medie dimensioni, probabilmente hai incontrato il **multi-module Maven**: un `pom.xml` padre che dichiara una lista di moduli (`<modules>`), ognuno con il suo `pom.xml` figlio. Puoi buildare tutto insieme dall'alto oppure entrare in un modulo specifico e buildare solo quello.

In .NET la stessa idea prende il nome di **Solution**, e vive in un file con estensione `.sln`. La Solution non contiene codice — è solo una mappa che dice "questi sono i progetti che appartengono a questa soluzione". Il motore di build (MSBuild) la usa per capire l'ordine di compilazione e le dipendenze tra progetti.

---

## Come è fatto un file `.sln`

Onestamente? È un formato testuale un po' brutto, pensato per essere letto da Visual Studio più che dagli esseri umani. Non ti preoccupare di capirlo a fondo — raramente lo modificherai a mano.

```
Microsoft Visual Studio Solution File, Format Version 12.00
Project("{FAE04EC0-301F-11D3-BF4B-00C04F79EFBC}") = "OrderService.Api",
    "OrderService.Api\OrderService.Api.csproj", "{A1B2C3D4-...}"
EndProject
Project("{FAE04EC0-301F-11D3-BF4B-00C04F79EFBC}") = "OrderService.Core",
    "OrderService.Core\OrderService.Core.csproj", "{E5F6G7H8-...}"
EndProject
```

Quei GUID sembrano intimidatori ma li genera automaticamente la CLI. Tutto quello che devi sapere è che ogni riga `Project(...)` punta a un `.csproj`. Userai sempre `dotnet sln` per modificare questo file, mai a mano.

---

## Confronto con Maven multi-module

| Concetto | Maven | .NET Solution |
|---|---|---|
| File contenitore | `pom.xml` padre | `NomeProgetto.sln` |
| Dichiarazione moduli | `<modules><module>ordine</module></modules>` | `dotnet sln add Ordine/Ordine.csproj` |
| Build di tutto | `mvn package` dalla root | `dotnet build` dalla root |
| Build di un modulo | `mvn package -pl ordine` | `dotnet build OrderService.Api` |
| Dipendenza tra moduli | `<dependency>` con `groupId` interno | `<ProjectReference>` nel `.csproj` |

Una differenza importante: in Maven il `pom.xml` padre gestisce anche versioni e configurazioni condivise. In .NET queste responsabilità sono separate — la `.sln` è solo la mappa dei progetti, mentre le configurazioni condivise vanno nel `Directory.Build.props` che abbiamo visto nel 1.3.

## Come si lavora con le Solution da CLI

```bash
# Crea una nuova solution
dotnet new sln -n OrderService

# Crea i progetti
dotnet new webapi -n OrderService.Api -o OrderService.Api
dotnet new classlib -n OrderService.Core -o OrderService.Core
dotnet new classlib -n OrderService.Infrastructure -o OrderService.Infrastructure
dotnet new xunit -n OrderService.Tests -o OrderService.Tests

# Aggiunge i progetti alla solution
dotnet sln add OrderService.Api/OrderService.Api.csproj
dotnet sln add OrderService.Core/OrderService.Core.csproj
dotnet sln add OrderService.Infrastructure/OrderService.Infrastructure.csproj
dotnet sln add OrderService.Tests/OrderService.Tests.csproj

# Vedi i progetti nella solution
dotnet sln list

# Rimuovi un progetto dalla solution (non lo cancella dal disco)
dotnet sln remove OrderService.Tests/OrderService.Tests.csproj
```

## Una struttura tipica multi-progetto

Questa è la struttura che useremo per `OrderService` nel capitolo 4, e che ritrovi in quasi tutti i progetti .NET professionali:

```
OrderService.sln
│
├── OrderService.Api/           ← il progetto Web API (entry point)
│   └── OrderService.Api.csproj
│
├── OrderService.Core/          ← dominio, entità, interfacce (nessuna dipendenza esterna)
│   └── OrderService.Core.csproj
│
├── OrderService.Infrastructure/← implementazioni: EF Core, HTTP client, Service Bus
│   └── OrderService.Infrastructure.csproj
│
└── OrderService.Tests/         ← test unitari e di integrazione
    └── OrderService.Tests.csproj
```

Le dipendenze tra progetti scorrono in una sola direzione:

```
Api  →  Infrastructure  →  Core
Tests  →  (tutti)
```

`Core` non dipende da nessun altro progetto interno — è il cuore del dominio, puro e senza dipendenze esterne. Questo schema si chiama **Clean Architecture** (o architettura a cipolla) e lo vedremo in dettaglio nel capitolo 4. Per ora tienilo a mente come riferimento.

---

## Progetto singolo o multi-progetto?

Non è detto che tu debba sempre usare una struttura multi-progetto. Per un microservizio piccolo, un singolo `.csproj` senza `.sln` va benissimo:

```bash
dotnet new webapi -n OrderService
cd OrderService
dotnet run
```

La `.sln` diventa utile quando hai più progetti che devono lavorare insieme — tipicamente quando separi il dominio dall'infrastruttura, o quando aggiungi un progetto di test separato. È una scelta architetturale, non un obbligo.

Un buon criterio pratico: parti con un singolo progetto, aggiungi la struttura multi-progetto quando ne senti il bisogno. Non costruire complessità in anticipo.

---

## In sintesi

- La **Solution** (`.sln`) è il contenitore di tutti i progetti — come il `pom.xml` padre in Maven, ma separato dalle configurazioni condivise.
- Non modificare mai il file `.sln` a mano: usa sempre `dotnet sln add/remove`.
- Le dipendenze tra progetti si dichiarano con `<ProjectReference>` nel `.csproj`.
- La struttura **Api / Core / Infrastructure / Tests** è lo schema più diffuso nei microservizi .NET — la ritroveremo nel capitolo 4.
- Per microservizi semplici un singolo progetto è più che sufficiente: aggiungi struttura quando serve davvero.
