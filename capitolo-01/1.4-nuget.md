# 1.4 NuGet: il gestore di pacchetti

> **Analogia Java**: equivalente di Maven Central — il registry pubblico da cui scarichi le dipendenze del tuo progetto

---

## Il concetto che già conosci

In Java, quando aggiungi una dipendenza nel `pom.xml`, Maven la scarica da **Maven Central** (o da un registry privato come Nexus/Artifactory). Il meccanismo è trasparente: dichiari cosa vuoi, il tool lo scarica, lo mette in cache locale, e lo aggiunge al classpath.

In .NET funziona esattamente così, con **NuGet** come registry pubblico e `dotnet restore` come comando che effettua il download. Se hai già lavorato con Maven Central, ti sentirai subito a casa.

---

## NuGet.org: il registry pubblico

Il registry ufficiale è **[nuget.org](https://www.nuget.org)** — l'equivalente di Maven Central. Puoi cercare pacchetti, leggere la documentazione, controllare le versioni disponibili e vedere quante volte un pacchetto è stato scaricato.

I pacchetti NuGet hanno estensione `.nupkg` — un archivio ZIP che contiene l'assembly compilato, i metadati e, spesso, i simboli di debug. L'equivalente del `.jar` in Java.

---

## Aggiungere una dipendenza

Hai due modi, esattamente come in Maven:

**Via CLI** (consigliato):
```bash
dotnet add package Serilog.AspNetCore
dotnet add package Microsoft.EntityFrameworkCore --version 10.0.0
```

**Modificando direttamente il `.csproj`** (come editare `pom.xml` a mano):
```xml
<ItemGroup>
  <PackageReference Include="Serilog.AspNetCore" Version="8.0.0" />
  <PackageReference Include="Microsoft.EntityFrameworkCore" Version="10.0.0" />
</ItemGroup>
```

Dopo aver modificato il `.csproj` a mano, esegui `dotnet restore` per scaricare i pacchetti — come `mvn dependency:resolve`.

---

## La cache locale

NuGet mantiene una cache locale dei pacchetti scaricati, come la `~/.m2/repository` di Maven:

| | Maven | NuGet |
|---|---|---|
| Cache locale | `~/.m2/repository` | `~/.nuget/packages` |
| Pulire la cache | `mvn dependency:purge-local-repository` | `dotnet nuget locals all --clear` |

Se vuoi vedere dove NuGet tiene i file:
```bash
dotnet nuget locals all --list
```

## Versioning: SemVer e range di versione

NuGet usa **Semantic Versioning** (SemVer), come npm — e come Maven. La differenza è nel modo in cui si specificano i range:

| Sintassi | Significato |
|---|---|
| `Version="8.0.0"` | Esattamente 8.0.0 (versione minima in realtà) |
| `Version="[8.0.0]"` | Esattamente 8.0.0 (versione fissa) |
| `Version="[8.0.0, 9.0.0)"` | >= 8.0.0 e < 9.0.0 |
| `Version="*"` | L'ultima versione disponibile |

Nella pratica, in quasi tutti i progetti reali vedrai versioni fisse (`Version="8.0.0"`) — meno sorprese in produzione, come in Maven con versioni non-SNAPSHOT.

## Pacchetti Microsoft vs community

Una cosa utile da sapere: molti pacchetti fondamentali sono mantenuti direttamente da Microsoft e distribuiti con lo stesso ciclo di rilascio di .NET. Con ogni nuova versione di .NET escono insieme anche `Microsoft.EntityFrameworkCore 10.0.0`, `Microsoft.AspNetCore.* 10.0.0`, e così via.

Per i pacchetti di terze parti, il panorama è simile a quello Java:
- Alcuni sono open source e gratuiti (Serilog, FluentValidation, AutoMapper…)
- Alcuni sono freemium con una licenza commerciale per uso enterprise (lo vediamo meglio nel 1.8)
- Alcuni sono puramente commerciali

## Registry privati: l'equivalente di Nexus/Artifactory

In azienda raramente si usa solo il registry pubblico. NuGet supporta registry privati, utile per:
- Pacchetti interni che non vuoi pubblicare su nuget.org
- Mirror del registry pubblico per ambienti senza accesso a internet
- Pacchetti commerciali con licenza

I registry privati più comuni sono **Azure Artifacts** (se sei nell'ecosistema Azure/DevOps) e **GitHub Packages**. La configurazione avviene tramite il file `nuget.config`:

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
    <add key="MyPrivateFeed" value="https://pkgs.dev.azure.com/myorg/_packaging/myfeed/nuget/v3/index.json" />
  </packageSources>
</configuration>
```

Equivale al blocco `<repositories>` nel `pom.xml`.

## Comandi NuGet utili

```bash
# Aggiungi un pacchetto
dotnet add package FluentValidation.AspNetCore

# Rimuovi un pacchetto
dotnet remove package FluentValidation.AspNetCore

# Lista le dipendenze del progetto
dotnet list package

# Cerca pacchetti obsoleti
dotnet list package --outdated

# Cerca vulnerabilità note nelle dipendenze
dotnet list package --vulnerable

# Ripristina tutte le dipendenze
dotnet restore
```

`dotnet list package --vulnerable` non ha un equivalente immediato in Maven base — è come avere OWASP Dependency-Check integrato nella CLI.

---

## Il file `packages.lock.json`

Opzionale ma consigliato in team: NuGet può generare un **lock file** che fissa le versioni transitive di tutte le dipendenze — funzionalità che Maven non offre nativamente.

Per abilitarlo, aggiungi questa proprietà al `.csproj` (o al `Directory.Build.props`):

```xml
<PropertyGroup>
  <RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>
</PropertyGroup>
```

Maven non ha un lock file nativo — è uno dei punti in cui NuGet si avvicina di più all'ecosistema JavaScript moderno.

---

## In sintesi

- **NuGet** è il Maven Central di .NET — il registry pubblico è nuget.org.
- Le dipendenze si dichiarano con `<PackageReference>` nel `.csproj` o con `dotnet add package`.
- La cache locale è in `~/.nuget/packages`, come `~/.m2` per Maven.
- Per i registry privati si usa `nuget.config`, come `<repositories>` in Maven.
- `dotnet list package --outdated` e `--vulnerable` sono comodissimi e integrati nativamente.
- Il `packages.lock.json` è opzionale ma consigliato per build riproducibili in team.
