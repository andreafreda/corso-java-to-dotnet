# 1.1 Il runtime: CLR vs JVM

> **Analogia Java**: entrambi compilano in bytecode intermedio, ma il CLR usa IL (Intermediate Language)

---

## Il concetto che già conosci

Se vieni da Java, hai già interiorizzato un'idea fondamentale: il tuo codice sorgente non gira direttamente sulla CPU. Viene prima compilato in un formato intermedio — il **bytecode** — e poi eseguito da una macchina virtuale che si interpone tra il codice e l'hardware.

Questo modello ti ha regalato la portabilità ("write once, run anywhere") e una serie di servizi runtime come garbage collection, gestione della memoria e sicurezza. .NET funziona esattamente con la stessa filosofia, con nomi diversi e qualche differenza importante sotto il cofano.

---

## JVM e CLR: la stessa idea, due implementazioni

| Concetto | Java | .NET |
|---|---|---|
| Macchina virtuale | JVM (Java Virtual Machine) | CLR (Common Language Runtime) |
| Formato intermedio | Bytecode (`.class`) | IL / CIL (Intermediate Language) |
| Compilazione al volo | JIT Compiler (HotSpot) | JIT Compiler (RyuJIT) |
| Garbage Collector | G1, ZGC, Shenandoah... | GC generazionale (con modalità server/workstation) |
| Metadati del tipo | Reflection API | Reflection + Type metadata |
| Standard di runtime | JVM Spec | ECMA-335 (CLI Standard) |

Il tuo codice C# viene compilato dal compilatore **Roslyn** in **IL** (Intermediate Language, a volte chiamato MSIL o CIL). Il file risultante è un assembly `.dll` o `.exe` che contiene IL + metadati. Quando lo esegui, il CLR interviene e compila IL → codice macchina nativo tramite il JIT compiler **RyuJIT**.

```
Codice C#  →  [Roslyn]  →  IL (.dll)  →  [RyuJIT]  →  Codice macchina nativo
Codice Java →  [javac]  →  bytecode (.class) →  [JIT HotSpot]  →  Codice macchina nativo
```

---

## Una differenza che conta: multi-linguaggio by design

La JVM supporta altri linguaggi (Kotlin, Scala, Groovy), ma è nata per Java. Il CLR invece è stato progettato sin dall'inizio per ospitare **più linguaggi di primo livello**: C#, F#, VB.NET compilano tutti in IL e interoperano senza frizioni nello stesso progetto.

Questo si traduce in una cosa concreta: puoi avere un progetto .NET con una libreria scritta in F# (magari per la parte di calcolo funzionale) e consumarla direttamente da C# senza wrapper o serializzazioni.

## .NET 10 e il runtime moderno

Quando si parla di .NET oggi, si intende **.NET Core** e la sua evoluzione — non il vecchio **.NET Framework** (che esisteva solo su Windows ed è ormai in manutenzione). Il percorso storico, per orientarsi:

```
.NET Framework 1.0–4.8   →  Solo Windows, legacy
.NET Core 1.0–3.1        →  Cross-platform, open source, nuova architettura
.NET 5                   →  Unificazione: "Core" sparisce dal nome
.NET 6, 7, 8, 9, 10     →  Rilasci annuali (LTS ogni 2 anni)
```

**.NET 10** (rilasciato a novembre 2025) è l'ultima versione LTS. È cross-platform: gira su Windows, Linux e macOS con lo stesso binario e le stesse API. Il runtime è open source su GitHub.

> **Nota per chi viene da Spring Boot**: se il tuo team usava `.NET Framework` alcuni anni fa e ne è rimasto scottato (deploy solo su IIS, solo Windows, performance meno brillanti), sappi che .NET moderno è una piattaforma completamente diversa.

---

## In sintesi

- Il CLR è l'equivalente .NET della JVM: astrae l'hardware, gestisce la memoria, esegue codice intermedio via JIT.
- Il formato intermedio si chiama **IL** invece di bytecode, ma il concetto è identico.
- Il compilatore di C# si chiama **Roslyn**; il JIT si chiama **RyuJIT**.
- .NET è multi-linguaggio by design: C#, F#, VB.NET coesistono sulla stessa runtime.
- **.NET moderno (5+)** è cross-platform e open source — dimentica .NET Framework.
