# 1.3 Il file `.csproj` e MSBuild

> **Analogia Java**: equivalente del `pom.xml` — descrive il progetto, le dipendenze e le istruzioni di build

---

## Il concetto che già conosci

Se hai lavorato con Maven, hai passato un bel po' di tempo a guardare `pom.xml`: un file nella root del progetto che descrive tutto — il nome dell'artefatto, la versione di Java, le dipendenze, i plugin, le fasi di build.

In .NET questo file si chiama **`.csproj`** (o `.fsproj` per F#, `.vbproj` per VB.NET) ed è processato da **MSBuild**, il sistema di build di Microsoft. Buona notizia: è XML, come il `pom.xml`, ma molto più snello di quanto ricordi.

---

## Un `.csproj` minimo

Questo è un file `.csproj` reale per una Web API .NET 10 — niente di tagliato:

```xml
<Project Sdk="Microsoft.NET.Sdk.Web">

  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>

</Project>
```

Tre righe di configurazione. Tutto il resto viene inferito dall'SDK. Se arrivi da un `pom.xml` con 200 righe, benvenuto nel futuro.

---

## Confronto diretto con Maven

| Concetto | Maven (`pom.xml`) | `.csproj` |
|---|---|---|
| Target di compilazione | `<java.version>17</java.version>` | `<TargetFramework>net10.0</TargetFramework>` |
| Nome artefatto | `<artifactId>order-service</artifactId>` | Nome del file `.csproj` |
| Versione | `<version>1.0.0</version>` | `<Version>1.0.0</Version>` |
| Dipendenze | `<dependencies>` | `<ItemGroup>` con `<PackageReference>` |
| Dipendenza tra moduli | `<dependency>` con `<type>pom</type>` | `<ProjectReference>` |
| Plugin/task custom | `<plugins>` | `<Target>` MSBuild |

## Le sezioni principali

### `PropertyGroup` — le proprietà del progetto

```xml
<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>   <!-- versione .NET target -->
  <Nullable>enable</Nullable>                  <!-- abilita nullable reference types -->
  <ImplicitUsings>enable</ImplicitUsings>      <!-- using impliciti (System, Linq, ecc.) -->
  <Version>1.0.0</Version>                     <!-- versione dell'artefatto -->
  <AssemblyName>OrderService</AssemblyName>    <!-- nome dell'assembly prodotto -->
  <RootNamespace>OrderService</RootNamespace>  <!-- namespace radice -->
</PropertyGroup>
```

`Nullable` e `ImplicitUsings` sono opzioni moderne di C# che vedrai sempre abilitate nei nuovi progetti — le approfondiremo nel capitolo 2.

### `ItemGroup` con `PackageReference` — le dipendenze NuGet

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.EntityFrameworkCore" Version="10.0.0" />
  <PackageReference Include="FluentValidation.AspNetCore" Version="11.3.0" />
  <PackageReference Include="Serilog.AspNetCore" Version="8.0.0" />
</ItemGroup>
```

Identico al blocco `<dependencies>` di Maven. La versione è fissa — niente range di versione per default, come in Maven. `dotnet restore` scarica tutto da NuGet.

### `ProjectReference` — dipendenze tra progetti della stessa solution

```xml
<ItemGroup>
  <ProjectReference Include="..\OrderService.Core\OrderService.Core.csproj" />
</ItemGroup>
```

Equivale alla dipendenza da un modulo interno in un progetto multi-module Maven. MSBuild risolve automaticamente l'ordine di compilazione.

## MSBuild: il motore sotto il cofano

**MSBuild** è il sistema di build che processa il `.csproj`. Quando esegui `dotnet build`, stai invocando MSBuild. Non devi conoscerlo in profondità per lavorare bene, ma è utile sapere che esiste per capire alcuni messaggi di errore.

La cosa più importante: l'SDK `.NET` definisce una serie di **target MSBuild predefiniti** (Restore, Build, Test, Publish) — esattamente come Maven definisce le fasi del lifecycle. Puoi aggiungere target custom direttamente nel `.csproj`:

```xml
<Target Name="PrintVersion" AfterTargets="Build">
  <Message Text="Versione: $(Version)" Importance="high" />
</Target>
```

Nella pratica quotidiana non scriverai quasi mai target MSBuild a mano — il confronto con i plugin Maven è più storico che operativo.

---

## Multi-targeting: compilare per più framework

Una feature che Maven non ha nativamente: con un solo `.csproj` puoi compilare per più versioni di .NET (o anche .NET Framework) contemporaneamente.

```xml
<PropertyGroup>
  <!-- Nota: TargetFrameworks al plurale -->
  <TargetFrameworks>net8.0;net10.0</TargetFrameworks>
</PropertyGroup>
```

Utile quando scrivi una libreria che deve supportare più versioni. `dotnet build` produrrà un output per ciascun framework target.

---

## Directory.Build.props: proprietà condivise tra tutti i progetti

Se hai una solution con dieci progetti e vuoi che abbiano tutti la stessa versione di .NET e le stesse impostazioni, non devi copiarle ovunque. Crea un file `Directory.Build.props` nella root della solution:

```xml
<Project>
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
  </PropertyGroup>
</Project>
```

MSBuild lo importa automaticamente in ogni `.csproj` della directory. È l'equivalente del `pom.xml` padre in un multi-module Maven, ma senza dover dichiarare esplicitamente il parent in ogni file figlio.

---

## In sintesi

- Il `.csproj` è il tuo nuovo `pom.xml` — XML, ma molto più snello.
- **`PropertyGroup`** configura il progetto (framework target, versione, opzioni del compilatore).
- **`PackageReference`** dichiara le dipendenze NuGet, come `<dependency>` in Maven.
- **`ProjectReference`** collega i progetti interni della stessa solution.
- **MSBuild** è il motore di build — di solito lo usi indirettamente tramite `dotnet build`.
- **`Directory.Build.props`** centralizza le proprietà comuni, come il `pom.xml` padre in Maven.
