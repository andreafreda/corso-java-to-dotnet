## Prima di iniziare: async/await vs multithreading

Se hai letto il 2.7, potresti chiederti: "ma non avevamo giГ  risolto la concorrenza con async/await?". La risposta ГЁ: dipende dal problema.

- **`async/await`** risolve la **concorrenza I/O-bound** вЂ” aspettare risposte da rete, database, file system senza bloccare thread. Г€ il 90% dei casi nelle Web API.
- **Multithreading** risolve la **concorrenza CPU-bound** вЂ” lavoro parallelo su piГ№ core, accesso condiviso a risorse in memoria, sincronizzazione tra thread. Г€ il restante 10%, ma quando serve, serve davvero.

Se stai pensando "ma io faccio solo Web API, perchГ© dovrei preoccuparmi?" вЂ” perchГ© le cache in memoria, i contatori, le code in-process e certi pattern di background processing toccano tutti questo territorio.

## `Thread` вЂ” il thread esplicito

In Java crei un thread con `new Thread(runnable)`. In C# ГЁ praticamente identico:

```csharp
// C# вЂ” Thread esplicito
var thread = new Thread(() =>
{
    Console.WriteLine($"Thread {Thread.CurrentThread.ManagedThreadId} in esecuzione");
});
thread.Start();
thread.Join(); // aspetta che finisca вЂ” come thread.join() in Java

// Thread con nome (utile per il debug)
var thread2 = new Thread(Lavora) { Name = "WorkerThread", IsBackground = true };
thread2.Start();
```

`IsBackground = true` significa che il thread non impedisce all'applicazione di terminare вЂ” equivalente di `thread.setDaemon(true)` in Java.

Nella pratica moderna raramente crei `Thread` esplicitamente вЂ” usi `Task.Run` per il lavoro CPU-bound:

```csharp
// Preferibile al Thread esplicito per lavoro CPU-bound
await Task.Run(() => ElaborazioneIntensiva());
```

## `lock` вЂ” la sezione critica

In Java usi `synchronized` per proteggere una sezione critica. In C# usi `lock`:

```csharp
private readonly object _lock = new object();
private int _contatore = 0;

// C#
public void Incrementa()
{
    lock (_lock)
    {
        _contatore++;
    }
}
```

```java
// Java вЂ” per confronto
private final Object lock = new Object();
private int contatore = 0;

public void incrementa() {
    synchronized (lock) {
        contatore++;
    }
}
```

Praticamente identici. La differenza: in Java puoi usare `synchronized` anche sulla firma del metodo (`public synchronized void incrementa()`), in C# non esiste questa sintassi вЂ” usi sempre `lock` nel corpo.

`lock` in C# ГЁ zucchero sintattico per `Monitor.Enter` / `Monitor.Exit` вЂ” esattamente come `synchronized` in Java ГЁ zucchero per `monitorenter` / `monitorexit` nella JVM.

> **Regola d'oro**: l'oggetto su cui fai `lock` deve essere `private`, `readonly`, e non deve essere un tipo di valore (`int`, `struct`). Usare `lock(this)` o `lock("stringa")` ГЁ un antipattern вЂ” esattamente come in Java.

## `Monitor` вЂ” controllo fine della sincronizzazione

`Monitor` ГЁ l'equivalente di `Object.wait()` / `Object.notify()` in Java вЂ” ti dГ  controllo esplicito su enter, exit, wait e pulse:

```csharp
Monitor.Enter(_lock);
try
{
    while (!_condizione)
        Monitor.Wait(_lock);     // equivalente di lock.wait() in Java

    // lavoro protetto
    Monitor.PulseAll(_lock);     // equivalente di lock.notifyAll()
}
finally
{
    Monitor.Exit(_lock);
}
```

Nella pratica lo usi raramente direttamente вЂ” `lock` copre la maggior parte dei casi, e per i pattern producer/consumer esistono strutture dati dedicate come `Channel<T>` (ne parliamo tra poco).

## `Interlocked` вЂ” operazioni atomiche

Equivalente di `AtomicInteger`, `AtomicLong` in Java вЂ” operazioni atomiche su tipi primitivi senza bisogno di lock:

```csharp
private int _contatore = 0;

// Incremento atomico вЂ” thread-safe senza lock
Interlocked.Increment(ref _contatore);
Interlocked.Decrement(ref _contatore);

// Add atomico
Interlocked.Add(ref _contatore, 5);

// CompareAndSwap вЂ” come compareAndSet in Java
int vecchio = Interlocked.CompareExchange(ref _contatore, nuovoValore, valorAtteso);

// Read atomica su long (necessaria su sistemi 32-bit)
long valore = Interlocked.Read(ref _contatoreLong);
```

In Java usi `AtomicInteger` come oggetto wrapper (`new AtomicInteger(0)`). In C# `Interlocked` ГЁ una classe statica che opera su variabili normali passate per `ref` вЂ” piГ№ leggero, nessuna allocazione.

## `SemaphoreSlim` вЂ” limitare l'accesso concorrente

Equivalente di `Semaphore` in Java, ma ottimizzato per uso nello stesso processo e compatibile con `async/await`:

```csharp
// Massimo 3 thread/task concurrent
private readonly SemaphoreSlim _semaforo = new SemaphoreSlim(3);

public async Task ElaboraAsync(string item)
{
    await _semaforo.WaitAsync();  // aspetta un posto libero вЂ” non blocca il thread
    try
    {
        await FaiQualcosaAsync(item);
    }
    finally
    {
        _semaforo.Release();
    }
}
```

Il punto chiave ГЁ `WaitAsync()` вЂ” a differenza di `Semaphore.acquire()` in Java (che blocca il thread), `WaitAsync()` ГЁ asincrono e libera il thread mentre aspetta. Questo lo rende molto utile per throttling di chiamate HTTP, accesso a risorse limitate, ecc.

## `ReaderWriterLockSlim` вЂ” letture concorrenti, scritture esclusive

Equivalente di `ReentrantReadWriteLock` in Java вЂ” permette letture simultanee ma scritture esclusive:

```csharp
private readonly ReaderWriterLockSlim _rwLock = new ReaderWriterLockSlim();
private Dictionary<int, Ordine> _cache = new();

public Ordine? Leggi(int id)
{
    _rwLock.EnterReadLock();
    try { return _cache.GetValueOrDefault(id); }
    finally { _rwLock.ExitReadLock(); }
}

public void Scrivi(int id, Ordine ordine)
{
    _rwLock.EnterWriteLock();
    try { _cache[id] = ordine; }
    finally { _rwLock.ExitWriteLock(); }
}
```

## Collezioni thread-safe

Come in Java (`ConcurrentHashMap`, `CopyOnWriteArrayList`вЂ¦), .NET ha un namespace dedicato:

| Java | C# | Uso |
|---|---|---|
| `ConcurrentHashMap<K,V>` | `ConcurrentDictionary<K,V>` | Dizionario thread-safe |
| `CopyOnWriteArrayList<T>` | `ImmutableList<T>` + swap | Lista immutabile |
| `LinkedBlockingQueue<T>` | `Channel<T>` | Producer/consumer asincrono |
| `ArrayBlockingQueue<T>` | `Channel<T>` con capacitГ  | Producer/consumer bounded |
| `ConcurrentLinkedQueue<T>` | `ConcurrentQueue<T>` | Coda lock-free |

`Channel<T>` merita una menzione speciale вЂ” ГЁ il modo moderno per fare producer/consumer in .NET, completamente asincrono:

```csharp
var channel = Channel.CreateUnbounded<Ordine>();

// Producer
await channel.Writer.WriteAsync(nuovoOrdine);
channel.Writer.Complete(); // segnala che non ci sono altri elementi

// Consumer
await foreach (var ordine in channel.Reader.ReadAllAsync())
{
    await ElaboraOrdineAsync(ordine);
}
```

Molto piГ№ pulito di `BlockingCollection<T>` o `LinkedBlockingQueue` вЂ” e non blocca i thread.

## `volatile` вЂ” visibilitГ  tra thread

Come in Java, `volatile` garantisce che le letture/scritture su una variabile siano visibili a tutti i thread senza caching in registro:

```csharp
private volatile bool _running = true;

public void Stop() => _running = false;

public void Run()
{
    while (_running)
    {
        // lavoro
    }
}
```

Stesso comportamento di `volatile` in Java. Stessa limitazione: garantisce visibilitГ , non atomicitГ  delle operazioni composte.

## In sintesi

- **`lock`** ГЁ il `synchronized` di C# вЂ” stessa idea, sintassi diversa, non disponibile sulla firma del metodo.
- **`Interlocked`** sostituisce `AtomicInteger`/`AtomicLong` вЂ” opera su variabili normali via `ref`, nessuna allocazione.
- **`SemaphoreSlim`** con `WaitAsync()` ГЁ il modo giusto per throttling asincrono вЂ” non blocca il thread come `Semaphore.acquire()` in Java.
- **`ConcurrentDictionary`** ГЁ il `ConcurrentHashMap` di .NET.
- **`Channel<T>`** ГЁ il modo moderno per producer/consumer вЂ” completamente asincrono, molto piГ№ pulito di `BlockingCollection`.
- Se stai scrivendo Web API e usi solo `async/await`, non avrai bisogno di quasi nulla di questo capitolo вЂ” ma sapere che esiste ГЁ importante per quando serve.
