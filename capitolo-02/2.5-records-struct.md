# 2.5 Records, struct e classi immutabili

> **Analogia Java**: equivalente di Java Records (Java 16+) e classi con `final` — ma con più opzioni e più potere

---

## Il problema dell'immutabilità in Java

In Java, creare un oggetto davvero immutabile richiede disciplina: tutti i campi `final`, nessun setter, costruttore che inizializza tutto, `equals()` e `hashCode()` scritti a mano (o generati dall'IDE). Dal Java 16 i **Records** hanno semplificato la cosa, ma restano limitati — non puoi estenderli, non puoi aggiungere logica facilmente.

In C# hai tre strumenti distinti per gestire l'immutabilità e la leggerezza degli oggetti: **struct**, **record** e **classi con init**. Ognuno ha il suo caso d'uso.

---

## `struct` — value type leggero

Lo `struct` è un **value type** — come abbiamo visto nel 2.3: vive sullo stack, viene copiato quando passato a un metodo o assegnato a una variabile. Non ha ereditarietà (a parte le interfacce).

```csharp
public struct Punto
{
    public double X { get; init; }
    public double Y { get; init; }

    public Punto(double x, double y)
    {
        X = x;
        Y = y;
    }

    public double Distanza => Math.Sqrt(X * X + Y * Y);
}

var p1 = new Punto(3, 4);
var p2 = p1;   // copia — p2 è indipendente da p1
p2 = new Punto(0, 0);
Console.WriteLine(p1.X); // 3 — p1 non è cambiato
```

**Quando usarlo**: per oggetti piccoli, immutabili, usati in grande quantità (coordinate, colori, intervalli di valori). Evita struct grandi — la copia ha un costo.

In Java non esiste un equivalente diretto: i tipi primitivi sono value type, ma non puoi creare i tuoi value type personalizzati.

---

## `record` — immutabilità con poco codice

I **record** sono la risposta C# ai Java Records — anzi, li precedono (introdotti in C# 9, 2020 vs Java 16, 2021). Sono classi pensate per contenere dati, con `Equals`, `GetHashCode`, `ToString` e la decostruzione generati automaticamente.

### Sintassi posizionale (la più compatta)

```csharp
public record Punto(double X, double Y);

var p = new Punto(3, 4);
Console.WriteLine(p);       // Punto { X = 3, Y = 4 }
Console.WriteLine(p.X);     // 3

// Equals strutturale — confronta i valori, non il riferimento
var p2 = new Punto(3, 4);
Console.WriteLine(p == p2); // true — stesso comportamento dei Java Records
```

### `with` — copia con modifiche

Una feature che i Java Records non hanno: l'espressione `with` crea una copia dell'oggetto con alcune property modificate, senza toccare l'originale.

```csharp
var originale = new Punto(3, 4);
var modificato = originale with { X = 10 };

Console.WriteLine(originale);  // Punto { X = 3, Y = 4 }
Console.WriteLine(modificato); // Punto { X = 10, Y = 4 }
```

Utilissimo nel DDD e nel CQRS quando devi creare versioni modificate di oggetti immutabili senza mutare lo stato.

### Record con corpo — quando serve logica

```csharp
public record Ordine(string Cliente, decimal Totale)
{
    // Validazione nel costruttore
    public Ordine : this(Cliente, Totale)
    {
        if (Totale < 0) throw new ArgumentException("Totale non può essere negativo");
    }

    // Property calcolata
    public decimal TotaleConIva => Totale * 1.22m;
}
```

### `record struct` — record che è anche value type

Da C# 10 puoi combinare i due mondi:

```csharp
public record struct Coordinate(double Lat, double Lon);
```

Value type + semantica di record. Utile per oggetti piccoli e immutabili usati in grande quantità.

## Classi immutabili con `init`

Se non vuoi un record ma vuoi comunque l'immutabilità, puoi usare `init` sulle properties (visto nel 2.4):

```csharp
public class Ordine
{
    public required string Cliente { get; init; }
    public required decimal Totale { get; init; }
    public DateTime CreatedAt { get; init; } = DateTime.UtcNow;
}

var ordine = new Ordine { Cliente = "Andrea", Totale = 99.99m };
ordine.Cliente = "Mario"; // ❌ errore — init è di sola lettura dopo la costruzione
```

La differenza rispetto ai record: non ottieni `Equals` strutturale e `ToString` automatici, ma hai più flessibilità (ereditarietà, logica complessa, integrazione con EF Core).

---

## Quando usare cosa

| Situazione | Strumento consigliato |
|---|---|
| DTO, Value Object DDD, oggetti dati puri | `record` |
| Oggetti piccoli usati in grande quantità (coordinate, range) | `struct` o `record struct` |
| Entità di dominio con logica, immutabili dopo la creazione | Classe con `init` e `required` |
| Entità EF Core (devono essere mutabili per il change tracking) | Classe normale con `{ get; set; }` |
| Risultati di metodi con più valori temporanei | Tuple o `record` anonimo |

---

## Confronto con Java

| | Java | C# |
|---|---|---|
| Record immutabile | `record Punto(double x, double y)` | `record Punto(double X, double Y)` |
| Copia con modifiche | Non esiste nativamente | `originale with { X = 10 }` |
| Value type personalizzato | Non esiste | `struct` / `record struct` |
| Classe immutabile | Campi `final` + costruttore | Properties `init` + `required` |
| Equals strutturale | Automatico nei Records | Automatico nei `record` |

---

## In sintesi

- **`struct`**: value type, copiato per valore, per oggetti piccoli e in grande quantità.
- **`record`**: classe con `Equals` strutturale, `ToString` e `with` automatici — il tuo nuovo migliore amico per DTO e Value Object.
- **`record struct`**: unisce i vantaggi di entrambi.
- **Classe con `init`**: quando vuoi immutabilità ma non la semantica completa del record (es. entità con ereditarietà).
- L'espressione **`with`** non ha equivalente in Java ed è una delle feature più comode per lavorare con oggetti immutabili.
