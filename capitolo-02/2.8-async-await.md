# 2.8 Async/await, Task, continuations e CancellationToken

> **Analogia Java**: equivalente di `CompletableFuture`, `thenApply`, `thenCompose` — ma con una sintassi che ti farà chiedere perché Java non l'ha copiata subito

---

## Il panorama Java è frammentato — in C# no
>`"Era già un concetto complesso, hanno pensato bene di renderlo complicato"`


Se vieni da un progetto Spring moderno, probabilmente hai incontrato più di un approccio all'asincronia e ti sei chiesto quale usare:

- **CompletableFuture** — API standard Java, verbosa ma nativa
- **RxJava** — programmazione reattiva con `Observable`, `Flowable`, `Single`… potente ma con una curva di apprendimento ripida
- **Project Reactor** — la base di Spring WebFlux, con `Mono<T>` e `Flux<T>`, più moderno di RxJava ma altrettanto complesso
- **Kotlin Coroutines** — non solo "se il tuo team ha migrato a Kotlin": Spring supporta `suspend fun` nativamente da Spring 5.2+, quindi puoi usarle anche in progetti misti Java/Kotlin senza riscrivere tutto
- **Virtual threads (Java 21)** — l'ultimo arrivato, ma attenzione: non sono "asincroni" nel senso tradizionale. Sono thread leggeri che rendono scalabile il modello **sincrono/bloccante** — riduci il costo dell'attesa senza cambiare il modello di programmazione. Non sostituiscono Reactor o RxJava per backpressure o stream di dati; li affiancano per I/O-bound workloads dove vuoi scalabilità senza riscrivere il codice in stile reattivo

La specifica **Reactive Streams** nasce nel 2013–2015 come iniziativa indipendente su reactive-streams.org, promossa da Pivotal, Netflix, Twitter, **Lightbend** (Akka) e altri. RxJava e Reactor si sono adattati alla specifica mentre era ancora in sviluppo. Solo successivamente, con Java 9 (2017), è stata inglobata nello standard come **`java.util.concurrent.Flow`** — un'interfaccia comune per garantire interoperabilità. L'idea era buona: almeno un vocabolario condiviso. In pratica però non ha eliminato la frammentazione — hai ancora bisogno di scegliere *quale* libreria usare sopra quella specifica, e i modelli mentali di Reactor e RxJava restano molto diversi tra loro.

Risultato: nell'ecosistema Java esistono almeno quattro modi "ufficiali" o largamente adottati per gestire l'asincronia, spesso in conflitto tra loro. Passare da un progetto Spring MVC a uno Spring WebFlux significa imparare un modello mentale completamente diverso.

In C# la situazione è molto più semplice: **`async`/`await` è l'unico modello**. Non esiste un equivalente di Reactor o RxJava come approccio alternativo dominante. Esiste una libreria `System.Reactive` (RxNET) per i casi in cui serve la programmazione reattiva pura, ma non è mai diventata mainstream — nella stragrande maggioranza dei progetti .NET non la vedrai mai.

---

## Il problema della programmazione asincrona

Quando fai una chiamata HTTP, leggi un file, o interroghi un database, il tuo thread si ferma ad aspettare. In un'applicazione web con centinaia di richieste concorrenti, questo è un problema serio: ogni thread in attesa occupa memoria e risorse.

La soluzione è la programmazione asincrona: il thread viene liberato mentre aspetta, e riprende il lavoro quando la risposta è pronta. In C# si fa con `async/await` — vediamo come.

---

## `Task<T>` — l'equivalente di `CompletableFuture<T>`

Il tipo base della programmazione asincrona in C# è `Task<T>` — rappresenta un'operazione che produrrà un valore di tipo `T` in futuro. Se non c'è valore di ritorno, si usa `Task` senza generics.

| Java | C# |
|---|---|
| `CompletableFuture<T>` | `Task<T>` |
| `CompletableFuture<Void>` | `Task` |
| `Future<T>` | `Task<T>` (ma non usare `.Result` — vedi dopo) |

---

## `async` e `await` — la magia

In Java, comporre operazioni asincrone richiede callback o catene di `thenApply`/`thenCompose`:

```java
// Java — CompletableFuture
CompletableFuture<Ordine> futuro = repository.findByIdAsync(id)
    .thenApply(entity -> mapper.toOrdine(entity))
    .thenCompose(ordine -> validatore.validaAsync(ordine));
```

In C# le keyword `async` e `await` ti permettono di scrivere codice asincrono che *sembra* sincrono:

```csharp
// C# — async/await
public async Task<Ordine> GetOrdineAsync(int id)
{
    var entity = await repository.FindByIdAsync(id);   // aspetta senza bloccare il thread
    var ordine = mapper.ToOrdine(entity);
    await validatore.ValidaAsync(ordine);
    return ordine;
}
```

Stessa logica, leggibilità completamente diversa. Il compilatore trasforma questo codice in una macchina a stati che gestisce i callback internamente — tu scrivi codice lineare, lui si occupa del resto.

## Le regole fondamentali

### 1. `async` va sulla firma del metodo, `await` sul punto di attesa

```csharp
public async Task<string> GetDatiAsync()
{
    var risposta = await httpClient.GetStringAsync("https://api.example.com/data");
    return risposta;
}
```

### 2. Un metodo `async` deve restituire `Task`, `Task<T>` o `void` (evita `async void`)

```csharp
// ✅ corretto
public async Task FaiQualcosaAsync() { ... }
public async Task<int> CalcolaAsync() { ... }

// ⚠️ evita — async void inghiotte le eccezioni, usalo solo per event handler
public async void OnClick(object sender, EventArgs e) { ... }
```

### 3. `await` può essere usato solo dentro un metodo `async`

```csharp
// ❌ non puoi fare questo fuori da un metodo async
var result = await GetDatiAsync();
```

### 4. La convenzione vuole il suffisso `Async` sul nome del metodo

```csharp
public async Task<Ordine> GetOrdineAsync(int id) { ... }  // ✅ convenzione rispettata
public async Task<Ordine> GetOrdine(int id) { ... }       // ⚠️ funziona, ma non convenzionale
```

## Eseguire operazioni in parallelo — `Task.WhenAll`

Equivalente di `CompletableFuture.allOf()` in Java:

```csharp
// ❌ sequenziale — aspetta il primo, poi il secondo (2 secondi totali)
var ordine = await GetOrdineAsync(id);
var cliente = await GetClienteAsync(clienteId);

// ✅ parallelo — entrambe le chiamate partono insieme (1 secondo totale)
var ordineTask = GetOrdineAsync(id);
var clienteTask = GetClienteAsync(clienteId);

await Task.WhenAll(ordineTask, clienteTask);

var ordine = ordineTask.Result;    // già completato, Result non blocca
var cliente = clienteTask.Result;
```

`Task.WhenAny` invece completa quando *almeno una* delle task è finita — equivalente di `CompletableFuture.anyOf()`.

## `CancellationToken` — cancellare operazioni in corso

Questa è una feature senza equivalente diretto in Java base (esiste qualcosa con `ExecutorService` ma è molto più macchinoso). Il `CancellationToken` permette di segnalare a un'operazione asincrona che deve interrompersi.

In ASP.NET Core ogni richiesta HTTP ha un `CancellationToken` che viene attivato automaticamente se il client chiude la connessione. Passarlo alle operazioni asincrone evita di continuare a lavorare su richieste già abbandonate:

```csharp
[HttpGet("{id}")]
public async Task<IActionResult> GetOrdine(int id, CancellationToken cancellationToken)
{
    // Se il client chiude la connessione, cancellationToken viene attivato
    var ordine = await repository.GetByIdAsync(id, cancellationToken);
    return Ok(ordine);
}
```

Lato repository/servizio, passi il token ai metodi che lo supportano:

```csharp
public async Task<Ordine?> GetByIdAsync(int id, CancellationToken ct = default)
{
    return await dbContext.Ordini
        .FirstOrDefaultAsync(o => o.Id == id, ct);  // EF Core lo supporta nativamente
}
```

Se vuoi cancellare manualmente dopo un timeout:

```csharp
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));

try
{
    var risultato = await operazioneLentaAsync(cts.Token);
}
catch (OperationCanceledException)
{
    Console.WriteLine("Operazione annullata per timeout");
}
```

## La trappola di `.Result` e `.Wait()`

Questa è quella che fa più vittime, soprattutto chi arriva da Java e tenta di usare `Task` come un `Future`:

```csharp
// ❌ DEADLOCK in alcune situazioni — blocca il thread sincrono che aspetta
var ordine = GetOrdineAsync(id).Result;
GetOrdineAsync(id).Wait();

// ✅ sempre usa await
var ordine = await GetOrdineAsync(id);
```

`.Result` e `.Wait()` bloccano il thread corrente fino al completamento del Task — esattamente come `Future.get()` in Java. In ASP.NET Core questo può causare deadlock perché il contesto di sincronizzazione rimane bloccato. Regola semplice: **non usarli mai**, a meno che tu non sappia esattamente cosa stai facendo.

## `ValueTask<T>` — per i casi ad alte performance

`ValueTask<T>` è una versione ottimizzata di `Task<T>` per operazioni che spesso completano in modo sincrono (es. cache hit). Evita l'allocazione sull'heap quando non è necessaria:

```csharp
public async ValueTask<Ordine?> GetDaCacheAsync(int id)
{
    if (_cache.TryGetValue(id, out var ordine))
        return ordine;  // completa sincrono — nessuna allocazione Task

    return await _repository.GetByIdAsync(id);
}
```

Nella pratica quotidiana usi `Task<T>` per quasi tutto. `ValueTask<T>` è un'ottimizzazione consapevole per hot path.

## Confronto Java vs C# — riepilogo

| Java | C# |
|---|---|
| `CompletableFuture<T>` | `Task<T>` |
| `.thenApply(fn)` | `await` + codice normale |
| `.thenCompose(fn)` | `await` + codice normale |
| `CompletableFuture.allOf(...)` | `await Task.WhenAll(...)` |
| `CompletableFuture.anyOf(...)` | `await Task.WhenAny(...)` |
| `future.get()` | `.Result` (ma evitalo!) |
| Nessun equivalente nativo | `CancellationToken` |

---

## In sintesi

- `Task<T>` è il `CompletableFuture<T>` di C# — rappresenta un'operazione futura.
- `async`/`await` trasforma codice asincrono in codice che *sembra* sincrono — molto più leggibile delle catene Java.
- Per operazioni parallele usa `Task.WhenAll` — non fare `await` uno per uno se puoi farli partire insieme.
- Passa sempre il `CancellationToken` nelle Web API — ASP.NET Core lo gestisce automaticamente per te.
- Non usare mai `.Result` o `.Wait()` — porta a deadlock. Usa sempre `await`.
