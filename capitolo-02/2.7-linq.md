# 2.7 LINQ: interrogare collezioni in stile funzionale

> **Analogia Java**: equivalente della Stream API — stessa filosofia funzionale, sintassi diversa e qualche vantaggio extra

---

## Il concetto che già conosci

Se hai usato la Stream API di Java, hai già capito la filosofia: invece di scrivere loop imperativo con variabili di accumulazione, dichiari *cosa* vuoi ottenere e lasci che il framework si occupi del *come*. Filtri, trasformazioni, aggregazioni — tutto in catena.

```java
// Java — Stream API
List<String> nomi = ordini.stream()
    .filter(o -> o.getTotale() > 100)
    .map(Ordine::getCliente)
    .sorted()
    .collect(Collectors.toList());
```

LINQ fa la stessa cosa, ma è integrato nel linguaggio fin dal 2007 — molto prima che Java avesse le Stream. E ha un asso nella manica: la **query syntax**, che assomiglia all'SQL.

---

## Due sintassi per la stessa cosa

LINQ ha due modi di scrivere le stesse operazioni — puoi usarle entrambe, mischiarle, scegliere quella che preferisci.

### Method syntax — quella che userai di più

```csharp
var nomi = ordini
    .Where(o => o.Totale > 100)
    .Select(o => o.Cliente)
    .OrderBy(o => o)
    .ToList();
```

È la più simile alla Stream API Java — catena di metodi con lambda. Nella pratica quotidiana è quella dominante.

### Query syntax — quella che sembra SQL

```csharp
var nomi = (from o in ordini
            where o.Totale > 100
            orderby o.Cliente
            select o.Cliente)
           .ToList();
```

Stessa operazione, sintassi dichiarativa simile a SQL. Non è più o meno efficiente — il compilatore la converte nella method syntax prima di compilare. Utile per query complesse con join e groupby, dove la leggibilità migliora.

---

## Le operazioni fondamentali

### Filtrare — `Where`

```csharp
// C#
var grandi = ordini.Where(o => o.Totale > 100);

// Java
var grandi = ordini.stream().filter(o -> o.getTotale() > 100);
```

### Trasformare — `Select`

```csharp
// C#
var clienti = ordini.Select(o => o.Cliente);
var dto = ordini.Select(o => new OrdineDto(o.Id, o.Cliente, o.Totale));

// Java
var clienti = ordini.stream().map(Ordine::getCliente);
```

### Appiattire — `SelectMany`

```csharp
// C# — appiattisce una lista di liste
var tutteLeVoci = ordini.SelectMany(o => o.Voci);

// Java
var tutteLeVoci = ordini.stream().flatMap(o -> o.getVoci().stream());
```

### Ordinare — `OrderBy`, `OrderByDescending`, `ThenBy`

```csharp
var ordinati = ordini
    .OrderBy(o => o.Cliente)
    .ThenByDescending(o => o.Totale);

// Java
var ordinati = ordini.stream()
    .sorted(Comparator.comparing(Ordine::getCliente)
        .thenComparing(Comparator.comparing(Ordine::getTotale).reversed()));
```

### Raggruppare — `GroupBy`

```csharp
var perCliente = ordini
    .GroupBy(o => o.Cliente)
    .Select(g => new { Cliente = g.Key, Totale = g.Sum(o => o.Totale) });

// Java
var perCliente = ordini.stream()
    .collect(Collectors.groupingBy(Ordine::getCliente,
             Collectors.summingDouble(Ordine::getTotale)));
```

### Aggregare — `Sum`, `Count`, `Min`, `Max`, `Average`

```csharp
decimal totaleGlobale = ordini.Sum(o => o.Totale);
int numOrdini = ordini.Count(o => o.Totale > 100);
decimal massimo = ordini.Max(o => o.Totale);
double media = ordini.Average(o => (double)o.Totale);
```

### Verificare — `Any`, `All`, `Contains`

```csharp
bool haGrandiOrdini = ordini.Any(o => o.Totale > 1000);   // Java: anyMatch
bool tuttiPositivi = ordini.All(o => o.Totale > 0);        // Java: allMatch
bool contieneOrdine = ordini.Contains(unOrdine);            // Java: contains
```

### Prendere e saltare — `Take`, `Skip`

```csharp
var primaCinque = ordini.Take(5);         // Java: limit(5)
var dalSesto = ordini.Skip(5);            // Java: skip(5)
var pagina2 = ordini.Skip(10).Take(10);   // paginazione
```

### Trovare elementi — `First`, `FirstOrDefault`, `Single`, `SingleOrDefault`

```csharp
var primo = ordini.First(o => o.Totale > 100);           // lancia eccezione se non trovato
var primoONull = ordini.FirstOrDefault(o => o.Totale > 100); // null se non trovato
var unico = ordini.Single(o => o.Id == id);              // lancia eccezione se != 1 risultato
var unicoONull = ordini.SingleOrDefault(o => o.Id == id);
```

In Java hai `findFirst()` che restituisce un `Optional`. In C# `FirstOrDefault` restituisce `null` (o il valore di default del tipo) — nessun wrapper, ma devi ricordarti di controllare il null.

---

## Esecuzione differita — come in Java

Come la Stream API, LINQ usa l'**esecuzione differita**: la query non viene eseguita finché non itero il risultato o chiamo un metodo terminale.

```csharp
// Questa riga NON esegue nulla — costruisce solo la query
var query = ordini.Where(o => o.Totale > 100).Select(o => o.Cliente);

// L'esecuzione avviene qui — quando materializzo il risultato
var lista = query.ToList();      // esegue e restituisce List<string>
var array = query.ToArray();     // esegue e restituisce string[]
var primo = query.FirstOrDefault(); // esegue e restituisce il primo elemento

// Anche il foreach materializza la query
foreach (var nome in query) { ... }
```

La differenza con Java: in Java il terminal operation (`.collect()`, `.findFirst()`) è obbligatorio e chiude lo stream. In C# puoi tenere la query come `IEnumerable<T>` e materializzarla più volte — ma ogni volta esegue di nuovo la query. Con EF Core questo ha implicazioni importanti (lo vedremo nel capitolo 5).

---

## `IEnumerable<T>` vs `IQueryable<T>`

Questa distinzione è fondamentale quando lavori con EF Core, ma vale la pena introdurla subito.

- **`IEnumerable<T>`**: LINQ in memoria — i dati sono già caricati, le operazioni avvengono in C#.
- **`IQueryable<T>`**: LINQ tradotto in SQL — le operazioni vengono convertite in query SQL ed eseguite sul database.

```csharp
// IQueryable — il Where viene tradotto in SQL: WHERE Totale > 100
var query = dbContext.Ordini.Where(o => o.Totale > 100);

// IEnumerable — carica TUTTO dal database, poi filtra in memoria ⚠️
var query = dbContext.Ordini.AsEnumerable().Where(o => o.Totale > 100);
```

La seconda variante è una delle trappole più comuni per chi inizia con EF Core — carica milioni di righe in memoria e poi le filtra in C#. Da evitare.

## Operazioni su più collezioni — `Join` e `Zip`

```csharp
// Join tra due liste
var risultato = ordini.Join(
    clienti,
    o => o.ClienteId,
    c => c.Id,
    (o, c) => new { Ordine = o, Cliente = c }
);

// Zip — combina due liste elemento per elemento
var coppie = nomi.Zip(prezzi, (nome, prezzo) => $"{nome}: {prezzo}€");
```

---

## Operazioni su insiemi — `Distinct`, `Union`, `Intersect`, `Except`

```csharp
var unici = ordini.Select(o => o.Cliente).Distinct();
var tutti = lista1.Union(lista2);          // unione senza duplicati
var comuni = lista1.Intersect(lista2);     // intersezione
var solo1 = lista1.Except(lista2);         // differenza
```

---

## In sintesi

- LINQ è la Stream API di C# — stessa filosofia, integrata nel linguaggio dal 2007.
- **Method syntax** (`.Where().Select()`) è quella dominante nella pratica; **query syntax** (`from ... where ... select`) è utile per query complesse.
- L'esecuzione è **differita** — come in Java, la query si esegue solo quando materializzi il risultato con `ToList()`, `ToArray()`, `First()` o un `foreach`.
- `FirstOrDefault` restituisce `null` invece di `Optional` — controlla sempre il null.
- La distinzione **`IEnumerable<T>` vs `IQueryable<T>`** è critica con EF Core: con `IQueryable` il filtro va in SQL, con `IEnumerable` carica tutto in memoria prima di filtrare.
