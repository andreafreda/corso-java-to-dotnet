# 2.4 Properties al posto di getter/setter

> **Analogia Java**: nessun equivalente diretto — in Java serve Lombok (`@Getter`, `@Setter`), in C# è integrato nel linguaggio

---

## Il problema che Lombok risolve in Java

In Java, incapsulare un campo significa scrivere (o generare) getter e setter esplicitamente:

```java
// Java — senza Lombok
public class Ordine {
    private String cliente;
    private decimal totale;

    public String getCliente() { return cliente; }
    public void setCliente(String cliente) { this.cliente = cliente; }
    public decimal getTotale() { return totale; }
    public void setTotale(decimal totale) { this.totale = totale; }
}
```

Con Lombok diventa:

```java
// Java — con Lombok
@Getter @Setter
public class Ordine {
    private String cliente;
    private decimal totale;
}
```

In C# non hai bisogno di Lombok perché le **properties** sono una feature nativa del linguaggio. Sono getter e setter con una sintassi molto più compatta, direttamente supportata dal compilatore.

---

## Properties: la sintassi base

```csharp
public class Ordine
{
    public string Cliente { get; set; }
    public decimal Totale { get; set; }
}
```

Tre righe invece di otto. Il compilatore genera automaticamente il campo privato di backing (`_cliente`, `_totale`) e i metodi di accesso. Si usano come campi pubblici, ma internamente sono metodi:

```csharp
var ordine = new Ordine();
ordine.Cliente = "Andrea";   // chiama il setter
var nome = ordine.Cliente;   // chiama il getter
```

---

## Valori di default

Puoi assegnare un valore di default direttamente sulla property:

```csharp
public class Ordine
{
    public string Cliente { get; set; } = string.Empty;
    public decimal Totale { get; set; } = 0m;
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public List<string> Voci { get; set; } = new();
}
```

`new()` senza il tipo è una feature C# 9 — il compilatore lo deduce dal tipo della property. Molto più pulito di `new List<string>()`.

---

## Controllo dell'accesso: `get` e `set` separati

Puoi rendere il getter pubblico e il setter privato (o protected), esattamente come faresti con Lombok `@Getter` senza `@Setter`:

```csharp
public class Ordine
{
    public string Cliente { get; private set; }  // solo questa classe può modificarlo
    public decimal Totale { get; protected set; } // solo questa classe e le sottoclassi
    public Guid Id { get; init; }                 // può essere impostato solo nel costruttore o nell'initializer
}
```

`init` è una keyword C# 9: permette di impostare la property solo durante l'inizializzazione dell'oggetto (`new Ordine { Id = Guid.NewGuid() }`), dopo di che diventa di sola lettura. Perfetto per oggetti semi-immutabili.

---

## Properties con logica: il backing field

Quando ti serve logica nel getter o nel setter, puoi scrivere il corpo esplicitamente con un campo privato di backing:

```csharp
public class Ordine
{
    private decimal _totale;

    public decimal Totale
    {
        get => _totale;
        set
        {
            if (value < 0)
                throw new ArgumentException("Il totale non può essere negativo");
            _totale = value;
        }
    }
}
```

La stessa cosa in Java richiederebbe un setter esplicito con la validazione — qui la struttura è identica, solo più integrata.

---

## Expression-bodied properties — la versione compatta

Per properties di sola lettura calcolate, c'è una sintassi ancora più compatta:

```csharp
public class Ordine
{
    public decimal Imponibile { get; set; }
    public decimal Iva { get; set; }

    // Property calcolata — solo getter, niente campo di backing
    public decimal Totale => Imponibile + Iva;

    // Equivalente più verboso
    public decimal Totale2 { get { return Imponibile + Iva; } }
}
```

`=>` è la sintassi expression-bodied — la troverai ovunque in C# moderno, non solo nelle properties ma anche nei metodi.

---

## Object initializer — costruire oggetti senza costruttori

Una cosa che in Java richiede il builder pattern o un costruttore con molti parametri, in C# si risolve spesso con l'**object initializer**:

```csharp
var ordine = new Ordine
{
    Cliente = "Andrea",
    Totale = 99.99m,
    CreatedAt = DateTime.UtcNow
};
```

Non è un costruttore — è zucchero sintattico che chiama i setter uno per uno dopo la costruzione. Funziona con qualsiasi property che abbia un setter (o `init`) accessibile.

In Java il pattern equivalente è il Builder:

```java
// Java
Ordine ordine = Ordine.builder()
    .cliente("Andrea")
    .totale(new BigDecimal("99.99"))
    .build();
```

In C# il builder non serve quasi mai per questo scopo — l'object initializer lo rende superfluo.

---

## Properties nullable — campi opzionali

Combinando le properties con i nullable reference types che abbiamo visto nel 2.3, ottieni un modo molto espressivo per distinguere i campi obbligatori da quelli opzionali:

```csharp
public class Ordine
{
    public string Cliente { get; set; } = string.Empty; // obbligatorio
    public string? Note { get; set; }                   // opzionale — può essere null
    public decimal Totale { get; set; }                 // obbligatorio
    public DateTime? DataConsegna { get; set; }         // opzionale — può essere null
}
```

La `?` sul tipo è tutto quello che serve: chi legge il codice capisce immediatamente cosa è obbligatorio e cosa no, senza bisogno di commenti o documentazione. EF Core e il serializzatore JSON di ASP.NET Core rispettano questa distinzione automaticamente — una colonna `string?` diventa nullable nel database, una property `DateTime?` viene serializzata come `null` nel JSON se non valorizzata.

```csharp
var ordine = new Ordine
{
    Cliente = "Andrea",
    Totale = 99.99m
    // Note e DataConsegna non valorizzate — null per default, nessun problema
};
```

---

## Required properties — C# 11

Da C# 11 puoi marcare una property come obbligatoria: il compilatore si lamenta se provi a creare l'oggetto senza valorizzarla:

```csharp
public class Ordine
{
    public required string Cliente { get; set; }
    public required decimal Totale { get; set; }
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
}

// ✅ ok
var ordine = new Ordine { Cliente = "Andrea", Totale = 99.99m };

// ❌ errore di compilazione — Cliente e Totale sono required
var ordine2 = new Ordine();
```

È un'alternativa ai costruttori per garantire che certi campi siano sempre valorizzati — più flessibile perché si combina con l'object initializer.

---

## In sintesi

- Le **properties** (`{ get; set; }`) sostituiscono getter e setter Java — sono native nel linguaggio, niente Lombok.
- `private set` e `init` ti danno controllo fine sull'immutabilità senza costruttori complicati.
- Le **expression-bodied properties** (`=>`) sono perfette per valori calcolati.
- L'**object initializer** (`new Ordine { Campo = valore }`) sostituisce il builder pattern Java per la maggior parte dei casi.
- `required` (C# 11) garantisce che certi campi siano sempre valorizzati — a compile time, non a runtime.
