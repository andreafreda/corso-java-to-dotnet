# 2.10 Extension methods e interfacce

> **Analogia Java**: nessun equivalente diretto per gli extension methods — in Java serve il pattern decorator o una classe di utility statica. Le interfacce invece si assomigliano, con qualche differenza importante.

---

## Extension methods — aggiungere metodi senza toccare la classe

Immagina di voler aggiungere un metodo `IsNullOrEmpty()` direttamente su `string`, senza modificare la classe `String` (che è sealed e appartiene al framework). In Java non puoi — finisci a scrivere una classe di utility statica:

```java
// Java — l'unica via
public class StringUtils {
    public static boolean isNullOrEmpty(String s) {
        return s == null || s.isEmpty();
    }
}
// Uso: StringUtils.isNullOrEmpty(nome)
```

In C# puoi aggiungere quel metodo direttamente alla classe `string` tramite un **extension method**, e chiamarlo come se fosse un metodo nativo:

```csharp
// Definizione — classe statica, metodo statico, primo parametro con `this`
public static class StringExtensions
{
    public static bool IsNullOrEmpty(this string? value)
        => string.IsNullOrEmpty(value);

    public static string Truncate(this string value, int maxLength)
        => value.Length <= maxLength ? value : value[..maxLength] + "…";
}

// Uso — si chiama come se fosse un metodo di string
string nome = "Andrea Fred";
bool vuoto = nome.IsNullOrEmpty();          // false
string corto = nome.Truncate(6);            // "Andrea…"
```

Il trucco è il `this` davanti al primo parametro — dice al compilatore "questo metodo estende il tipo che segue". La classe deve essere statica, il metodo deve essere statico.

---

## Perché è utile nella pratica

Gli extension methods sono ovunque in .NET — LINQ stesso è implementato come extension methods su `IEnumerable<T>`. Quando scrivi:

```csharp
ordini.Where(o => o.Totale > 100).OrderBy(o => o.Cliente).ToList();
```

Stai chiamando extension methods definiti in `System.Linq.Enumerable`. Non sono metodi di `List<T>` — sono metodi aggiunti dall'esterno, esattamente come faresti tu.

Nella pratica li userai per:
- Aggiungere metodi helper su tipi esistenti che non puoi modificare
- Rendere il codice più leggibile con una sintassi fluente
- Estendere interfacce con comportamento comune

## Estendere un'interfaccia

Puoi scrivere extension methods su interfacce — e questo è particolarmente potente:

```csharp
public interface IRepository<T>
{
    Task<T?> GetByIdAsync(int id, CancellationToken ct = default);
    Task<IEnumerable<T>> GetAllAsync(CancellationToken ct = default);
}

// Extension method sull'interfaccia — disponibile su qualsiasi implementazione
public static class RepositoryExtensions
{
    public static async Task<T> GetByIdOrThrowAsync<T>(
        this IRepository<T> repo, int id, CancellationToken ct = default)
    {
        var entity = await repo.GetByIdAsync(id, ct);
        return entity ?? throw new KeyNotFoundException($"Entità {typeof(T).Name} con id {id} non trovata");
    }
}

// Uso
var ordine = await ordineRepository.GetByIdOrThrowAsync(42);
```

## Interfacce — le differenze rispetto a Java

Le interfacce C# ti sembreranno familiari, ma ci sono alcune differenze che vale la pena conoscere subito.

### Sintassi base — quasi identica

```csharp
public interface INotificatore
{
    Task InviaAsync(string destinatario, string messaggio, CancellationToken ct = default);
    string Nome { get; }  // property — in Java saresti costretto a un getter
}
```

In Java le property non esistono nelle interfacce — dichiareresti `String getNome()`. In C# puoi dichiarare direttamente una property.

### Default interface methods — come Java 8+

Da C# 8, le interfacce possono avere implementazioni di default, esattamente come Java 8+:

```csharp
public interface INotificatore
{
    Task InviaAsync(string destinatario, string messaggio, CancellationToken ct = default);

    // Metodo con implementazione di default
    Task InviaConRitentativiAsync(string destinatario, string messaggio, int tentativi = 3)
    {
        // implementazione di default con retry
    }
}
```

Stesso scopo di Java: evolvere le interfacce senza rompere le implementazioni esistenti.

### Nessun `implements` — solo `:` per tutto

In Java usi `extends` per le classi e `implements` per le interfacce. In C# c'è solo `:` per entrambi, e il compilatore distingue automaticamente:

```csharp
// C# — tutto con ':'
public class EmailNotificatore : BaseNotificatore, INotificatore, IDisposable
{
    // ...
}
```

```java
// Java — per confronto
public class EmailNotificatore extends BaseNotificatore implements INotificatore, Closeable {
    // ...
}
```

### Interfacce come vincoli sui generics

Un uso molto comune in C# che in Java si fa diversamente:

```csharp
// Accetta qualsiasi tipo che implementa INotificatore
public async Task NotificaTutti<T>(IEnumerable<T> notificatori, string msg)
    where T : INotificatore
{
    foreach (var n in notificatori)
        await n.InviaAsync("tutti", msg);
}
```

Il `where T : INotificatore` è l'equivalente del bounded type parameter `<T extends INotificatore>` in Java.

## Il pattern comune: interface + extension methods

La combinazione di interfacce snelle + extension methods è un pattern molto diffuso in C# — mantieni l'interfaccia piccola e aggiungi comportamento derivato tramite extension methods:

```csharp
// Interfaccia minimale
public interface ILogger
{
    void Log(LogLevel level, string message);
}

// Comportamento derivato come extension methods — non appesantisce l'interfaccia
public static class LoggerExtensions
{
    public static void LogInfo(this ILogger logger, string msg)
        => logger.Log(LogLevel.Information, msg);

    public static void LogError(this ILogger logger, string msg)
        => logger.Log(LogLevel.Error, msg);

    public static void LogError(this ILogger logger, Exception ex, string msg)
        => logger.Log(LogLevel.Error, $"{msg}: {ex.Message}");
}

// Uso
logger.LogInfo("Ordine creato");
logger.LogError(ex, "Errore durante il pagamento");
```

`Microsoft.Extensions.Logging` usa esattamente questo pattern — l'interfaccia `ILogger` ha pochi metodi, tutto il resto sono extension methods.

---

## In sintesi

- Gli **extension methods** aggiungono metodi a tipi esistenti senza modificarli — la classe utility statica di Java, ma con una sintassi da metodo di istanza.
- LINQ è interamente costruito su extension methods su `IEnumerable<T>` — quando usi `.Where()` stai già usando questa feature.
- Le **interfacce** C# sono molto simili a quelle Java: stessa sintassi `:` per tutto (no `implements`), supporto per default methods da C# 8, property dichiarabili direttamente.
- Il pattern **interfaccia snella + extension methods** è idiomatico in .NET — mantieni pochi metodi astratti, aggiungi il resto come estensioni.
