## Date e ora: il mapping con Java

Se il tuo riferimento Ă¨ `Calendar` ti sono vicino, e ti abbraccio. 
Se hai lavorato con `java.time` (introdotto in Java 8), il modello mentale Ă¨ lo stesso: esistono tipi separati per data, ora, data+ora, e con/senza fuso orario. I nomi sono diversi, ma la filosofia Ă¨ identica.

| Java | C# | Cosa rappresenta |
|---|---|---|
| `LocalDate` | `DateOnly` | Solo la data (anno, mese, giorno) |
| `LocalTime` | `TimeOnly` | Solo l'ora (ore, minuti, secondi) |
| `LocalDateTime` | `DateTime` | Data + ora, senza fuso orario |
| `ZonedDateTime` / `OffsetDateTime` | `DateTimeOffset` | Data + ora + offset dal UTC |
| `Instant` | `DateTime` con `Kind = Utc` | Istante preciso nel tempo in UTC |
| `Duration` / `Period` | `TimeSpan` | Intervallo di tempo |

> **Nota storica**: `DateOnly` e `TimeOnly` sono stati introdotti in .NET 6. Prima di allora si usava `DateTime` per tutto â€” nei progetti legacy li incontrerai ancora spesso.

## `DateTime` â€” il piĂ¹ comune

```csharp
// Ora corrente (ora locale della macchina)
DateTime adesso = DateTime.Now;

// Ora corrente in UTC â€” preferibile nelle API e nei database
DateTime adessoUtc = DateTime.UtcNow;

// Data specifica
DateTime compleanno = new DateTime(1990, 6, 15);

// Data + ora specifica
DateTime evento = new DateTime(2025, 12, 31, 23, 59, 0);

// Operazioni
DateTime domani = adesso.AddDays(1);
DateTime unOraFa = adesso.AddHours(-1);
TimeSpan differenza = domani - adesso; // restituisce un TimeSpan
```

La trappola classica: `DateTime.Now` usa il fuso orario della macchina, `DateTime.UtcNow` usa UTC. Nelle Web API e nei database usa **sempre** UTC â€” ti risparmia un'infinitĂ  di bug legati ai fusi orari.

## `DateTimeOffset` â€” quando il fuso orario conta

`DateTimeOffset` Ă¨ l'equivalente di `ZonedDateTime` in Java: porta con sĂ© l'offset rispetto a UTC. Ăˆ il tipo consigliato quando devi conservare il fuso orario originale dell'utente.

```csharp

                                           TimeSpan.FromHours(1)); // UTC+1

Console.WriteLine(evento.UtcDateTime); // convertito in UTC
```

## `DateOnly` e `TimeOnly` â€” quando non ti serve l'ora

Introdotti in .NET 6, risolvono il problema classico di dover usare `DateTime` anche quando hai solo una data (con l'ora impostata a mezzanotte come convenzione):

```csharp
DateOnly oggi = DateOnly.FromDateTime(DateTime.Now);
DateOnly scadenza = new DateOnly(2025, 12, 31);

TimeOnly apertura = new TimeOnly(9, 0);   // 09:00
TimeOnly chiusura = new TimeOnly(18, 30); // 18:30

bool aperto = TimeOnly.FromDateTime(DateTime.Now) >= apertura;
```

Se stai modellando una data di nascita, una scadenza, un orario di apertura â€” usa questi tipi. Sono semanticamente piĂ¹ precisi di `DateTime`.

## `TimeSpan` â€” intervalli di tempo

Equivalente di `Duration` in Java. Rappresenta una durata, non un momento:

```csharp

TimeSpan trenta = TimeSpan.FromMinutes(30);
TimeSpan combined = unOra + trenta; // 1h 30min

Console.WriteLine(unOra.TotalMinutes); // 60
Console.WriteLine(trenta.Seconds);     // 0 (solo i secondi "residui", non il totale)
Console.WriteLine(trenta.TotalSeconds);// 1800 (il totale in secondi)
```

Attenzione alla differenza tra `Seconds` e `TotalSeconds` â€” Ă¨ la stessa trappola di `Duration.getSeconds()` vs `toSeconds()` in Java.

## Parsing e formattazione

```csharp
// Da stringa a DateTime
DateTime d = DateTime.Parse("2025-12-31");
bool ok = DateTime.TryParse("non una data", out DateTime risultato); // false, non lancia eccezione

// Formato personalizzato
DateTime d2 = DateTime.ParseExact("31/12/2025", "dd/MM/yyyy",
                                   System.Globalization.CultureInfo.InvariantCulture);

// Da DateTime a stringa
string iso = DateTime.UtcNow.ToString("yyyy-MM-ddTHH:mm:ssZ"); // ISO 8601
string leggibile = DateTime.Now.ToString("dd MMMM yyyy");       // "09 aprile 2026"
```

Il consiglio pratico: nelle API REST usa sempre il formato **ISO 8601** (`yyyy-MM-ddTHH:mm:ssZ`) â€” Ă¨ lo standard e ASP.NET Core lo gestisce automaticamente nella serializzazione JSON.

## `Guid` â€” l'equivalente di `UUID`

In .NET i `Guid` (Globally Unique Identifier) sono ovunque â€” come chiavi primarie, come identificatori di correlazione nei log, come token. Sono l'equivalente di `java.util.UUID`.

```csharp
// Genera un nuovo GUID
Guid id = Guid.NewGuid();
Console.WriteLine(id); // es: 3f2504e0-4f89-11d3-9a0c-0305e82c3301

// GUID vuoto (tutti zeri) â€” utile come valore di default o sentinella
Guid vuoto = Guid.Empty;
Console.WriteLine(vuoto == Guid.Empty); // true

// Da stringa
Guid parsed = Guid.Parse("3f2504e0-4f89-11d3-9a0c-0305e82c3301");
bool ok = Guid.TryParse("non valido", out Guid risultato); // false

// Confronto
Guid a = Guid.NewGuid();
Guid b = Guid.NewGuid();
Console.WriteLine(a == b); // quasi certamente false
```

Nei modelli EF Core e nelle Web API vedrai `Guid` praticamente ovunque come tipo della chiave primaria â€” Ă¨ la convenzione dominante in .NET, piĂ¹ di quanto `UUID` non lo sia in Java/Spring.

## `Uri` â€” per gli URL

Quando lavori con URL nelle Web API o negli `HttpClient`, trovi spesso il tipo `Uri` invece della semplice stringa:

```csharp

Uri endpoint = new Uri(baseUrl, "/orders/123");

Console.WriteLine(endpoint.Host);   // api.example.com
Console.WriteLine(endpoint.PathAndQuery); // /orders/123
```

Equivalente di `java.net.URI`. Nella pratica quotidiana lo usi soprattutto quando configuri `HttpClient` â€” non devi memorizzarne tutta l'API.

## In sintesi

- Per le date usa `DateOnly`, per le ore `TimeOnly`, per data+ora `DateTime` (preferendo UTC) o `DateTimeOffset` quando il fuso orario Ă¨ rilevante.
- `TimeSpan` Ă¨ il tuo `Duration` â€” attenzione a `Seconds` vs `TotalSeconds`.
- Nelle API REST serializza sempre in **ISO 8601** â€” ASP.NET Core lo fa automaticamente.
- `Guid` Ă¨ l'`UUID` di .NET â€” lo troverai ovunque come chiave primaria.
- `Uri` esiste ed Ă¨ utile, ma nella pratica lo usi principalmente con `HttpClient`.
