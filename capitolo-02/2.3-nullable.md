# 2.3 Nullable reference types e `null` safety

> **Analogia Java**: equivalente di `Optional<T>` ma a livello di tipo — più pervasivo, meno verboso

---

## Il problema che conosci bene

La `NullPointerException` è probabilmente l'eccezione che hai visto più spesso in vita tua. Java ci ha convissuto per decenni, e la risposta ufficiale è arrivata con `Optional<T>` in Java 8: un wrapper esplicito per dire "questo valore potrebbe non esserci".

```java
// Java — Optional
Optional<String> nome = Optional.ofNullable(utente.getNome());
String risultato = nome.orElse("anonimo");
```

Funziona, ma è verboso e richiede disciplina: nulla ti impedisce di ignorare `Optional` e usare riferimenti `null` normali ovunque. Il compilatore non ti aiuta.

C# ha preso una strada diversa — e dal punto di vista della sicurezza è più efficace.

---

## Nullable value types — `int?`, `bool?`, `DateTime?`

Prima di parlare dei reference types, c'è una feature che Java non ha: i **nullable value types**. In Java i tipi primitivi (`int`, `boolean`, `double`…) non possono mai essere `null` — se vuoi null devi usare i wrapper (`Integer`, `Boolean`…). In C# puoi rendere nullable qualsiasi tipo di valore con il suffisso `?`:

```csharp
int numero = 42;
int? forse = null;      // nullable int — può essere null
bool? flag = null;      // nullable bool — può essere null
DateTime? scadenza = null; // nullable DateTime

// Controllare se ha valore
if (forse.HasValue)
    Console.WriteLine(forse.Value);

// Oppure con l'operatore null-coalescing
int valore = forse ?? 0; // se forse è null, usa 0
```

Utile soprattutto con i database: una colonna nullable in SQL diventa un `int?` o `DateTime?` nel modello EF Core. Molto più pulito dei wrapper Java.

---

## Nullable reference types — la grande novità

Da C# 8 (abilitato di default nei nuovi progetti .NET) esiste la feature dei **nullable reference types**: il compilatore distingue tra riferimenti che possono essere `null` e riferimenti che non possono esserlo.

```csharp
string nome = "Andrea";   // NON nullable — non può essere null
string? cognome = null;   // nullable — può essere null
```

Se provi ad assegnare `null` a una variabile non-nullable, il compilatore ti avvisa:

```csharp
string nome = null; // ⚠️ warning: assigning null to non-nullable reference
```

E se usi una variabile nullable senza controllare che non sia null prima, il compilatore ti avvisa anche lì:

```csharp
string? cognome = null;
Console.WriteLine(cognome.Length); // ⚠️ warning: possible null dereference
```

Sono **warning**, non errori — il codice compila lo stesso. Ma in un progetto con `TreatWarningsAsErrors=true` (che ti consiglio) diventano errori a tutti gli effetti.

---

## Gli operatori per gestire il null

C# ha una famiglia di operatori dedicati che rendono la gestione del null molto più fluida di Java:

### `??` — null-coalescing (valore di default)

```csharp
string? nome = null;
string risultato = nome ?? "anonimo"; // "anonimo" se nome è null

// Equivalente Java
String risultato = Optional.ofNullable(nome).orElse("anonimo");
```

### `??=` — null-coalescing assignment

```csharp
string? cache = null;
cache ??= CaricaDalDatabase(); // assegna solo se cache è null
```

### `?.` — null-conditional (safe navigation)

```csharp
string? nome = utente?.Nome;           // null se utente è null
int? lunghezza = utente?.Nome?.Length; // null se utente o Nome sono null

// Equivalente Java (verboso)
String nome = utente != null ? utente.getNome() : null;
```

### `!` — null-forgiving (usalo con parsimonia)

Dice al compilatore "fidati di me, questo non è null" — disabilita il warning:

```csharp
string nome = GetNome()!; // il ! dice: so che non è null, non avvisarmi
```

Usalo solo quando sei davvero sicuro e il compilatore non riesce a dedurlo da solo. Abusarne vanifica tutta la protezione del sistema.

---

## Confronto con `Optional<T>` Java

| | Java `Optional<T>` | C# nullable reference types |
|---|---|---|
| Sintassi | `Optional<String>` | `string?` |
| Valore di default | `.orElse("default")` | `?? "default"` |
| Accesso sicuro | `.map(s -> s.length())` | `?.Length` |
| Obbligatorietà | Convenzione, non enforcement | Warning del compilatore |
| Runtime overhead | Sì (oggetto wrapper) | No (solo compile-time) |

Il vantaggio di C# è che non c'è overhead a runtime — è tutto risolto a compile time, come le annotazioni `@NotNull`/`@Nullable` ma integrato nel linguaggio.

---

## Come si configura

Nei nuovi progetti .NET la feature è abilitata di default tramite il `.csproj`:

```xml
<PropertyGroup>
  <Nullable>enable</Nullable>
</PropertyGroup>
```

Se lavori su un progetto legacy che non ce l'ha, puoi abilitarla file per file con una direttiva:

```csharp
#nullable enable

string? nome = null; // ora il compilatore controlla
```

## Il pattern pratico nelle Web API

Nei modelli e nei DTO vedrai spesso questa distinzione:

```csharp
public class CreaOrdineRequest
{
    public string NomeCliente { get; set; } = string.Empty; // obbligatorio
    public string? Note { get; set; }                        // opzionale
    public decimal Totale { get; set; }                     // obbligatorio
    public DateTime? DataConsegna { get; set; }             // opzionale
}
```

`string.Empty` come valore di default è un pattern comune per i campi obbligatori non-nullable — evita il warning del compilatore e rende esplicito che il campo deve essere valorizzato (dalla validazione con FluentValidation, che vedremo nel capitolo 4).

## In sintesi

- `int?`, `DateTime?` e simili sono **nullable value types** — più puliti dei wrapper `Integer`/`Optional` Java per i casi semplici.
- I **nullable reference types** (`string?` vs `string`) danno al compilatore la capacità di avvisarti prima che arrivi la `NullPointerException` — zero overhead a runtime.
- `??`, `?.`, `??=` sono i tuoi nuovi migliori amici — li userai ogni giorno.
- L'operatore `!` (null-forgiving) è una valvola di sfogo: usalo solo quando sai davvero quello che stai facendo.
- Abilita sempre `<Nullable>enable</Nullable>` nei nuovi progetti — e considera `TreatWarningsAsErrors` per avere protezione reale.

> 🎯 **Ai colloqui piace chiedere la differenza tra reference type e value type — scolpisci nella tua mente la risposta:**
>
> **Value type** (`int`, `bool`, `double`, `decimal`, `struct`, `enum`): la variabile **contiene direttamente il valore**. Vive sullo **stack**. Quando assegni una variabile a un'altra, viene **copiato il valore** — le due variabili sono completamente indipendenti.
> ```csharp
> int a = 10;
> int b = a;
> b = 99;
> Console.WriteLine(a); // 10 — a non è cambiato
> ```
>
> **Reference type** (`class`, `string`, array, `interface`): la variabile **contiene un riferimento** (un puntatore) all'oggetto, che vive nell'**heap**. Quando assegni una variabile a un'altra, copi il riferimento — **entrambe puntano allo stesso oggetto**.
> ```csharp
> var lista1 = new List<int> { 1, 2, 3 };
> var lista2 = lista1;       // copia il riferimento, non la lista
> lista2.Add(4);
> Console.WriteLine(lista1.Count); // 4 — lista1 è cambiata!
> ```
>
> Aggiungere `?` a un **value type** (`int?`) lo avvolge in `Nullable<T>` — struttura reale che può contenere null. Aggiungere `?` a un **reference type** (`string?`) è solo un'annotazione per il compilatore — a runtime non cambia nulla, i reference type possono già essere null per natura.
