# 2.1 Tipi, variabili e type inference (`var`)

> **Analogia Java**: come `var` in Java 10+, ma con qualche differenza che vale la pena conoscere

---

## Il concetto che già conosci

In Java hai tipi primitivi (`int`, `boolean`, `double`…), tipi reference (`String`, `List<T>`…) e dal Java 10 in poi hai `var` per l'inferenza del tipo. C# funziona in modo molto simile — se sai leggere codice Java, leggerai codice C# senza problemi già dal primo giorno.

Le differenze ci sono, ma sono dettagli. Vediamole una per una.

---

## Tipi primitivi: stessa idea, nomi diversi

| Java | C# | Note |
|---|---|---|
| `int` | `int` | Identico — 32 bit con segno |
| `long` | `long` | Identico — 64 bit con segno |
| `double` | `double` | Identico — 64 bit floating point |
| `float` | `float` | Identico — 32 bit floating point |
| `boolean` | `bool` | Stesso concetto, nome più corto |
| `char` | `char` | Identico — Unicode 16 bit |
| `byte` | `byte` | Identico — 8 bit senza segno |
| `short` | `short` | Identico — 16 bit con segno |
| — | `decimal` | 128 bit, precisione decimale — per calcoli finanziari |

Alcune differenze che vale la pena conoscere subito:

**`decimal` al posto di `BigDecimal`**: in Java per i calcoli finanziari usi `BigDecimal`, che è una classe con la sua verbosità (`new BigDecimal("19.99")`, `.add()`, `.multiply()`…). In C# `decimal` è un tipo primitivo a 128 bit — stessa precisione, ma si usa come un normale numero:

```csharp
decimal prezzo = 19.99m;          // la 'm' finale indica un literal decimal
decimal totale = prezzo * 3;      // operatori normali, niente .multiply()
decimal arrotondato = Math.Round(prezzo, 2);
```

**`BigInteger`**: esiste anche in C#, con lo stesso nome, nel namespace `System.Numerics`. Per numeri interi di dimensione arbitraria — raro nella pratica, ma c'è:

```csharp
using System.Numerics;

BigInteger enorme = BigInteger.Pow(2, 100);
Console.WriteLine(enorme); // 1267650600228229401496703205376
```

**`double` e precisione floating point**: stessi problemi di Java — `0.1 + 0.2 != 0.3`. Se stai gestendo soldi, usa sempre `decimal`, mai `double` o `float`.

```csharp
Console.WriteLine(0.1 + 0.2);           // 0.30000000000000004 — sorpresa!
Console.WriteLine(0.1m + 0.2m);         // 0.3 — corretto con decimal
```

## Dichiarazione di variabili

In C# la sintassi è identica a Java:

```csharp
int età = 30;
string nome = "Andrea";
bool attivo = true;
```

Nota: `string` in C# è minuscolo (è un alias per `System.String`), mentre in Java è `String` maiuscolo. Entrambi funzionano in C#, ma per convenzione si usa quello minuscolo.

---

## `var` — type inference

Esattamente come in Java 10+, puoi usare `var` e lasciare che il compilatore inferisca il tipo:

```csharp
var età = 30;           // int
var nome = "Andrea";    // string
var prezzi = new List<decimal>(); // List<decimal>
```

Una differenza rispetto a Java: in C# `var` esiste **da sempre** (dal 2007, C# 3.0), molto prima che arrivasse in Java. Nell'ecosistema .NET è normalissimo usarlo ovunque — non è una novità recente da usare con parsimonia come in Java, ma uno stile consolidato e incoraggiato.

Stessa regola di Java però: `var` funziona solo quando il tipo è deducibile dall'inizializzatore. Questo non compila:

```csharp
var x; // ❌ errore — il compilatore non sa che tipo è
x = 42;
```

## Costanti

In Java usi `final` per le costanti. In C# hai due opzioni:

```csharp
// const — valore fissato a compile time (equivalente di static final in Java)
const double Pi = 3.14159;

// readonly — valore fissato a runtime, può essere assegnato nel costruttore
readonly int maxRetry;

public MyService(int max)
{
    maxRetry = max; // ok, siamo nel costruttore
}
```

`const` è più restrittivo: funziona solo per valori noti a compile time (numeri, stringhe, `null`). `readonly` è l'equivalente più fedele di `final` in Java.

## Tipi complessi

### Array

Sintassi leggermente diversa da Java — le parentesi quadre vanno dopo il tipo, non dopo il nome:

```csharp
// C#
int[] numeri = new int[5];
int[] inizializzato = { 1, 2, 3, 4, 5 };
string[,] matrice = new string[3, 4]; // array bidimensionale

// Java — per confronto
int[] numeri = new int[5];       // uguale
int[][] matrice = new int[3][4]; // Java usa array di array, non matrici native
```

### Enum

Funzionano come in Java ma sono più semplici — non sono classi, non hanno metodi nativamente:

```csharp
enum Stato { Attivo, Inattivo, Sospeso }

Stato s = Stato.Attivo;
Console.WriteLine(s);        // "Attivo"
Console.WriteLine((int)s);   // 0 — ogni valore ha un intero sottostante
```

Se vuoi aggiungere comportamento a un enum in C# devi usare extension methods (li vedremo nel 2.8). In Java invece un enum è una vera classe con costruttori e metodi — qui è una differenza reale da tenere a mente.

### Tuple

Le tuple non hanno un equivalente diretto idiomatico in Java. Ti permettono di raggruppare valori temporanei senza creare una classe:

```csharp
// Tuple senza nome
(int, string) coppia = (42, "ciao");
Console.WriteLine(coppia.Item1); // 42

// Tuple con nome — molto più leggibili
(int Id, string Nome) utente = (1, "Andrea");
Console.WriteLine(utente.Nome);  // "Andrea"

// Utilissime come return type quando vuoi restituire più valori
(bool Successo, string Errore) Valida(string input)
{
    if (string.IsNullOrEmpty(input))
        return (false, "Input vuoto");
    return (true, "");
}
```

In Java ritorneresti un oggetto, un `Map.Entry` o (dal Java 16) un record. Le tuple C# sono più leggere per i casi temporanei — ma se il tipo diventa importante, passa a un record o a una classe.

### Collezioni principali

Le collezioni le approfondiremo nel 2.5 con LINQ, ma vale la pena avere subito la mappa mentale:

| Java | C# | Note |
|---|---|---|
| `ArrayList` / `List<T>` | `List<T>` | Lista dinamica |
| `HashMap<K,V>` | `Dictionary<K,V>` | Mappa chiave-valore |
| `HashSet<T>` | `HashSet<T>` | Insieme senza duplicati |
| `LinkedList<T>` | `LinkedList<T>` | Lista doppiamente collegata |
| `Stack<T>` | `Stack<T>` | LIFO |
| `Queue<T>` | `Queue<T>` | FIFO |

Tutte nel namespace `System.Collections.Generic` — non devi importare nulla di speciale se hai `ImplicitUsings` abilitato (come nel `.csproj` che abbiamo visto nel 1.3).

## `object` — la base di tutto

In Java ogni classe estende implicitamente `Object`. In C# vale la stessa cosa, e il tipo si chiama `object` (minuscolo — alias di `System.Object`):

```csharp
object qualsiasi = 42;
qualsiasi = "ora sono una stringa";
qualsiasi = new List<int>();
```

Puoi metterci dentro qualsiasi cosa, ma perdi il tipo a compile time — esattamente come in Java con `Object`. Il cast esplicito ti riporta al tipo originale:

```csharp
object valore = 42;
int numero = (int)valore; // cast esplicito
```

Nella pratica moderna non lo usi quasi mai direttamente — i generic (`List<T>`, `Dictionary<K,V>`) rendono superfluo passare per `object`. Lo incontri principalmente nelle API legacy o in contesti di serializzazione.

## `dynamic` — tipizzazione a runtime

Questo non ha equivalente in Java. Il tipo `dynamic` dice al compilatore: *"fidati di me, risolvi tutto a runtime"*. Nessun controllo statico dei tipi, nessun errore a compile time — gli errori arrivano solo quando il codice viene eseguito.

```csharp
dynamic valore = 42;
Console.WriteLine(valore + 1);   // 43 — funziona

valore = "ciao";
Console.WriteLine(valore.Length); // 4 — funziona

valore.MetodoCheNonEsiste();      // ✅ compila, ❌ esplode a runtime
```
*ATTENTO!* Può sembrare il sacro graal e ti verrebbe da usarlo ovunque! Non è un coltello `miracle blade`! Usalo a tuo rischio e pericolo! Non hai idea di quanti problemi abbiamo avuto in produzione causati da un uso poco safe di dynamic.

Quando usarlo? Quasi mai nel codice che scrivi tu. Lo incontrerai principalmente quando:
- Interagisci con librerie COM (automazione Office, ad esempio)
- Lavori con JSON o dati da sorgenti esterne non tipizzate
- Leggi codice legacy

Il messaggio pratico è semplice: se ti ritrovi a usare `dynamic` in un progetto nuovo, fermati un secondo e chiediti se non esiste un'alternativa tipizzata. Quasi sempre esiste.

## Conversioni di tipo

Le conversioni implicite ed esplicite funzionano come in Java:

```csharp
int x = 10;
long y = x;        // implicita — ok, long è più grande
double z = x;      // implicita — ok

long grande = 1000L;
int piccolo = (int)grande;  // cast esplicito — come in Java
```

Una comodità in più: i metodi `Convert` e il parsing sono un po' più ergonomici rispetto a Java:

```csharp
// Da stringa a numero
int n = int.Parse("42");
bool ok = int.TryParse("abc", out int risultato); // non lancia eccezione se fallisce

// Equivalente Java
int n = Integer.parseInt("42");
```

`TryParse` è un pattern molto usato in C#: invece di lanciare un'eccezione se la conversione fallisce, restituisce `false` e mette il risultato in un parametro `out`. Lo vedrai spesso.

## Interpolazione di stringhe

In Java usi `String.format()` o (dal Java 15) i text block. In C# l'interpolazione è integrata nella sintassi con il prefisso `$`:

```csharp
string nome = "Andrea";
int anni = 30;

// C# — string interpolation
string messaggio = $"Ciao {nome}, hai {anni} anni";

// Equivalente Java
String messaggio = "Ciao %s, hai %d anni".formatted(nome, anni);
```

Il prefisso `$` è disponibile da C# 6 (2015) — anche qui, non una novità recente ma uno stile consolidato che troverai ovunque.

---

## In sintesi

- I tipi primitivi sono quasi identici a Java — ricorda solo `bool` al posto di `boolean` e il tipo `decimal` per i calcoli finanziari.
- `var` funziona come in Java 10+, ma in C# è lo stile normale fin dal 2007 — usalo senza remore.
- Per le costanti: `const` per valori compile-time, `readonly` per l'equivalente di `final` Java.
- `TryParse` è il pattern idiomatico per le conversioni da stringa — evita eccezioni e si legge bene.
- L'interpolazione `$"..."` è la via C# a `String.format()` — più compatta e leggibile.
