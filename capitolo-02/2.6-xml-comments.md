# 2.6 Commenti XML e documentazione del codice

> **Analogia Java**: equivalente di Javadoc

---

## Introduzione

In Java documenti il codice con Javadoc — blocchi `/** */` sopra classi e metodi, che `javadoc` trasforma in HTML. In C# si usa la stessa idea con una sintassi diversa: i commenti XML con `///`. Il risultato è identico — IntelliSense mostra la documentazione mentre scrivi, e strumenti come Swagger la leggono automaticamente.

---

```java
     * @throws IllegalArgumentException se {@code id} è negativo
     */
    public Order getById(int id) {
        // ...
    }
}
```

## I commenti XML in C#

```csharp
/// <summary>
/// Servizio per la gestione degli ordini.
/// Fornisce operazioni CRUD sull'entità <see cref="Order"/>.
/// </summary>
public class OrderService
{
    /// <summary>
    /// Recupera un ordine per ID.
    /// </summary>
    /// <param name="id">L'identificativo dell'ordine.</param>
    /// <returns>L'ordine trovato, o <c>null</c> se non esiste.</returns>
    /// <exception cref="ArgumentException">Se <paramref name="id"/> è negativo.</exception>
    public Order? GetById(int id)
    {
        // ...
    }
}
```

Tre slash `///` — il compilatore riconosce il blocco come documentazione XML. VS Code e Visual Studio mostrano il contenuto nel tooltip mentre scrivi.

## I tag principali

```csharp
/// <summary>Descrizione breve — sempre presente.</summary>

/// <param name="id">Descrizione del parametro.</param>

/// <returns>Cosa ritorna il metodo.</returns>

/// <exception cref="KeyNotFoundException">Quando viene lanciata.</exception>

/// <remarks>
/// Note aggiuntive, dettagli di implementazione, esempi d'uso.
/// Usalo per tutto quello che non entra nel summary.
/// </remarks>

/// <example>
/// <code>
/// var order = service.GetById(42);
/// </code>
/// </example>

/// <see cref="OrderDto"/>          <!-- link a un tipo -->
/// <seealso cref="IOrderService"/> <!-- link correlato, in fondo -->

/// <c>null</c>      <!-- codice inline, come {@code} in Javadoc -->
/// <paramref name="id"/>  <!-- riferimento a un parametro -->
```

## Documentare record e interfacce

```csharp
/// <summary>Rappresenta un ordine nel sistema.</summary>
/// <param name="Id">Identificativo univoco.</param>
/// <param name="Customer">Nome del cliente.</param>
/// <param name="Total">Totale in euro, IVA inclusa.</param>
/// <param name="Status">Stato corrente dell'ordine.</param>
/// <param name="CreatedAt">Data di creazione in UTC.</param>
public record OrderDto(int Id, string Customer, decimal Total, string Status, DateTime CreatedAt);

/// <summary>Contratto per il service degli ordini.</summary>
public interface IOrderService
{
    /// <summary>Restituisce tutti gli ordini.</summary>
    Task<IEnumerable<OrderDto>> GetAllAsync(CancellationToken ct = default);

    /// <summary>Restituisce un ordine per ID.</summary>
    /// <param name="id">L'ID dell'ordine.</param>
    /// <returns><c>null</c> se l'ordine non esiste.</returns>
    Task<OrderDto?> GetByIdAsync(int id, CancellationToken ct = default);
}
```

## Swagger legge i commenti XML

Questo è il vantaggio pratico che vedrai in cap 4: se abiliti la generazione del file XML nel `.csproj`, Swagger usa i commenti come descrizione degli endpoint — senza scrivere niente in più.

```xml
<!-- OrderService.Api.csproj -->
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
</PropertyGroup>
```

```csharp
// Program.cs
builder.Services.AddOpenApi(options =>
{
    options.AddDocumentTransformer((doc, context, ct) =>
    {
        doc.Info.Title = "OrderService API";
        return Task.CompletedTask;
    });
});
```

Il `/// <summary>` del controller diventa la descrizione dell'endpoint in Swagger UI. Scrivi la documentazione una volta, la usi ovunque.

---

## Quanto documentare?

Non tutto ha bisogno di un commento XML. Una regola pratica:

- **Sì** — interfacce pubbliche, metodi pubblici di classi di servizio, DTO condivisi tra layer
- **No** — implementazioni private ovvie, metodi il cui nome dice già tutto (`GetById`, `Save`)
- **Sì** — comportamenti non ovvi: quando ritorna `null`, quali eccezioni può lanciare, precondizioni

Un commento che ripete il nome del metodo è rumore. Un commento che spiega il *perché* o il *quando* è valore.

## Confronto con Java

| Javadoc | C# XML comments |
|---|---|
| `/** */` | `///` |
| `@param name desc` | `<param name="name">desc</param>` |
| `@return desc` | `<returns>desc</returns>` |
| `@throws ExType desc` | `<exception cref="ExType">desc</exception>` |
| `{@link ClassName}` | `<see cref="ClassName"/>` |
| `{@code value}` | `<c>value</c>` |
| `javadoc` genera HTML | `<GenerateDocumentationFile>` genera XML |
| SpringDoc legge Javadoc | Swagger legge i commenti XML |
