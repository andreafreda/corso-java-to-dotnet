# 10.4 Cenni su Kubernetes: deploy del microservizio

> **Analogia Java**: Deployment, Service, ConfigMap — stesso modello

---

## Introduzione

Kubernetes è lo standard per orchestrare microservizi in produzione. I concetti sono identici tra Java e .NET — `Deployment`, `Service`, `ConfigMap`, `Secret`. Cambia solo l'immagine Docker. Questo paragrafo mostra i manifest minimi per deployare `OrderService` su un cluster Kubernetes.

---

## I concetti chiave

```
Pod          → una o più istanze del container — l'unità base
Deployment   → gestisce i Pod: quanti, quale immagine, rolling update
Service      → endpoint stabile per un Deployment (DNS interno al cluster)
ConfigMap    → configurazione non sensibile (equivalente di application.yml)
Secret       → configurazione sensibile (connection string, JWT key)
Ingress      → espone il Service all'esterno del cluster via HTTP/HTTPS
```

---

## Struttura dei manifest

```
k8s/
├── deployment.yaml
├── service.yaml
├── configmap.yaml
├── secret.yaml
└── ingress.yaml
```

## `configmap.yaml` — configurazione non sensibile

```yaml
# k8s/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: order-service-config
  namespace: default
data:
  ASPNETCORE_ENVIRONMENT: "Production"
  Jwt__Issuer:            "order-service"
  Jwt__Audience:          "order-service-api"
  Services__InventoryUrl: "http://inventory-service"
  Serilog__MinimumLevel__Default: "Information"
```

---

## `secret.yaml` — configurazione sensibile

```yaml
# k8s/secret.yaml
# I valori sono Base64 encoded — NON cifrati
# In produzione usa Azure Key Vault + External Secrets Operator
apiVersion: v1
kind: Secret
metadata:
  name: order-service-secrets
  namespace: default
type: Opaque
data:
  # echo -n "valore" | base64
  ConnectionStrings__Default: U2VydmVyPW15c3FsO0RhdGFiYXNlPU9yZGVyRGI7...
  Jwt__Key:                    c3VwZXItc2VjcmV0LWtleQ==
  ConnectionStrings__Redis:    cmVkaXM6NjM3OQ==
```

> I `Secret` Kubernetes sono Base64, non cifrati. In produzione usa **Azure Key Vault** con External Secrets Operator o il CSI Secret Store driver.

## `deployment.yaml`

```yaml
# k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: order-service
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
        - name: order-service
          image: myregistry.azurecr.io/order-service:1.0.0
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: order-service-config   # inietta tutte le chiavi del ConfigMap
            - secretRef:
                name: order-service-secrets  # inietta tutti i segreti
          resources:
            requests:
              memory: "128Mi"
              cpu: "100m"
            limits:
              memory: "256Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /health/live
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /health/ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
```

## `service.yaml`

```yaml
# k8s/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: order-service
  namespace: default
spec:
  selector:
    app: order-service
  ports:
    - port: 80          # porta esposta all'interno del cluster
      targetPort: 8080  # porta del container
  type: ClusterIP       # solo interno al cluster — usa Ingress per l'esterno
```

---

## `ingress.yaml` — espone il servizio all'esterno

```yaml
# k8s/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: order-service-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
    - host: api.orderservice.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: order-service
                port:
                  number: 80
```

## Deploy

```bash
# Applica tutti i manifest
kubectl apply -f k8s/

# Verifica lo stato
kubectl get pods
kubectl get services
kubectl get deployments

# Log di un pod
kubectl logs -l app=order-service --tail=100 -f

# Rolling update — aggiorna l'immagine
kubectl set image deployment/order-service \
    order-service=myregistry.azurecr.io/order-service:1.1.0

# Rollback all'ultima versione funzionante
kubectl rollout undo deployment/order-service

# Scale orizzontale
kubectl scale deployment/order-service --replicas=4
```

---

## Spiegazione

**`replicas: 2`** — almeno due repliche per alta disponibilità. Kubernetes distribuisce le repliche tra i nodi del cluster. Se un nodo cade, le repliche sull'altro nodo continuano a servire il traffico.

**`resources.requests` e `limits`** — `requests` è la risorsa garantita al pod, `limits` è il massimo. Senza limits un pod può consumare tutta la CPU/memoria del nodo e affamare gli altri. In produzione imposta sempre entrambi.

**`ClusterIP` vs `LoadBalancer`** — `ClusterIP` espone il servizio solo all'interno del cluster — è la scelta corretta per microservizi che parlano tra loro. `LoadBalancer` provvede un IP pubblico (via cloud provider) — usato solo per l'entry point esterno, meglio gestito con `Ingress`.

**`envFrom` vs `env`** — `envFrom` inietta tutte le chiavi del ConfigMap/Secret come variabili d'ambiente. `env` permette di selezionare chiavi specifiche e rinominarle. Per semplicità usiamo `envFrom` — tutte le chiavi del ConfigMap diventano variabili d'ambiente con lo stesso nome.

## Confronto con Java

| Spring Boot su K8s | ASP.NET Core su K8s |
|---|---|
| Stessa struttura manifest | Stessa struttura manifest |
| `SPRING_DATASOURCE_URL` in ConfigMap | `ConnectionStrings__Default` in ConfigMap |
| `SPRING_PROFILES_ACTIVE=production` | `ASPNETCORE_ENVIRONMENT=Production` |
| `/actuator/health/liveness` | `/health/live` |
| `/actuator/health/readiness` | `/health/ready` |
| `server.port=8080` in `application.yml` | `ASPNETCORE_URLS=http://+:8080` |
