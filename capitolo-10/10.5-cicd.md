# 10.5 CI/CD con GitHub Actions per .NET

> **Analogia Java**: Build, test e push immagine Docker automatizzati

---

## Introduzione

Ogni `git push` su `main` dovrebbe produrre automaticamente: build, test, immagine Docker, deploy. In Java usavi GitHub Actions con Maven. In .NET il workflow è identico — cambia solo la CLI (`dotnet` invece di `mvn`).

---

## Come si faceva in Java

```yaml
# .github/workflows/ci.yml — Java/Spring Boot
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - run: mvn verify
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: myregistry/order-service:${{ github.sha }}
```

## Workflow CI/CD per .NET

```yaml
# .github/workflows/ci.yml
name: CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY:    ghcr.io
  IMAGE_NAME:  ${{ github.repository }}/order-service

jobs:

  # ── Build e Test ─────────────────────────────────────────────────────
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore dependencies
        run: dotnet restore

      - name: Build
        run: dotnet build --no-restore --configuration Release

      - name: Test
        run: dotnet test --no-build --configuration Release --verbosity normal \
               --collect:"XPlat Code Coverage"

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          files: '**/coverage.cobertura.xml'

  # ── Build e Push immagine Docker ─────────────────────────────────────
  docker:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # solo su push a main, non su PR

    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=sha-
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha     # cache layer Docker tra i run
          cache-to:   type=gha,mode=max

  # ── Deploy su Kubernetes ──────────────────────────────────────────────
  deploy:
    needs: docker
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    environment: production   # richiede approvazione manuale se configurato

    steps:
      - uses: actions/checkout@v4

      - name: Set up kubectl
        uses: azure/setup-kubectl@v3

      - name: Set AKS context
        uses: azure/aks-set-context@v3
        with:
          resource-group: ${{ secrets.AKS_RESOURCE_GROUP }}
          cluster-name:   ${{ secrets.AKS_CLUSTER_NAME }}
          creds:          ${{ secrets.AZURE_CREDENTIALS }}

      - name: Deploy to AKS
        run: |
          kubectl set image deployment/order-service \
            order-service=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:sha-${{ github.sha }}
          kubectl rollout status deployment/order-service --timeout=120s
```

## Secrets in GitHub Actions

```bash
# Configura i secrets nel repository GitHub
# Settings → Secrets and variables → Actions

# Credenziali Azure per il deploy
AZURE_CREDENTIALS          # output di: az ad sp create-for-rbac --sdk-auth
AKS_RESOURCE_GROUP         # nome del resource group
AKS_CLUSTER_NAME           # nome del cluster AKS
```

## Workflow per PR — solo build e test

```yaml
# .github/workflows/pr.yml — workflow leggero per le PR
name: PR Check

on:
  pull_request:
    branches: [main]

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - run: dotnet restore
      - run: dotnet build --no-restore
      - run: dotnet test --no-build

      # Formato del codice — fallisce se il codice non è formattato
      - run: dotnet format --verify-no-changes
```

---

## Spiegazione

**`needs: test`** — il job `docker` parte solo se `test` è passato. `needs: docker` garantisce che il deploy parta solo se l'immagine è stata pushata. È la dipendenza tra job che garantisce il flusso corretto.

**`cache-from: type=gha`** — GitHub Actions Cache per i layer Docker. Senza cache ogni build scarica tutte le dipendenze NuGet e ricompila tutto. Con cache il layer `dotnet restore` viene riusato se `*.csproj` non è cambiato — tempo di build dimezzato.

**`environment: production`** — GitHub Environments permette di richiedere una review manuale prima del deploy. Configuri i reviewer in `Settings → Environments` — nessun codice aggiuntivo.

**`dotnet format --verify-no-changes`** — verifica che il codice sia formattato secondo le regole di `.editorconfig`. Fallisce la PR se qualcuno ha dimenticato di formattare. Equivalente di `mvn checkstyle:check` in Java.

**Tagging con SHA del commit** — taggare l'immagine con lo SHA del commit (`sha-abc1234`) permette di tracciare esattamente quale commit è in produzione e di fare rollback a una versione specifica con `kubectl set image`.

## Confronto con Java

| Java (GitHub Actions + Maven) | .NET (GitHub Actions) |
|---|---|
| `actions/setup-java` | `actions/setup-dotnet` |
| `mvn verify` | `dotnet test` |
| `mvn package -DskipTests` | `dotnet publish` (nel Dockerfile) |
| Fat JAR come artifact | Immagine Docker come artifact |
| `mvn checkstyle:check` | `dotnet format --verify-no-changes` |
| Stessa struttura jobs: test → docker → deploy | Stessa struttura |
