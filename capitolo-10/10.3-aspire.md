# 10.3 .NET Aspire: orchestrazione locale moderna

> **Analogia Java**: alternativa a docker-compose, integrata con .NET

---

## Introduzione

`docker-compose.yml` funziona ma è disconnesso dall'applicazione — se aggiungi un servizio devi aggiornare sia il codice che il Compose. **.NET Aspire** è un'alternativa moderna: definisci l'intero stack in C#, ottieni una dashboard di osservabilità integrata, service discovery automatico e zero configurazione manuale delle connection string. È la risposta .NET al problema dell'orchestrazione locale.

> **Nota**: .NET Aspire è un'alternativa a docker-compose per lo sviluppo locale — non un sostituto di Kubernetes in produzione.

---

## Come si differenzia da docker-compose

| docker-compose | .NET Aspire |
|---|---|
| YAML esterno al codice | C# — stesso linguaggio dell'app |
| Connection string hardcoded | Iniettate automaticamente da Aspire |
| Nessuna dashboard integrata | Dashboard con logs, traces, metrics inclusa |
| Restart manuale dopo code change | Hot reload integrato |
| Service discovery manuale | Automatico — Aspire gestisce i nomi |

## Setup

```bash
# Installa il workload Aspire
dotnet workload install aspire

# Aggiungi i progetti Aspire alla solution
dotnet new aspire-apphost -n OrderService.AppHost
dotnet new aspire-servicedefaults -n OrderService.ServiceDefaults
dotnet sln add OrderService.AppHost
dotnet sln add OrderService.ServiceDefaults

# Referenzia ServiceDefaults da Api
dotnet add OrderService.Api reference OrderService.ServiceDefaults
```

## `AppHost/Program.cs` — definisce lo stack

```csharp
// OrderService.AppHost/Program.cs
var builder = DistributedApplication.CreateBuilder(args);

// Database
var sql = builder.AddSqlServer("azuresql")
    .WithDataVolume()
    .AddDatabase("Default");  // Aspire inietta ConnectionStrings:Default — deve corrispondere al key usato in codice

// Cache
var redis = builder.AddRedis("redis");

// Service Bus emulator
var serviceBus = builder.AddAzureServiceBus("servicebus")
    .RunAsEmulator();

// Il microservizio — riceve le dipendenze come reference
var orderService = builder.AddProject<Projects.OrderService_Api>("orderservice")
    .WithReference(sql)
    .WithReference(redis)
    .WithReference(serviceBus)
    .WithHttpEndpoint(port: 8080);

builder.Build().Run();
```

Aspire inietta automaticamente le connection string come variabili d'ambiente — non devi scriverle a mano.

## `ServiceDefaults/Extensions.cs` — configurazione condivisa

```csharp
// OrderService.ServiceDefaults/Extensions.cs
public static class Extensions
{
    public static IHostApplicationBuilder AddServiceDefaults(
        this IHostApplicationBuilder builder)
    {
        builder.ConfigureOpenTelemetry();
        builder.AddDefaultHealthChecks();
        builder.Services.AddServiceDiscovery();
        builder.Services.ConfigureHttpClientDefaults(http =>
        {
            http.AddStandardResilienceHandler();
            http.AddServiceDiscovery();
        });
        return builder;
    }
}
```

```csharp
// Api/Program.cs — una riga per attivare tutti i default
builder.AddServiceDefaults();
```

## Service discovery automatico

Con Aspire non usi URL hardcoded — usi il nome del servizio:

```csharp
// Infrastructure/DependencyInjection.cs — con Aspire
services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    // "inventoryservice" è il nome registrato nell'AppHost
    client.BaseAddress = new Uri("http://inventoryservice");
});
```

Aspire risolve `http://inventoryservice` all'endpoint corretto — in locale è `localhost:porta-assegnata`, in produzione è il DNS del cluster.

## Spiegazione

**Aspire vs docker-compose** — docker-compose è universale (qualsiasi linguaggio, qualsiasi stack). Aspire è specifico .NET — vale la complessità aggiuntiva se il tuo team lavora interamente in .NET e vuole osservabilità integrata senza configurazione.

**`ServiceDefaults`** — il progetto `ServiceDefaults` è lo stesso che vedi in tutti i template Aspire. Centralizza la configurazione di OpenTelemetry, health checks e resilienza — ogni microservizio chiama `AddServiceDefaults()` e ottiene tutto. È il pattern di standardizzazione per i team con più microservizi.

**Produzione** — Aspire non va in produzione. L'`AppHost` è solo per sviluppo locale. In produzione usi Kubernetes (10.4) o Azure Container Apps — Aspire genera i manifest Kubernetes con `dotnet aspire publish`.
