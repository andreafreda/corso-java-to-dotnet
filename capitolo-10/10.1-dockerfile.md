# 10.1 Dockerfile per un'applicazione .NET 10

> **Analogia Java**: multi-stage build con Maven + JDK/JRE

---

## Introduzione

Containerizzare `OrderService` significa produrre un'immagine Docker che gira identicamente in sviluppo, staging e produzione. In Java il Dockerfile tipico copiava il fat JAR prodotto da Maven. In .NET si usa un **multi-stage build** — il pattern ufficiale che separa la fase di build dalla fase di runtime, producendo immagini leggere e sicure.

---

## Come si faceva in Java

```dockerfile
# Java — multi-stage con Maven
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre-alpine AS runtime
WORKDIR /app
COPY --from=build /app/target/order-service-*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

---

## Dockerfile per .NET 10

```dockerfile
# Dockerfile — multi-stage build per OrderService
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src

# Copia i file di progetto e ripristina le dipendenze
# Layer separato — viene cachato se i .csproj non cambiano
COPY OrderService.sln .
COPY OrderService.Api/OrderService.Api.csproj             OrderService.Api/
COPY OrderService.Core/OrderService.Core.csproj           OrderService.Core/
COPY OrderService.Infrastructure/OrderService.Infrastructure.csproj OrderService.Infrastructure/
RUN dotnet restore

# Copia il codice sorgente e pubblica
COPY . .
RUN dotnet publish OrderService.Api \
    --configuration Release \
    --no-restore \
    --output /app/publish

# Runtime stage — immagine finale leggera
FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS runtime
WORKDIR /app

# Utente non-root per sicurezza — --chown garantisce ownership corretta
# USER prima di COPY non ha effetto: Docker copia sempre come root
RUN adduser --disabled-password --gecos "" appuser
COPY --chown=appuser:appuser --from=build /app/publish .
USER appuser

EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080

ENTRYPOINT ["dotnet", "OrderService.Api.dll"]
```

## Le immagini base .NET

```
mcr.microsoft.com/dotnet/sdk:10.0
    ↑ Include compilatore, CLI, tutto — solo per la build stage
    ↑ ~800MB

mcr.microsoft.com/dotnet/aspnet:10.0
    ↑ Solo il runtime ASP.NET Core — per la runtime stage
    ↑ ~200MB

mcr.microsoft.com/dotnet/runtime:10.0
    ↑ Solo il runtime .NET (no ASP.NET) — per worker/console app
    ↑ ~180MB

mcr.microsoft.com/dotnet/runtime-deps:10.0
    ↑ Solo le dipendenze native — per Native AOT (cap. 11)
    ↑ ~10MB
```

## `.dockerignore`

```
# .dockerignore — esclude file non necessari dalla build context
**/bin/
**/obj/
**/.vs/
**/.git/
**/logs/
**/TestResults/
*.md
.env
appsettings.Development.json
```

Senza `.dockerignore` Docker copia `bin/` e `obj/` nel build context — rallenta il trasferimento e può causare conflitti.

## Build e run locale

```bash
# Build dell'immagine
docker build -t order-service:latest .

# Run locale
docker run -p 8080:8080 \
  -e ConnectionStrings__Default="Server=host.docker.internal,1433;..." \
  -e Jwt__Key="my-secret-key" \
  order-service:latest

# Verifica
curl http://localhost:8080/health/ready
```

## Ottimizzazioni

### Layer caching per le dipendenze

```dockerfile
# ✓ Copia i .csproj mantenendo la struttura di directory — layer cachato
# ⚠ Non usare COPY **/*.csproj ./ — appiattisce le directory e rompe dotnet restore
COPY OrderService.sln .
COPY OrderService.Api/OrderService.Api.csproj             OrderService.Api/
COPY OrderService.Core/OrderService.Core.csproj           OrderService.Core/
COPY OrderService.Infrastructure/OrderService.Infrastructure.csproj OrderService.Infrastructure/
RUN dotnet restore

# ✓ Poi copia il codice — il layer restore non viene rieseguito se i .csproj non cambiano
COPY . .
RUN dotnet publish ...
```

### Self-contained vs framework-dependent

```bash
# Self-contained — include il runtime .NET nell'immagine
# Immagine più grande ma zero dipendenze esterne
dotnet publish OrderService.Api \
    --configuration Release \
    --runtime linux-x64 \
    --self-contained true \
    --output /app/publish

# Framework-dependent (default) — usa il runtime dall'immagine base
# Immagine più piccola, dipende da mcr.microsoft.com/dotnet/aspnet
dotnet publish OrderService.Api \
    --configuration Release \
    --output /app/publish
```

Per microservizi usa **framework-dependent** con l'immagine `aspnet` — immagini più piccole e aggiornamenti del runtime gestiti dall'immagine base.

## Spiegazione

**`dotnet publish` vs `dotnet build`** — `build` compila il codice. `publish` prepara l'output per il deploy: ottimizza, risolve i file di runtime, produce la struttura finale. Usa sempre `publish` nel Dockerfile.

**`--no-restore`** — il restore è già stato fatto nel layer precedente. `--no-restore` evita di rifarlo durante il publish — risparmia tempo e garantisce che vengano usate esattamente le dipendenze già scaricate.

**Utente non-root** — per default i container girano come root. Creare un utente dedicato (`appuser`) riduce la superficie di attacco: se il container viene compromesso, l'attaccante non ha privilegi root sull'host.

**`ASPNETCORE_URLS=http://+:8080`** — ASP.NET Core di default ascolta su porta 5000/5001. In un container è convenzione usare la porta 8080 (non privilegiata). Kubernetes e i load balancer si aspettano questa configurazione.

## Confronto con Java

| Java | .NET |
|---|---|
| `FROM maven:3.9 AS build` | `FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build` |
| `mvn package -DskipTests` | `dotnet publish --configuration Release` |
| `FROM eclipse-temurin:21-jre-alpine AS runtime` | `FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS runtime` |
| Fat JAR (`app.jar`) | DLL + dipendenze (`OrderService.Api.dll`) |
| `ENTRYPOINT ["java", "-jar", "app.jar"]` | `ENTRYPOINT ["dotnet", "OrderService.Api.dll"]` |
| `server.port=8080` in `application.yml` | `ASPNETCORE_URLS=http://+:8080` |
