# 10.2 docker-compose per l'ambiente locale multi-servizio

> **Analogia Java**: Compose con DB, Azure Service Bus emulato e microservizi

---

## Introduzione

Nei capitoli precedenti il `docker-compose.yml` cresceva incrementalmente — prima Azure SQL, poi Service Bus emulator. Ora lo consolidiamo: un file completo che avvia l'intero stack locale con un solo `docker compose up`. `OrderService` compilato e pronto, database, messaggistica, Redis, tutto coordinato.

---

## Come si faceva in Java

```yaml
# Spring Boot — docker-compose.yml tipico
services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:sqlserver://db:1433;databaseName=OrderDb;encrypt=false
      SPRING_DATASOURCE_USERNAME: sa
      SPRING_DATASOURCE_PASSWORD: Password123!
    depends_on:
      db:
        condition: service_healthy

  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: "Y"
      SA_PASSWORD: "Password123!"
    healthcheck:
      test: ["CMD", "/opt/mssql-tools/bin/sqlcmd", "-S", "localhost", "-U", "sa", "-P", "Password123!", "-Q", "SELECT 1"]
      interval: 10s
      retries: 5
```

## `docker-compose.yml` completo

```yaml
# docker-compose.yml
services:

  # ── Database ────────────────────────────────────────────────────────
  azuresql:
    image: mcr.microsoft.com/azure-sql-edge:latest
    environment:
      ACCEPT_EULA: "Y"
      MSSQL_SA_PASSWORD: "Password123!"
    ports:
      - "1433:1433"
    healthcheck:
      test: ["CMD", "/opt/mssql-tools/bin/sqlcmd",
             "-S", "localhost", "-U", "sa", "-P", "Password123!",
             "-Q", "SELECT 1"]
      interval: 10s
      retries: 10
      start_period: 30s
    volumes:
      - sqldata:/var/opt/mssql

  # ── Messaggistica ───────────────────────────────────────────────────
  servicebus:
    image: mcr.microsoft.com/azure-messaging/servicebus-emulator:latest
    environment:
      ACCEPT_EULA: "Y"
    ports:
      - "5672:5672"
      - "9090:9090"
    volumes:
      - ./servicebus-config.json:/ServiceBus_Emulator/ConfigFiles/Config.json

  # ── Cache ────────────────────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

  # ── Applicazione ─────────────────────────────────────────────────────
  orderservice:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    environment:
      ASPNETCORE_ENVIRONMENT: "Development"
      ConnectionStrings__Default: "Server=azuresql,1433;Database=OrderDb;User Id=sa;Password=Password123!;TrustServerCertificate=True"
      ConnectionStrings__ServiceBus: "Endpoint=sb://servicebus;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=SAS_KEY_VALUE;UseDevelopmentEmulator=true;"
      ConnectionStrings__Redis: "redis:6379"
      Jwt__Key: "super-secret-key-almeno-32-caratteri-per-hs256"
      Jwt__Issuer: "order-service"
      Jwt__Audience: "order-service-api"
      Services__InventoryUrl: "http://inventoryservice:8081"
    depends_on:
      azuresql:
        condition: service_healthy
      servicebus:
        condition: service_started
      redis:
        condition: service_started

  # ── Inventory Service (simulato) ─────────────────────────────────────
  inventoryservice:
    image: mcr.microsoft.com/dotnet/samples:aspnetapp  # placeholder
    ports:
      - "8081:8080"

volumes:
  sqldata:
```

## Override per ambienti diversi

```yaml
# docker-compose.override.yml — override per sviluppo locale
# Viene caricato automaticamente da docker compose
services:
  orderservice:
    environment:
      Serilog__MinimumLevel__Default: "Debug"
    volumes:
      - ./logs:/app/logs   # monta la cartella log locale
```

```yaml
# docker-compose.prod.yml — override per produzione (non versionato)
services:
  orderservice:
    image: myregistry.azurecr.io/order-service:1.2.3
    environment:
      ASPNETCORE_ENVIRONMENT: "Production"
```

```bash
# Sviluppo — carica docker-compose.yml + docker-compose.override.yml automaticamente
docker compose up -d

# Produzione — specifica esplicitamente
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## Migration automatica all'avvio

Il container `orderservice` dipende dal database ma le migration non vengono applicate automaticamente. Tre approcci:

### Approccio 1 — `MigrateAsync` in `Program.cs` (semplice)

```csharp
// Api/Program.cs — solo in Development
if (app.Environment.IsDevelopment())
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();
}
```

### Approccio 2 — Init container (produzione)

`--migrate-only` non è un argomento nativo di ASP.NET Core — va gestito esplicitamente in `Program.cs`:

```csharp
// Api/Program.cs — da aggiungere prima di app.Run()
if (args.Contains("--migrate-only"))
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();
    return;  // termina senza avviare il server HTTP
}
```

```yaml
# docker-compose.yml — container che applica le migration e termina
  migrate:
    build:
      context: .
      dockerfile: Dockerfile
    command: ["dotnet", "OrderService.Api.dll", "--migrate-only"]
    environment:
      ConnectionStrings__Default: "Server=azuresql,1433;..."
    depends_on:
      azuresql:
        condition: service_healthy
```

### Approccio 3 — `dotnet ef database update` nella CI/CD pipeline

Il più sicuro in produzione — le migration vengono applicate come step separato prima del deploy, con possibilità di review e rollback.

## Comandi utili

```bash
# Avvia tutto
docker compose up -d

# Avvia solo i servizi di supporto (senza l'app)
docker compose up -d azuresql servicebus redis

# Log in tempo reale
docker compose logs -f orderservice

# Rebuild solo dell'applicazione
docker compose up -d --build orderservice

# Stop e rimozione volumi (reset completo)
docker compose down -v

# Stato dei servizi
docker compose ps
```

---

## Spiegazione

**`depends_on` con `condition: service_healthy`** — garantisce che il container del database sia in stato `healthy` prima di avviare l'applicazione. Senza questa condizione, l'app potrebbe tentare la connessione prima che SQL Server sia pronto — e fallire.

**`healthcheck` su Azure SQL Edge** — SQL Server impiega 20-30 secondi per avviarsi. Il `healthcheck` con `start_period: 30s` evita falsi negativi durante lo startup. `retries: 10` dà 100 secondi di tempo totale.

**Variabili d'ambiente con `__`** — ASP.NET Core converte automaticamente `ConnectionStrings__Default` in `ConnectionStrings:Default`. Il doppio underscore è il separatore per le chiavi gerarchiche nelle variabili d'ambiente — lo stesso che Kubernetes usa nei `ConfigMap`.

**Volume `sqldata`** — i dati del database sopravvivono al `docker compose down`. `docker compose down -v` rimuove anche i volumi — utile per un reset pulito.

## Confronto con Java

| Spring Boot / Docker | ASP.NET Core / Docker |
|---|---|
| `SPRING_DATASOURCE_URL` | `ConnectionStrings__Default` |
| `SPRING_PROFILES_ACTIVE=dev` | `ASPNETCORE_ENVIRONMENT=Development` |
| `docker-compose.override.yml` | `docker-compose.override.yml` — stesso pattern |
| Flyway migration all'avvio | `MigrateAsync()` all'avvio / init container |
| `healthcheck` con `curl /actuator/health` | `healthcheck` con `curl /health/live` |
