# 7.3 JWT: struttura, generazione e validazione del token

> **Analogia Java**: equivalente di Spring Security + JJWT

---

## Introduzione

JWT (JSON Web Token) è lo standard per trasportare claims in modo sicuro tra servizi. La struttura è indipendente dal linguaggio — un token generato in .NET è validabile in Java e viceversa. Questo paragrafo mostra come generare e validare JWT in ASP.NET Core con `System.IdentityModel.Tokens.Jwt`.

---

## Struttura di un JWT

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9    ← Header (Base64)
.
eyJzdWIiOiIxMjMiLCJlbWFpbCI6InVzZXJAZXhhbXBsZS5jb20iLCJyb2xlIjoiQWRtaW4iLCJleHAiOjE3MDAwMDAwMDB9
.                                          ← Payload (Base64)
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c  ← Signature (HMAC-SHA256)
```

```json
// Header decodificato
{ "alg": "HS256", "typ": "JWT" }

// Payload decodificato
{
  "sub": "123",
  "email": "user@example.com",
  "role": "Admin",
  "exp": 1700000000,
  "iss": "order-service",
  "aud": "order-service-api"
}
```

Il payload è **Base64 encodato, non cifrato** — non mettere dati sensibili (password, numeri di carta) nei claims.

## Come si faceva in Java

```java
// JJWT — generazione token
String token = Jwts.builder()
    .subject(user.getId().toString())
    .claim("email", user.getEmail())
    .claim("role", user.getRole())
    .issuedAt(new Date())
    .expiration(new Date(System.currentTimeMillis() + 3600_000))
    .signWith(Keys.hmacShaKeyFor(secretKey.getBytes()))
    .compact();
```

```java
// JJWT — validazione token
Claims claims = Jwts.parser()
    .verifyWith(Keys.hmacShaKeyFor(secretKey.getBytes()))
    .build()
    .parseSignedClaims(token)
    .getPayload();

String userId = claims.getSubject();
String role   = claims.get("role", String.class);
```

```java
// Spring Security — configurazione validazione JWT automatica
http.oauth2ResourceServer(oauth2 -> oauth2
    .jwt(jwt -> jwt.decoder(NimbusJwtDecoder
        .withSecretKey(new SecretKeySpec(secretKey.getBytes(), "HmacSHA256"))
        .build())));
```

## Generazione del token

```csharp
// Core/Options/JwtSettings.cs
namespace OrderService.Core.Options;

public record JwtSettings(
    string Key,
    string Issuer,
    string Audience,
    int    ExpiryMinutes);
```

```csharp
// Core/Services/TokenService.cs
using System.IdentityModel.Tokens.Jwt;
using System.Security.Claims;
using System.Text;
using Microsoft.Extensions.Options;
using Microsoft.IdentityModel.Tokens;
using OrderService.Core.Options;

namespace OrderService.Core.Services;

public class TokenService
{
    private readonly JwtSettings _settings;

    // IOptions<T> — il Core riceve i valori già bindati, non IConfiguration
    public TokenService(IOptions<JwtSettings> options)
    {
        _settings = options.Value;
    }

    public string GenerateToken(string userId, string email, string role)
    {
        var key         = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_settings.Key));
        var credentials = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
        var expiry      = DateTime.UtcNow.AddMinutes(_settings.ExpiryMinutes);

        var claims = new[]
        {
            new Claim(JwtRegisteredClaimNames.Sub,   userId),
            new Claim(JwtRegisteredClaimNames.Email, email),
            new Claim(ClaimTypes.Role,               role),
            new Claim(JwtRegisteredClaimNames.Jti,   Guid.NewGuid().ToString())
        };

        var token = new JwtSecurityToken(
            issuer:             _settings.Issuer,
            audience:           _settings.Audience,
            claims:             claims,
            expires:            expiry,
            signingCredentials: credentials);

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
```

## Validazione del token — configurazione in `Program.cs`

```csharp
// Api/Program.cs
using System.Text;
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;

// Binda JwtSettings — IOptions<JwtSettings> disponibile in tutto il DI
builder.Services.Configure<JwtSettings>(builder.Configuration.GetSection("Jwt"));

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer           = true,
            ValidateAudience         = true,
            ValidateLifetime         = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer              = builder.Configuration["Jwt:Issuer"],
            ValidAudience            = builder.Configuration["Jwt:Audience"],
            IssuerSigningKey         = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!)),
            ClockSkew                = TimeSpan.Zero  // nessuna tolleranza sull'expiry
        };
    });

builder.Services.AddAuthorization();
// TokenService e AuthService registrati in Core/DependencyInjection.cs (vedi 7.4)
```

`ClockSkew` di default è 5 minuti — ASP.NET Core accetta token scaduti da meno di 5 minuti. `TimeSpan.Zero` lo disabilita per ambienti dove la precisione è importante.

## Leggere il token decodificato nel controller

```csharp
// Nel controller — HttpContext.User è già popolato da UseAuthentication
[HttpGet("me")]
[Authorize]
public IActionResult GetCurrentUser()
{
    var userId = User.FindFirst(JwtRegisteredClaimNames.Sub)?.Value;
    var email  = User.FindFirst(JwtRegisteredClaimNames.Email)?.Value;
    var role   = User.FindFirst(ClaimTypes.Role)?.Value;

    return Ok(new { userId, email, role });
}
```

## Spiegazione

**`JwtRegisteredClaimNames` vs `ClaimTypes`** — `JwtRegisteredClaimNames` contiene i nomi standard JWT (`sub`, `email`, `jti`). `ClaimTypes` contiene i nomi namespace Microsoft più verbosi (`http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier`). ASP.NET Core mappa automaticamente `sub` → `ClaimTypes.NameIdentifier`, ma è più chiaro usare `JwtRegisteredClaimNames.Sub` quando generi il token e `ClaimTypes.NameIdentifier` quando lo leggi — o usare sempre `JwtRegisteredClaimNames` per consistenza.

**`Jti` claim** — è l'ID univoco del token (JWT ID). Utile per il token revocation: se mantieni una lista di `jti` revocati puoi invalidare token specifici prima della loro scadenza naturale.

**Chiave simmetrica vs asimmetrica** — `HS256` usa una chiave simmetrica: chi genera e chi valida usano la stessa chiave. È semplice ma richiede che tutti i servizi abbiano la stessa chiave segreta. `RS256` usa una coppia pubblica/privata: solo l'AuthServer conosce la chiave privata, i servizi validano con la chiave pubblica. Preferisci `RS256` in ambienti multi-servizio — vedi 7.4.

## Confronto con Java

| JJWT / Spring Security | ASP.NET Core |
|---|---|
| `Jwts.builder().signWith(key).compact()` | `new JwtSecurityTokenHandler().WriteToken(token)` |
| `Jwts.parser().verifyWith(key).parseSignedClaims(token)` | Validazione automatica in `AddJwtBearer` |
| `claims.getSubject()` | `User.FindFirst(JwtRegisteredClaimNames.Sub)` |
| `claims.get("role", String.class)` | `User.FindFirst(ClaimTypes.Role)` |
| `Keys.hmacShaKeyFor(bytes)` | `new SymmetricSecurityKey(bytes)` |
| `http.oauth2ResourceServer().jwt()` | `AddAuthentication().AddJwtBearer(options => ...)` |
| `clockSkewSeconds` in `NimbusJwtDecoder` | `ClockSkew = TimeSpan.Zero` in `TokenValidationParameters` |
