# 7.1 Il modello di sicurezza in ASP.NET Core: panoramica

> **Analogia Java**: equivalente di Spring Security — architettura a confronto

---

## Introduzione

Spring Security è un framework monolitico: filtri, provider, context holder — tutto accoppiato. ASP.NET Core ha un modello più modulare: **middleware pipeline** per l'autenticazione, **policy system** per l'autorizzazione. Il risultato è simile ma l'architettura è diversa — vale la pena capirla prima di scrivere codice.

---

## Come funziona Spring Security

```
Request
   ↓
[FilterChain]
   ├── SecurityContextPersistenceFilter
   ├── UsernamePasswordAuthenticationFilter
   ├── BasicAuthenticationFilter
   ├── BearerTokenAuthenticationFilter  ← legge JWT, popola SecurityContext
   └── ExceptionTranslationFilter
   ↓
[SecurityContext] → Authentication → Principal → Authorities
   ↓
[@PreAuthorize("hasRole('ADMIN')")] → accesso consentito o 403
```

```java
// Spring Security — configurazione
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated())
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))
            .build();
    }
}
```

## Come funziona ASP.NET Core

```
Request
   ↓
[Middleware Pipeline]
   ├── UseRouting
   ├── UseAuthentication   ← legge il token, popola HttpContext.User
   ├── UseAuthorization    ← valuta le policy, restituisce 401/403
   └── UseEndpoints / MapControllers
   ↓
[HttpContext.User] → ClaimsPrincipal → Claims
   ↓
[[Authorize(Policy = "AdminOnly")]] → accesso consentito o 403
```

```csharp
// ASP.NET Core — configurazione in Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer           = true,
            ValidateAudience         = true,
            ValidateLifetime         = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer              = builder.Configuration["Jwt:Issuer"],
            ValidAudience            = builder.Configuration["Jwt:Audience"],
            IssuerSigningKey         = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(builder.Configuration["Jwt:Key"]!))
        };
    });

builder.Services.AddAuthorization();

// ...

app.UseAuthentication();  // prima
app.UseAuthorization();   // dopo — l'ordine è obbligatorio
```

## I concetti chiave

### ClaimsPrincipal e Claims

In Spring Security hai `Authentication` con `Principal` e `GrantedAuthority`. In ASP.NET Core hai `ClaimsPrincipal` con una collezione di `Claim` — coppie chiave/valore che rappresentano tutto ciò che sai sull'utente.

```csharp
// Leggere i claims dall'HttpContext
var userId   = User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
var email    = User.FindFirst(ClaimTypes.Email)?.Value;
var role     = User.FindFirst(ClaimTypes.Role)?.Value;
var isAdmin  = User.IsInRole("Admin");
```

```java
// Equivalente Spring Security
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();
boolean isAdmin = auth.getAuthorities().stream()
    .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));
```

### Authentication Schemes

ASP.NET Core supporta più schemi di autenticazione simultaneamente — JWT Bearer, Cookie, ApiKey. Puoi usarli insieme e specificare quale usare per ogni endpoint.

```csharp
builder.Services.AddAuthentication()
    .AddJwtBearer("Bearer", options => { ... })
    .AddCookie("Cookie", options => { ... });
```

### Policy-based authorization

Invece di `@PreAuthorize("hasRole('ADMIN')")`, ASP.NET Core usa le **policy** — regole nominali che possono contenere logica arbitraria:

```csharp
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireRole("Admin"));

    options.AddPolicy("SeniorEmployee", policy =>
        policy.RequireClaim("department", "engineering")
              .RequireClaim("seniority", "senior", "lead", "staff"));
});
```

## L'ordine del middleware è obbligatorio

```csharp
// ✓ Ordine corretto
app.UseRouting();
app.UseAuthentication();  // popola HttpContext.User
app.UseAuthorization();   // usa HttpContext.User per valutare le policy
app.MapControllers();

// ✗ Ordine sbagliato — UseAuthorization prima di UseAuthentication
// → HttpContext.User è sempre anonimo → tutti gli endpoint ritornano 401
app.UseAuthorization();
app.UseAuthentication();
```

In Spring Security l'ordine è gestito internamente dalla `FilterChain` — qui è esplicito in `Program.cs`.

---

## Spiegazione

**`UseAuthentication` vs `UseAuthorization`** — sono due middleware distinti. `UseAuthentication` legge il token dalla request e popola `HttpContext.User`. `UseAuthorization` controlla se l'utente autenticato ha i permessi richiesti dall'endpoint. Separati per design — puoi autenticare senza autorizzare (endpoint pubblici con dati contestuali sull'utente).

**`ClaimsPrincipal` vs `Authentication`** — in Spring Security `SecurityContextHolder` è un thread-local. In ASP.NET Core `HttpContext.User` è legato alla request — più pulito in un modello async dove i thread cambiano durante l'elaborazione.

**Nessun `@EnableWebSecurity`** — ASP.NET Core non ha annotazioni di abilitazione. La sicurezza è parte della pipeline middleware — la aggiungi con `AddAuthentication` + `AddAuthorization` nel DI container e con `UseAuthentication` + `UseAuthorization` nella pipeline.

## Confronto con Java

| Spring Security | ASP.NET Core |
|---|---|
| `FilterChain` — filtri in sequenza | Middleware pipeline — stesso concetto |
| `SecurityContextHolder` (thread-local) | `HttpContext.User` (request-scoped) |
| `Authentication` → `Principal` + `Authorities` | `ClaimsPrincipal` → `Claims` |
| `@EnableWebSecurity` + `SecurityFilterChain` | `AddAuthentication` + `UseAuthentication` in `Program.cs` |
| `@PreAuthorize("hasRole('X')")` | `[Authorize(Policy = "X")]` |
| `GrantedAuthority` | `Claim` con `ClaimTypes.Role` |
| `UserDetailsService` | `IClaimsTransformation` / custom middleware |
| `oauth2ResourceServer().jwt()` | `AddJwtBearer(options => ...)` |
