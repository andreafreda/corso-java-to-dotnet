# 7.2 Autenticazione vs Autorizzazione: concetti chiave

> **Analogia Java**: `@PreAuthorize`, Roles e Policy in .NET

---

## Introduzione

Autenticazione e autorizzazione sono concetti distinti che in Spring Security tendono a mescolarsi. ASP.NET Core li separa esplicitamente — middleware diversi, API diverse. Questo paragrafo chiarisce la differenza e mostra come si implementano i meccanismi di autorizzazione più comuni: ruoli, claims e policy.

---

## Autenticazione — chi sei

L'autenticazione risponde alla domanda "chi è questo utente?". In ASP.NET Core avviene in `UseAuthentication`: il middleware legge il token Bearer, lo valida e popola `HttpContext.User` con i claims dell'utente.

```csharp
// Dopo UseAuthentication, HttpContext.User è popolato
var isAuthenticated = User.Identity?.IsAuthenticated;  // true/false
var userId          = User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
var email           = User.FindFirst(ClaimTypes.Email)?.Value;
```

Se il token non è presente o non è valido, `HttpContext.User` è un `ClaimsPrincipal` anonimo — `IsAuthenticated` è `false`.

---

## Autorizzazione — cosa puoi fare

L'autorizzazione risponde alla domanda "questo utente può fare questa cosa?". In ASP.NET Core avviene in `UseAuthorization` e tramite l'attributo `[Authorize]` sugli endpoint.

### `[Authorize]` base — richiede autenticazione

```csharp
[ApiController]
[Route("orders")]
[Authorize]  // richiede utente autenticato — equivalente di @PreAuthorize("isAuthenticated()")
public class OrderController : ControllerBase
{
    [HttpGet]
    public async Task<IActionResult> GetAll() { ... }

    [AllowAnonymous]  // override — questo endpoint è pubblico
    [HttpGet("public-stats")]
    public IActionResult GetStats() { ... }
}
```

### Autorizzazione per ruolo

```csharp
// Attributo sul controller o sul metodo
[Authorize(Roles = "Admin")]
[HttpDelete("{id}")]
public async Task<IActionResult> Delete(int id) { ... }

// Ruoli multipli — OR logico
[Authorize(Roles = "Admin,Manager")]
[HttpPut("{id}")]
public async Task<IActionResult> Update(int id, ...) { ... }
```

```java
// Equivalente Spring Security
@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable Long id) { ... }

@PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")
@PutMapping("/{id}")
public ResponseEntity<Order> update(@PathVariable Long id, ...) { ... }
```

### Autorizzazione per policy

Le policy sono il modo idiomatico in ASP.NET Core — più flessibili dei ruoli, testabili, riusabili:

```csharp
// Definizione in Program.cs
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireRole("Admin"));

    options.AddPolicy("CanManageOrders", policy =>
        policy.RequireAssertion(ctx =>
            ctx.User.IsInRole("Admin") ||
            ctx.User.IsInRole("Manager")));

    options.AddPolicy("VerifiedUser", policy =>
        policy.RequireClaim("email_verified", "true"));
});
```

```csharp
// Uso sull'endpoint
[Authorize(Policy = "CanManageOrders")]
[HttpPut("{id}")]
public async Task<IActionResult> Update(int id, ...) { ... }
```

## Authorization handler — logica custom

Per autorizzazione complessa (es. "l'utente può modificare solo i propri ordini") usi un `AuthorizationHandler`:

```csharp
// Core/Authorization/OrderOwnerRequirement.cs
public class OrderOwnerRequirement : IAuthorizationRequirement { }

public class OrderOwnerHandler : AuthorizationHandler<OrderOwnerRequirement, Order>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        OrderOwnerRequirement requirement,
        Order order)
    {
        var userId = context.User.FindFirst(ClaimTypes.NameIdentifier)?.Value;

        if (userId == order.OwnerId.ToString())
            context.Succeed(requirement);

        return Task.CompletedTask;
    }
}
```

```csharp
// Registrazione
builder.Services.AddSingleton<IAuthorizationHandler, OrderOwnerHandler>();
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("OrderOwner", policy =>
        policy.Requirements.Add(new OrderOwnerRequirement()));
});
```

```csharp
// Uso nel controller
[HttpPut("{id}")]
public async Task<IActionResult> Update(int id, UpdateOrderRequest request)
{
    var order = await _orderService.GetByIdAsync(id);
    if (order is null) return NotFound();

    var authResult = await _authorizationService.AuthorizeAsync(User, order, "OrderOwner");
    if (!authResult.Succeeded) return Forbid();

    // procedi con l'aggiornamento
}
```

## Leggere i claims nel controller

```csharp
// Nel controller — User è ClaimsPrincipal
public class OrderController : ControllerBase
{
    [HttpPost]
    [Authorize]
    public async Task<IActionResult> Create(CreateOrderRequest request)
    {
        // Leggi l'ID utente dal token JWT
        var userId = User.FindFirst(ClaimTypes.NameIdentifier)?.Value
            ?? throw new UnauthorizedAccessException();

        // Passa userId al service per associare l'ordine all'utente
        var order = await _orderService.CreateAsync(request, userId);
        return CreatedAtAction(nameof(GetById), new { id = order.Id }, order);
    }
}
```

```java
// Equivalente Spring Security
@PostMapping
@PreAuthorize("isAuthenticated()")
public ResponseEntity<Order> create(@RequestBody CreateOrderRequest request,
                                     @AuthenticationPrincipal Jwt jwt) {
    String userId = jwt.getSubject();
    Order order = orderService.create(request, userId);
    return ResponseEntity.created(...).body(order);
}
```

## Spiegazione

**Roles vs Policy** — i ruoli sono una scorciatoia per `RequireClaim(ClaimTypes.Role, "Admin")`. Le policy sono più flessibili: possono combinare claim, ruoli, logica custom. Il consiglio è usare policy anche per i casi semplici — sono testabili e riusabili.

**`[AllowAnonymous]` ha precedenza su `[Authorize]`** — se un controller ha `[Authorize]` a livello di classe, un metodo con `[AllowAnonymous]` è sempre accessibile. Utile per endpoint pubblici in un controller prevalentemente protetto.

**`IAuthorizationService` per autorizzazione imperativa** — `[Authorize(Policy = "X")]` è dichiarativa e valutata prima dell'esecuzione del metodo. Per autorizzazione che dipende dai dati (es. "sei il proprietario di questo ordine") usa `IAuthorizationService.AuthorizeAsync` nel corpo del metodo, dopo aver caricato la risorsa.

## Confronto con Java

| Spring Security | ASP.NET Core |
|---|---|
| `@PreAuthorize("isAuthenticated()")` | `[Authorize]` |
| `@PreAuthorize("hasRole('ADMIN')")` | `[Authorize(Roles = "Admin")]` |
| `@PreAuthorize("hasAnyRole('A','B')")` | `[Authorize(Roles = "A,B")]` |
| `@PreAuthorize` con SpEL custom | Policy con `RequireAssertion` |
| `PermissionEvaluator` per logica custom | `AuthorizationHandler<TRequirement, TResource>` |
| `@AuthenticationPrincipal Jwt jwt` | `User.FindFirst(ClaimTypes.NameIdentifier)` |
| `@PermitAll` / `permitAll()` | `[AllowAnonymous]` |
| `SecurityConfig.authorizeHttpRequests()` | `AddAuthorization(options => options.AddPolicy(...))` |
