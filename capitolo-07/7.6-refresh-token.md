# 7.6 Refresh token e gestione della sessione

> **Analogia Java**: pattern indipendente dal linguaggio — implementabile con Spring Security + JJWT

---

## Introduzione

Il JWT ha una scadenza breve — tipicamente 15-60 minuti. Alla scadenza l'utente dovrebbe fare login di nuovo, ma questo è inaccettabile per una UI. Il **refresh token** risolve il problema: è una stringa opaca a lunga durata che permette di ottenere un nuovo access token senza reinserire le credenziali. Il meccanismo è già parzialmente implementato in 7.4 — questo paragrafo lo completa e lo consolida.

---

## Il flusso completo

```
1. Login
   Client → POST /auth/login { email, password }
   Server → { accessToken (60min), refreshToken (30gg) }

2. Chiamata API
   Client → GET /orders
             Authorization: Bearer <accessToken>
   Server → 200 OK

3. Access token scaduto
   Client → GET /orders
             Authorization: Bearer <accessToken scaduto>
   Server → 401 Unauthorized

4. Refresh
   Client → POST /auth/refresh { refreshToken }
   Server → { nuovo accessToken (60min), nuovo refreshToken (30gg) }
            (il vecchio refreshToken è revocato)

5. Logout
   Client → POST /auth/logout { refreshToken }
   Server → revoca refreshToken → 204 No Content
```

---

## Come si faceva in Java

```java
// Spring Security — gestione refresh token
@PostMapping("/auth/refresh")
public ResponseEntity<TokenResponse> refresh(@RequestBody RefreshRequest request) {
    RefreshToken stored = refreshTokenRepository
        .findByToken(request.refreshToken())
        .orElseThrow(() -> new RefreshTokenException("Token non trovato"));

    if (stored.getExpiryDate().isBefore(Instant.now()))
        throw new RefreshTokenException("Token scaduto");

    User user = stored.getUser();
    String newAccessToken = jwtService.generateToken(user);

    // Refresh token rotation — revoca il vecchio, genera il nuovo
    refreshTokenRepository.delete(stored);
    RefreshToken newRefresh = refreshTokenService.createToken(user);

    return ResponseEntity.ok(new TokenResponse(newAccessToken, newRefresh.getToken()));
}
```

## Implementazione completa del refresh token

In 7.4 abbiamo usato un `Dictionary` statico in-memory. Qui lo rendiamo più robusto con scadenza e revoca esplicita — ancora in-memory per il corso, ma con la struttura che rispecchia un'implementazione su DB.

### Modello refresh token

```csharp
// Core/Entities/RefreshToken.cs
namespace OrderService.Core.Entities;

public class RefreshToken
{
    public string   Token     { get; set; } = string.Empty;
    public string   UserId    { get; set; } = string.Empty;
    public DateTime ExpiresAt { get; set; }
    public bool     Revoked   { get; set; }

    public bool IsValid => !Revoked && ExpiresAt > DateTime.UtcNow;
}
```

### `InMemoryRefreshTokenStore` — implementazione in Infrastructure

```csharp
// Infrastructure/Auth/InMemoryRefreshTokenStore.cs
using System.Collections.Concurrent;
using OrderService.Core.Entities;
using OrderService.Core.Interfaces;

namespace OrderService.Infrastructure.Auth;

// In produzione: sostituisci con un'implementazione che legge/scrive su DB
public class InMemoryRefreshTokenStore : IRefreshTokenStore
{
    // ConcurrentDictionary — thread-safe per richieste concorrenti
    private readonly ConcurrentDictionary<string, RefreshToken> _store = new();

    public void Store(RefreshToken token)
        => _store[token.Token] = token;

    public RefreshToken? Get(string token)
        => _store.GetValueOrDefault(token);

    public void Revoke(string token)
    {
        if (_store.TryGetValue(token, out var existing))
            existing.Revoked = true;
    }
}
```

### `AuthService` aggiornato

```csharp
// Core/Services/AuthService.cs
using System.Security.Cryptography;
using System.Text;
using OrderService.Core.Entities;
using OrderService.Core.Interfaces;

namespace OrderService.Core.Services;

public class AuthService
{
    private readonly TokenService       _tokenService;
    private readonly IRefreshTokenStore _refreshStore;  // interfaccia — introdotta in 7.4

    private static readonly List<AppUser> Users =
    [
        new() { Id = "1", Email = "admin@example.com",
                PasswordHash = HashPassword("admin123"), Role = "Admin" },
        new() { Id = "2", Email = "user@example.com",
                PasswordHash = HashPassword("user123"),  Role = "User"  }
    ];

    public AuthService(TokenService tokenService, IRefreshTokenStore refreshStore)
    {
        _tokenService = tokenService;
        _refreshStore = refreshStore;
    }

    public TokenResponse? Login(string email, string password)
    {
        var user = Users.FirstOrDefault(u =>
            u.Email == email && u.PasswordHash == HashPassword(password));

        if (user is null) return null;

        return IssueTokens(user);
    }

    public TokenResponse? Refresh(string refreshToken)
    {
        var stored = _refreshStore.Get(refreshToken);
        if (stored is null || !stored.IsValid) return null;

        var user = Users.FirstOrDefault(u => u.Id == stored.UserId);
        if (user is null) return null;

        // Refresh token rotation — il vecchio non è più valido
        _refreshStore.Revoke(refreshToken);

        return IssueTokens(user);
    }

    public bool Logout(string refreshToken)
    {
        var stored = _refreshStore.Get(refreshToken);
        if (stored is null) return false;
        _refreshStore.Revoke(refreshToken);
        return true;
    }

    private TokenResponse IssueTokens(AppUser user)
    {
        var accessToken  = _tokenService.GenerateToken(user.Id, user.Email, user.Role);
        var refreshToken = new RefreshToken
        {
            Token     = _tokenService.GenerateRefreshToken(),
            UserId    = user.Id,
            ExpiresAt = DateTime.UtcNow.AddDays(30),
            Revoked   = false
        };

        _refreshStore.Store(refreshToken);
        return new TokenResponse(accessToken, refreshToken.Token, ExpiresIn: 3600);
    }

    private static string HashPassword(string password)
    {
        var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(password));
        return Convert.ToHexString(bytes);
    }
}
```

### Endpoint logout in `AuthController`

```csharp
// Api/Controllers/AuthController.cs — aggiungi
[HttpPost("logout")]
public IActionResult Logout([FromBody] string refreshToken)
{
    var revoked = _authService.Logout(refreshToken);
    if (!revoked) return BadRequest(new { message = "Refresh token non trovato" });
    return NoContent();
}
```

## Refresh token rotation

Ogni volta che usi un refresh token per ottenerne uno nuovo, il vecchio viene revocato. Se un attaccante ruba il refresh token e lo usa, il token originale nell'utente legittimo smette di funzionare — al prossimo tentativo di refresh l'utente riceve 401 e viene forzato al login.

```
Utente legittimo:         refreshToken_A → [usa] → revocato → refreshToken_B
Attaccante (stesso token): refreshToken_A → [usa dopo] → già revocato → 401
```

---

## Spiegazione

**Refresh token non è un JWT** — è una stringa casuale opaca. Il suo significato è nel server (database), non nel token stesso. Questo è intenzionale: permette la revoca immediata. Un JWT non è revocabile prima della scadenza senza una blacklist.

**`InMemoryRefreshTokenStore` come `Singleton`** — il dictionary deve sopravvivere tra le richieste. È in Infrastructure perché è un'implementazione — la sua esistenza è un dettaglio. Il Core conosce solo `IRefreshTokenStore`. In produzione sostituisci con una classe che legge/scrive su DB: il Core non cambia, cambia solo la registrazione DI in Infrastructure.

**`ConcurrentDictionary` invece di `Dictionary`** — un web server gestisce richieste concorrenti su thread separati. `Dictionary` non è thread-safe: due richieste di refresh simultanee possono causare `InvalidOperationException`. `ConcurrentDictionary` gestisce la concorrenza internamente senza lock espliciti.

**Scadenza del refresh token** — 30 giorni è un valore comune. In app con alta sicurezza (banking) si usa 1-7 giorni con re-autenticazione periodica. In app consumer si arriva a 90-180 giorni con "remember me".

**Access token breve** — 60 minuti è un buon compromesso. Più breve (15min) aumenta la sicurezza ma aumenta il numero di refresh. Il refresh dovrebbe avvenire in background, trasparente all'utente — il client lo rileva dal 401 e fa refresh automaticamente.
