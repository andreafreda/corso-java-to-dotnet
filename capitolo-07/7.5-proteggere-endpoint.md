# 7.5 Proteggere gli endpoint: `[Authorize]` e Policy

> **Analogia Java**: equivalente di `@PreAuthorize` / `SecurityConfig`

---

## Introduzione

Abbiamo l'AuthServer, il token JWT, la configurazione di validazione. Ora applichiamo la protezione agli endpoint di `OrderService` — il passo concreto che rende il microservizio sicuro. Questo paragrafo mostra come usare `[Authorize]`, le policy e come gestire i casi edge: endpoint misti, autorizzazione per risorsa, risposta corretta su 401 vs 403.

## Come si faceva in Java

```java
// Spring Security — protezione globale
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/auth/**").permitAll()
            .requestMatchers(HttpMethod.GET, "/orders/**").hasAnyRole("USER", "ADMIN")
            .requestMatchers(HttpMethod.DELETE, "/orders/**").hasRole("ADMIN")
            .anyRequest().authenticated())
        .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))
        .build();
}
```

```java
// Oppure a livello di metodo
```

## Proteggere `OrderController`

> 📝 **Code Step 16** — Applica `[Authorize]` a `OrderController` e configura le policy

### Policy in `Program.cs`

```csharp
// Api/Program.cs — aggiungi le policy
builder.Services.AddAuthorization(options =>
{
    // ReadOrders non serve: [Authorize] sul controller è equivalente a RequireAuthenticatedUser()
    options.AddPolicy("WriteOrders",  policy => policy.RequireRole("Admin", "Manager"));
    options.AddPolicy("DeleteOrders", policy => policy.RequireRole("Admin"));
});
```

### `OrderController` protetto

```csharp
// Api/Controllers/OrderController.cs
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Mvc;
using OrderService.Core.Interfaces.Services;
using OrderService.Api.Models;

namespace OrderService.Api.Controllers;

[ApiController]
[Route("orders")]
[Authorize]  // tutti gli endpoint richiedono autenticazione per default
public class OrderController : ControllerBase
{
    private readonly IOrderService _orderService;

    public OrderController(IOrderService orderService)
    {
        _orderService = orderService;
    }

    [HttpGet]
    public async Task<IActionResult> GetAll(CancellationToken ct)
    {
        var orders = await _orderService.GetAllAsync(ct);
        return Ok(orders);
    }

    [HttpGet("{id:int}")]
    public async Task<IActionResult> GetById(int id, CancellationToken ct)
    {
        var order = await _orderService.GetByIdAsync(id, ct);
        return order is null ? NotFound() : Ok(order);
    }

    [HttpPost]
    [Authorize(Policy = "WriteOrders")]
    public async Task<IActionResult> Create(CreateOrderRequest request, CancellationToken ct)
    {
        var order = await _orderService.CreateAsync(request, ct);
        return CreatedAtAction(nameof(GetById), new { id = order.Id }, order);
    }

    [HttpDelete("{id:int}")]
    [Authorize(Policy = "DeleteOrders")]
    public async Task<IActionResult> Delete(int id, CancellationToken ct)
    {
        await _orderService.DeleteAsync(id, ct);
        return NoContent();
    }
}
```

## 401 vs 403 — la differenza

```
401 Unauthorized → non sei autenticato (token mancante o non valido)
403 Forbidden    → sei autenticato ma non hai i permessi (ruolo insufficiente)
```

ASP.NET Core gestisce questa distinzione automaticamente:
- Token mancante o invalido → `UseAuthentication` → `401`
- Token valido ma policy non soddisfatta → `UseAuthorization` → `403`

Per restituire un body significativo invece di una risposta vuota, personalizza i challenge/forbid handler:

```csharp
// Program.cs — personalizza le risposte 401 e 403
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters { /* ... */ };

        options.Events = new JwtBearerEvents
        {
            OnChallenge = async context =>
            {
                context.HandleResponse();
                context.Response.StatusCode  = 401;
                context.Response.ContentType = "application/problem+json";
                await context.Response.WriteAsJsonAsync(new
                {
                    type   = "https://tools.ietf.org/html/rfc7235#section-3.1",
                    title  = "Unauthorized",
                    status = 401,
                    detail = "Token mancante o non valido"
                });
            },
            OnForbidden = async context =>
            {
                context.Response.StatusCode  = 403;
                context.Response.ContentType = "application/problem+json";
                await context.Response.WriteAsJsonAsync(new
                {
                    type   = "https://tools.ietf.org/html/rfc7231#section-6.5.3",
                    title  = "Forbidden",
                    status = 403,
                    detail = "Permessi insufficienti per questa operazione"
                });
            }
        };
    });
```

## Endpoint misti: pubblico e protetto nello stesso controller

```csharp
[ApiController]
[Route("orders")]
[Authorize]  // default: autenticazione richiesta
public class OrderController : ControllerBase
{
    [HttpGet("stats")]
    [AllowAnonymous]  // questo endpoint è pubblico — override di [Authorize]
    public async Task<IActionResult> GetStats(CancellationToken ct)
    {
        var count = await _orderService.CountAsync(ct);
        return Ok(new { totalOrders = count });
    }

    [HttpGet]
    public async Task<IActionResult> GetAll(CancellationToken ct) { ... }  // protetto
}
```

## Leggere l'utente corrente negli endpoint protetti

```csharp
[HttpPost]
[Authorize(Policy = "WriteOrders")]
public async Task<IActionResult> Create(CreateOrderRequest request, CancellationToken ct)
{
    // I claims sono garantiti presenti — [Authorize] ha già validato il token
    var userId = User.FindFirst(JwtRegisteredClaimNames.Sub)!.Value;
    // userId disponibile qui se serve per audit log o ownership — non necessario per CreateAsync

    var order = await _orderService.CreateAsync(request, ct);
    return CreatedAtAction(nameof(GetById), new { id = order.Id }, order);
}
```

## Spiegazione

**`[Authorize]` a livello di controller vs metodo** — mettere `[Authorize]` sul controller protegge tutti i metodi per default. L'override con `[AllowAnonymous]` o `[Authorize(Policy = "X")]` su un singolo metodo prende precedenza. È il pattern più pulito: protezione per default, eccezioni esplicite.

**Policy vs Roles inline** — `[Authorize(Roles = "Admin")]` è una scorciatoia. Preferisci `[Authorize(Policy = "DeleteOrders")]` — il nome della policy è semantico, la logica interna è modificabile senza toccare i controller.

**`OnChallenge` e `OnForbidden`** — per default ASP.NET Core restituisce 401/403 senza body. Per un'API REST è meglio restituire un Problem Details (RFC 7807) — consistente con la gestione errori del capitolo 4.

## Confronto con Java

| Spring Security | ASP.NET Core |
|---|---|
| `authorizeHttpRequests().anyRequest().authenticated()` | `[Authorize]` sul controller |
| `@PreAuthorize("hasRole('ADMIN')")` | `[Authorize(Policy = "DeleteOrders")]` |
| `@PermitAll` / `.permitAll()` | `[AllowAnonymous]` |
| `AccessDeniedException` → 403 | Policy non soddisfatta → 403 automatico |
| `AuthenticationException` → 401 | Token invalido → 401 automatico |
| `AuthenticationEntryPoint` per 401 custom | `JwtBearerEvents.OnChallenge` |
| `AccessDeniedHandler` per 403 custom | `JwtBearerEvents.OnForbidden` |
| `@AuthenticationPrincipal Jwt jwt` | `User.FindFirst(JwtRegisteredClaimNames.Sub)` |
