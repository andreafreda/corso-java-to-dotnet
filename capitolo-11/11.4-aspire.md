# 11.4 .NET Aspire avanzato

> **Analogia Java**: Spring Boot DevTools + Micrometer + Spring Cloud Config messi insieme, ma integrato nativamente

---

## Introduzione

Il capitolo 10.3 ha introdotto .NET Aspire come alternativa a docker-compose per lo sviluppo locale. Questo paragrafo approfondisce le funzionalità avanzate: come integrare più microservizi in un unico AppHost, come usare la dashboard per il debug distribuito, come Aspire gestisce la configurazione tra servizi, e come generare i manifest per il deploy in produzione.

---

## Ricapitolo: cosa fa Aspire

```
Senza Aspire:
  docker-compose.yml  ←  YAML separato dal codice
  appsettings.json    ←  connection string hardcoded per ogni env
  Jaeger + Prometheus ←  setup manuale di osservabilità

Con Aspire:
  AppHost/Program.cs  ←  tutto in C#, stesso linguaggio dell'app
  connection string   ←  iniettate automaticamente tra i servizi
  Dashboard integrata ←  traces, logs, metrics senza config aggiuntiva
```

## Stack multi-microservizio in AppHost

Scenario realistico: `OrderService` chiama `InventoryService`, entrambi usano Azure SQL, condividono Redis, pubblicano su Service Bus.

```csharp
// OrderService.AppHost/Program.cs
var builder = DistributedApplication.CreateBuilder(args);

// ── Infrastruttura ────────────────────────────────────────────────────
var sql = builder.AddSqlServer("azuresql")
    .WithDataVolume()
    .WithLifetime(ContainerLifetime.Persistent);  // sopravvive al restart dell'AppHost

var orderDb     = sql.AddDatabase("OrderDb");
var inventoryDb = sql.AddDatabase("InventoryDb");

var redis = builder.AddRedis("redis")
    .WithRedisInsight();                          // apre RedisInsight UI su :8001

var serviceBus = builder.AddAzureServiceBus("servicebus")
    .RunAsEmulator();

// ── Microservizi ──────────────────────────────────────────────────────
var inventoryService = builder.AddProject<Projects.InventoryService_Api>("inventoryservice")
    .WithReference(inventoryDb)
    .WithReference(redis)
    .WithHttpEndpoint(port: 8081);

var orderService = builder.AddProject<Projects.OrderService_Api>("orderservice")
    .WithReference(orderDb)
    .WithReference(redis)
    .WithReference(serviceBus)
    .WithReference(inventoryService)              // inietta l'URL di InventoryService
    .WithHttpEndpoint(port: 8080);

builder.Build().Run();
```

Aspire inietta automaticamente in `orderservice`:
- `ConnectionStrings__OrderDb` → connection string verso il database
- `ConnectionStrings__Redis` → connection string Redis
- `ConnectionStrings__servicebus` → connection string Service Bus emulato
- `services__inventoryservice__http__0` → URL di InventoryService

## Leggere le connection string iniettate da Aspire

```csharp
// OrderService.Api/Program.cs
builder.AddServiceDefaults();  // OpenTelemetry + health checks + service discovery

// Aspire inietta la connection string — stessa chiave usata in docker-compose
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("OrderDb")));

builder.Services.AddStackExchangeRedisCache(options =>
    options.Configuration = builder.Configuration.GetConnectionString("Redis"));
```

```csharp
// Service discovery verso InventoryService — Aspire risolve l'URL
builder.Services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    client.BaseAddress = new Uri("http://inventoryservice");  // nome registrato nell'AppHost
});
```

## ServiceDefaults — configurazione condivisa avanzata

```csharp
// OrderService.ServiceDefaults/Extensions.cs
public static class Extensions
{
    public static IHostApplicationBuilder AddServiceDefaults(
        this IHostApplicationBuilder builder)
    {
        // OpenTelemetry — traces, metrics, logs verso la dashboard Aspire
        builder.ConfigureOpenTelemetry();

        // Health checks standard
        builder.AddDefaultHealthChecks();

        // Service discovery integrato con HttpClient
        builder.Services.AddServiceDiscovery();
        builder.Services.ConfigureHttpClientDefaults(http =>
        {
            http.AddStandardResilienceHandler();  // Polly v8 retry + circuit breaker
            http.AddServiceDiscovery();
        });

        return builder;
    }

    public static WebApplication MapDefaultEndpoints(this WebApplication app)
    {
        // Health endpoints standard — usati dai Kubernetes probes
        app.MapHealthChecks("/health/live",  new HealthCheckOptions
            { Predicate = r => r.Tags.Contains("live") });
        app.MapHealthChecks("/health/ready", new HealthCheckOptions
            { Predicate = r => r.Tags.Contains("ready") });

        return app;
    }
}
```

├──────────────────────────────────────────────────────────────────┤
│  orderservice       Container   http://localhost:8080   Running  │
│  inventoryservice   Container   http://localhost:8081   Running  │
│  azuresql           Container   localhost:1433           Running  │
│  redis              Container   localhost:6379           Running  │
│  servicebus         Container   localhost:5672           Running  │
├──────────────────────────────────────────────────────────────────┤
│  Traces                                                          │
│  POST /orders  orderservice → inventoryservice  145ms   ✓        │
│    ├── HTTP POST /orders                        12ms             │
│    ├── DB: INSERT INTO Orders                   45ms             │
│    ├── HTTP GET /inventory/check                78ms             │
│    └── ServiceBus: publish order-created        10ms             │

## Publish — genera manifest per produzione

Aspire può generare manifest Kubernetes o Azure Container Apps dalla stessa definizione C#:

```bash
# Genera manifest per Azure Container Apps
dotnet aspire publish --output-path ./deploy/aca

# Genera manifest Kubernetes
dotnet aspire publish --output-path ./deploy/k8s --format kubernetes
```

```
deploy/k8s/
├── orderservice-deployment.yaml
├── inventoryservice-deployment.yaml
├── redis-deployment.yaml
└── aspire-manifest.json
```

> Il publish AOT è in preview — verifica la compatibilità con le versioni correnti di Aspire.

## Aspire vs docker-compose vs Kubernetes

| | docker-compose | .NET Aspire | Kubernetes |
|---|---|---|---|
| Linguaggio config | YAML | C# | YAML |
| Osservabilità integrata | No | Sì | No (richiede stack separato) |
| Service discovery | Manuale (nomi container) | Automatico | Automatico (CoreDNS) |
| Hot reload | No | Sì | No |
| Uso consigliato | Dev locale multi-vendor | Dev locale team .NET | Produzione |
| Connection string | Hardcoded | Iniettate automaticamente | ConfigMap + Secret |
| Portabilità | Alta | Solo .NET | Alta |

## Spiegazione

**`WithLifetime(ContainerLifetime.Persistent)`** — per default Aspire ferma i container quando l'AppHost si chiude. Con `Persistent`, il container SQL Server sopravvive — i dati rimangono tra i restart. Utile per non perdere i dati del DB di sviluppo a ogni modifica del codice.

**`WithRedisInsight()`** — aggiunge RedisInsight come container aggiuntivo. È un'interfaccia grafica per esplorare le chiavi Redis in tempo reale — utile per debuggare la cache durante lo sviluppo.

**Aspire non va in produzione** — l'`AppHost` è esclusivamente per sviluppo locale e generazione di manifest. Non deployare l'AppHost su un server. In produzione si usano Kubernetes (cap. 10.4) o Azure Container Apps, eventualmente con i manifest generati da `dotnet aspire publish`.

**OpenTelemetry senza configurazione** — `AddServiceDefaults()` configura automaticamente il collector OpenTelemetry che manda i dati alla dashboard Aspire. Quando il codice va in produzione, basta puntare il collector verso Jaeger, Grafana Tempo, o Azure Monitor cambiando la configurazione — il codice di instrumentazione non cambia.

## Confronto con Java

| Spring Boot + Java | .NET Aspire |
|---|---|
| `docker-compose.yml` per dev locale | `AppHost/Program.cs` in C# |
| Spring Boot DevTools (hot reload) | Hot reload integrato in Aspire |
| Micrometer + Prometheus + Grafana | Dashboard integrata via OpenTelemetry |
| Spring Cloud Config (config server) | Connection string iniettate automaticamente |
| Nessun equivalente diretto | `MapSubscribeHandler` / service discovery automatico |
| Testcontainers per test | Testcontainers per test (stesso approccio) |
