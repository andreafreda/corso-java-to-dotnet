# 11.3 Dapr: runtime per microservizi portatile

> **Analogia Java**: equivalente di Spring Cloud (Service Discovery, Circuit Breaker, Pub/Sub) ma come sidecar indipendente dal linguaggio

---

## Introduzione

Nei capitoli precedenti abbiamo implementato a mano: service discovery con `IHttpClientFactory`, pub/sub con Azure Service Bus SDK, state management con Redis. Ogni pezzo usa l'SDK del vendor — cambiare da Azure Service Bus a RabbitMQ richiede riscrivere il codice. **Dapr** (Distributed Application Runtime) sposta tutta questa complessità in un sidecar: il tuo microservizio chiama un'API HTTP locale (`http://localhost:3500`), Dapr traduce verso il backend configurato — Service Bus, RabbitMQ, Redis, Cosmos DB — senza cambiare una riga di codice applicativo.

---

## Come funziona il sidecar

```
┌─────────────────────────────────────────────────┐
│  Pod Kubernetes                                   │
│                                                   │
│  ┌──────────────────┐    ┌───────────────────┐   │
│  │  OrderService    │◄──►│  Dapr sidecar     │   │
│  │  :8080           │    │  :3500 (HTTP)     │   │
│  │                  │    │  :50001 (gRPC)    │   │
│  └──────────────────┘    └────────┬──────────┘   │
│                                   │               │
└───────────────────────────────────┼───────────────┘
                                    │
              ┌─────────────────────┼──────────────┐
              ▼                     ▼               ▼
        Azure Service Bus        Redis           Cosmos DB
        (o RabbitMQ)         (o Memcached)    (o PostgreSQL)
```

Il microservizio non sa quale backend usa — lo sa solo il componente Dapr configurato nell'infrastruttura.

## Building block Dapr

| Building Block | Cosa fa | Equivalente implementato manualmente |
|---|---|---|
| **Service Invocation** | Chiama un altro microservizio via nome | `IHttpClientFactory` + DNS Kubernetes |
| **Pub/Sub** | Pubblica/consuma messaggi | Azure Service Bus SDK |
| **State Management** | Stato chiave-valore | `IDistributedCache` + Redis |
| **Secret Store** | Recupera segreti | Azure Key Vault + `DefaultAzureCredential` |
| **Bindings** | Input/Output verso sistemi esterni | Trigger custom |
| **Actors** | Oggetti con stato e concorrenza | Non trattato nel corso |

## Componenti Dapr — configurazione locale

```yaml
# dapr/components/pubsub.yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: order-pubsub
spec:
  type: pubsub.redis           # locale: Redis — produzione: Azure Service Bus
  version: v1
  metadata:
    - name: redisHost
      value: "localhost:6379"
```

```yaml
# dapr/components/statestore.yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: order-statestore
spec:
  type: state.redis
  version: v1
  metadata:
    - name: redisHost
      value: "localhost:6379"
    - name: actorStateStore
      value: "true"
```

In produzione su Azure, cambieresti `type: pubsub.redis` in `type: pubsub.azure.servicebus` — il codice applicativo non cambia.

## Publish/Subscribe con Dapr SDK

```bash
dotnet add package Dapr.AspNetCore
```

### Publisher — pubblica un evento

```csharp
// Infrastructure/Messaging/DaprOrderEventPublisher.cs
using Dapr.Client;

namespace OrderService.Infrastructure.Messaging;

public class DaprOrderEventPublisher
{
    private readonly DaprClient _dapr;
    private const string PubSubName = "order-pubsub";
    private const string TopicName  = "order-created";

    public DaprOrderEventPublisher(DaprClient dapr)
    {
        _dapr = dapr;
    }

    public async Task PublishOrderCreatedAsync(int orderId, string customer, decimal total)
    {
        var evt = new OrderCreatedEvent(orderId, customer, total, DateTime.UtcNow);
        await _dapr.PublishEventAsync(PubSubName, TopicName, evt);
    }
}

public record OrderCreatedEvent(int OrderId, string Customer, decimal Total, DateTime CreatedAt);
```

### Subscriber — consuma eventi

```csharp
// Api/Controllers/OrderEventsController.cs
using Dapr;
using Microsoft.AspNetCore.Mvc;

namespace OrderService.Api.Controllers;

[ApiController]
[Route("[controller]")]
public class OrderEventsController : ControllerBase
{
    private readonly ILogger<OrderEventsController> _logger;

    public OrderEventsController(ILogger<OrderEventsController> logger)
    {
        _logger = logger;
    }

    // Dapr chiama questo endpoint quando arriva un messaggio sul topic
    [HttpPost("order-created")]
    [Topic("order-pubsub", "order-created")]
    public async Task<IActionResult> HandleOrderCreated(OrderCreatedEvent evt)
    {
        _logger.LogInformation(
            "Ordine ricevuto via Dapr: {OrderId} da {Customer}",
            evt.OrderId, evt.Customer);

        // logica consumer
        await Task.CompletedTask;
        return Ok();
    }
}
```

## State Management con Dapr

```csharp
// Salva stato
await _dapr.SaveStateAsync("order-statestore", $"order-{orderId}", orderState);

// Leggi stato
```

## Service Invocation con Dapr

```csharp
// Chiama InventoryService senza conoscerne l'URL — Dapr lo risolve
var result = await _dapr.InvokeMethodAsync<InventoryResponse>(
    HttpMethod.Get,
    "inventory-service",    // nome del Dapr app-id del servizio target
    "inventory/check");
```

Nessun `IHttpClientFactory`, nessun URL hardcoded, nessun service discovery manuale.

## `Program.cs` — registrazione Dapr

```csharp
// Api/Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers().AddDapr();   // aggiunge DaprClient + [Topic] routing
builder.Services.AddDaprClient();

// Publisher come servizio
builder.Services.AddScoped<DaprOrderEventPublisher>();

var app = builder.Build();

app.UseRouting();
app.UseCloudEvents();          // necessario per il pub/sub Dapr (CloudEvents format)
app.MapSubscribeHandler();     // espone /dapr/subscribe — Dapr scopre i topic automaticamente
app.MapControllers();

app.Run();
```

## Dapr su Kubernetes

```yaml
# k8s/deployment.yaml — aggiunta annotazioni Dapr
spec:
  template:
    metadata:
      labels:
        app: order-service
      annotations:
        dapr.io/enabled:    "true"
        dapr.io/app-id:     "order-service"
        dapr.io/app-port:   "8080"
```

Con queste tre annotazioni, Kubernetes inietta automaticamente il sidecar Dapr nel pod — nessun altro cambiamento ai manifest.

Il codice .NET (`PublishEventAsync`, `[Topic]`) non cambia — cambia solo questo file YAML.

## Spiegazione

**Portabilità vendor** — il vantaggio principale di Dapr è che disaccoppia il codice applicativo dall'infrastruttura. Vuoi migrare da Redis a Memcached per lo state store? Cambi un YAML. Da Azure Service Bus a RabbitMQ? Cambi un YAML. Il codice C# non tocca mai l'SDK del vendor — chiama solo l'API Dapr locale.

**Overhead del sidecar** — ogni pod ha un container aggiuntivo (~20MB RAM). Per microservizi con decine di istanze, questo è trascurabile. Per workload con centinaia di istanze, valuta se i benefici giustificano l'overhead.

**Dapr vs implementazione manuale** — nei capitoli precedenti abbiamo implementato a mano: `IHttpClientFactory` (cap. 6.1), Service Bus SDK (cap. 6.3), outbox pattern (cap. 6.4), Redis cache (cap. 8.5). Tutto funziona. Dapr è un'alternativa — non un sostituto obbligatorio. Scegli Dapr quando il team gestisce microservizi in più linguaggi, o quando la portabilità tra cloud provider è un requisito reale.

**`[Topic]` e CloudEvents** — Dapr usa il formato CloudEvents per i messaggi pub/sub. `app.UseCloudEvents()` deserializza il wrapper CloudEvents automaticamente. `app.MapSubscribeHandler()` espone l'endpoint `/dapr/subscribe` che Dapr interroga per scoprire quali topic il microservizio vuole consumare — zero configurazione manuale degli subscription.

## Confronto con Java

| Spring Cloud / Java | Dapr / .NET |
|---|---|
| Spring Cloud OpenFeign | `DaprClient.InvokeMethodAsync` |
| Spring Cloud Stream + Binder | `[Topic]` + `PublishEventAsync` |
| Spring Data Redis | `DaprClient.SaveStateAsync` |
| Spring Cloud Vault | Dapr Secret Store component |
| `@EnableDiscoveryClient` (Eureka) | Dapr Service Invocation (nativo) |
| Cambio broker: riscrivere i binder | Cambio broker: aggiornare YAML componente |
