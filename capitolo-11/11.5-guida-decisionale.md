# 11.5 Guida decisionale: quale approccio scegliere

> **Analogia Java**: Spring Boot vs Quarkus vs Micronaut vs AWS Lambda — stessa domanda, stesso framework di valutazione

---

## Introduzione

Alla fine del corso hai quattro approcci per costruire un backend in .NET: microservizio classico (ASP.NET Core), serverless (Azure Functions), compilazione nativa (Native AOT), runtime portabile (Dapr). Nessuno è universalmente superiore — la scelta dipende dal carico, dal team, dai requisiti di startup, e dal vendor lock-in accettabile. Questa guida aiuta a scegliere.

## Matrice decisionale

| Criterio | Microservizio classico | Azure Functions | Native AOT | Con Dapr |
|---|---|---|---|---|
| **Startup** | 300-500ms | 300ms (isolated) | 20-50ms | +20ms (sidecar) |
| **Memoria** | 80-120MB | 80-120MB | 20-40MB | +20MB (sidecar) |
| **Cold start critico** | No | Sì | Sì | No |
| **EF Core** | ✅ | ✅ | ❌ | ✅ |
| **Traffico sporadico** | Costoso | Ideale (pay-per-use) | OK | OK |
| **Traffico costante** | Ideale | Meno efficiente | Ideale | Ideale |
| **Vendor lock-in** | Medio | Alto (Azure) | Basso | Minimo |
| **Portabilità infrastruttura** | Media | Bassa | Alta | Massima |
| **Complessità operativa** | Media | Bassa | Media | Alta (sidecar) |
| **Team multi-linguaggio** | No | No | No | Sì |

## Scenari e raccomandazioni

### Scenario 1 — API REST con traffico costante

**Scegli: Microservizio classico ASP.NET Core**

```
✓ EF Core per la persistenza
✓ Kubernetes per lo scaling
✓ Tutto il corso si applica direttamente
✗ Non serve serverless — hai bisogno di un processo always-on
```

Esempio: API di un e-commerce, backend di un'app mobile, servizio di autenticazione.

---

### Scenario 2 — Job schedulati, webhook, elaborazione asincrona

**Scegli: Azure Functions**

```
✓ Pay-per-use — non paghi quando non c'è traffico
✓ Timer trigger per batch notturni
✓ Service Bus trigger per elaborazione asincrona
✓ Zero gestione dell'infrastruttura
✗ Cold start se non usi Premium plan
✗ Vendor lock-in Azure
```

Esempio: generazione report notturni, invio email, elaborazione immagini al caricamento, webhook GitHub/Stripe.

---

### Scenario 3 — Startup critico, container denso, CLI distribuita

**Scegli: Native AOT**

```
✓ Avvio < 50ms — ideale per Functions con cold start
✓ Immagine Docker ~15MB — deployment veloce
✓ Memoria ridotta — più container per nodo
✗ Non compatibile con EF Core
✗ Build più lento (1-2 minuti)
✗ Librerie con reflection richiedono annotazioni
```

Esempio: sidecar container, CLI tool distribuito come singolo binario, Functions con cold start frequente, API gRPC ad alte performance.

---

### Scenario 4 — Team poliglotta, portabilità cloud, multi-vendor

**Scegli: Dapr (sopra qualsiasi approccio)**

```
✓ Stessa API per Redis, RabbitMQ, Azure Service Bus, Kafka
✓ Microservizi in .NET, Go, Python nella stessa architettura
✓ Cambio broker senza modifiche al codice
✗ Overhead del sidecar (~20MB RAM per pod)
✗ Complessità operativa aggiuntiva
✗ Curva di apprendimento per il team ops
```

Esempio: piattaforma con team multipli, migrazione da on-premise a cloud, architettura che deve girare su AWS e Azure indifferentemente.

---

### Scenario 5 — Startup in sviluppo, team piccolo

**Scegli: Microservizio classico + Aspire per sviluppo locale**

```
✓ Massima semplicità
✓ Tutto il corso come guida
✓ Aspire per osservabilità senza configurazione
✓ Kubernetes o Azure Container Apps per produzione
✗ Scala orizzontalmente solo con Kubernetes
```

## Albero decisionale

```
Hai bisogno di EF Core?
├── No → Considera Native AOT (se startup critico) o Dapper
└── Sì → Microservizio classico

Il traffico è sporadico o event-driven?
├── Sì → Azure Functions
└── No → Microservizio classico

Il team usa più linguaggi o hai requisiti di portabilità vendor?
├── Sì → Dapr sopra l'approccio scelto
└── No → Implementazione diretta (cap. 6-8)

Il cold start è critico (< 100ms)?
├── Sì → Native AOT (+ Functions se serverless)
└── No → Standard JIT
```

## Spiegazione

**Non esiste la scelta giusta** — ogni progetto reale finisce per usare una combinazione: API REST per gli endpoint sincroni, Functions per i job asincroni, Native AOT per i sidecar, Aspire per lo sviluppo locale. Il valore del corso è capire quando e perché, non scegliere uno e ignorare gli altri.

**Inizia semplice** — per un nuovo progetto, inizia con un microservizio classico ASP.NET Core. Aggiungi complessità solo quando hai un requisito specifico che la giustifica: un cold start misurabile che impatta gli utenti, un team che usa davvero più linguaggi, un requisito di portabilità vendor reale. La complessità prematura è il costo più alto nei microservizi.

**Riferimento al corso** — ogni paragrafo del corso è autocontenuto: puoi tornare a cap. 6.3 (Service Bus) quando devi aggiungere messaggistica, o a cap. 11.2 (Native AOT) quando hai un requisito di startup critico. Il corso è una reference, non solo una lettura sequenziale.
