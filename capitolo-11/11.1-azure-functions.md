# 11.1 Azure Functions: serverless con .NET

> **Analogia Java**: equivalente di AWS Lambda / Azure Functions con Spring Cloud Function

---

## Introduzione

`OrderService` è un microservizio sempre attivo — un processo che gira continuamente, scala orizzontalmente, ha un database. Non sempre serve questa complessità. Per operazioni event-driven, batch notturni, webhook, o API con traffico sporadico, il modello **serverless** è più semplice e più economico: scrivi una funzione, la piattaforma gestisce l'infrastruttura. Azure Functions è il serverless di Azure — supporta .NET nativamente.

## I trigger principali

Azure Functions supporta diversi trigger — ogni trigger definisce cosa avvia l'esecuzione della funzione:

| Trigger | Avvia quando | Equivalente Java |
|---|---|---|
| `HttpTrigger` | Arriva una request HTTP | Controller REST |
| `TimerTrigger` | Scadenza cron expression | `@Scheduled` in Spring |
| `ServiceBusTrigger` | Messaggio su Service Bus | `@JmsListener` |
| `BlobTrigger` | File caricato su Azure Blob | Event listener |
| `QueueTrigger` | Messaggio su Azure Queue | Queue consumer |

## HTTP Trigger

```csharp
// Functions/ProcessOrderFunction.cs
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;
using Microsoft.Extensions.Logging;
using System.Net;
using System.Text.Json;

namespace OrderProcessor;

public class ProcessOrderFunction
{
    private readonly ILogger<ProcessOrderFunction> _logger;

    public ProcessOrderFunction(ILogger<ProcessOrderFunction> logger)
    {
        _logger = logger;
    }

    [Function("ProcessOrder")]
    public async Task<HttpResponseData> Run(
        [HttpTrigger(AuthorizationLevel.Function, "post", Route = "orders/{id}/process")]
        HttpRequestData req,
        int id,
        FunctionContext context)
    {
        _logger.LogInformation("Elaborazione ordine {OrderId}", id);

        var body   = await req.ReadFromJsonAsync<ProcessOrderRequest>();
        var result = new { OrderId = id, Status = "Processed", Notes = body?.Notes, ProcessedAt = DateTime.UtcNow };

        var response = req.CreateResponse(HttpStatusCode.OK);
        await response.WriteAsJsonAsync(result);
        return response;
    }
}

public record ProcessOrderRequest(string Notes);
```

## Timer Trigger — job schedulato

```csharp
// Functions/DailyReportFunction.cs
public class DailyReportFunction
{
    private readonly ILogger<DailyReportFunction> _logger;

    public DailyReportFunction(ILogger<DailyReportFunction> logger)
    {
        _logger = logger;
    }

    // Cron: ogni giorno alle 02:00 UTC
    [Function("DailyReport")]
    public async Task Run(
        [TimerTrigger("0 0 2 * * *")] TimerInfo timer,
        FunctionContext context)
    {
        _logger.LogInformation("Generazione report giornaliero: {Time}", DateTime.UtcNow);

        if (timer.IsPastDue)
            _logger.LogWarning("Timer in ritardo — il job precedente ha impiegato troppo");

        // logica report
        await GenerateReportAsync();
    }

    private Task GenerateReportAsync() => Task.CompletedTask;
}
```

## Service Bus Trigger — consumer serverless

```csharp
// Functions/OrderCreatedConsumerFunction.cs
public class OrderCreatedConsumerFunction
{
    private readonly ILogger<OrderCreatedConsumerFunction> _logger;

    public OrderCreatedConsumerFunction(ILogger<OrderCreatedConsumerFunction> logger)
    {
        _logger = logger;
    }

    [Function("ProcessOrderCreated")]
    public async Task Run(
        [ServiceBusTrigger("order-created", "inventory-sub",
            Connection = "ServiceBusConnection")]
        string messageBody,
        FunctionContext context)
    {
        var evt = JsonSerializer.Deserialize<OrderCreatedEvent>(messageBody);
        _logger.LogInformation("Aggiornamento inventory per ordine {OrderId}", evt!.OrderId);

        // logica inventory
        await Task.CompletedTask;
    }
}

public record OrderCreatedEvent(int OrderId, string Customer, decimal Total);
```

## DI e configurazione in `Program.cs`

```csharp
// Program.cs
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;

var host = new HostBuilder()
    .ConfigureFunctionsWorkerDefaults()
    .ConfigureServices(services =>
    {
        services.AddApplicationInsightsTelemetryWorkerService();
        services.ConfigureFunctionsApplicationInsights();

        // Stessa DI di ASP.NET Core
        services.AddScoped<IOrderProcessor, OrderProcessor>();
    })
    .Build();

host.Run();
```

## Spiegazione

**Isolated worker vs in-process** — fino a .NET 8 esistevano due modelli di hosting per Azure Functions in .NET. **In-process** girava nello stesso processo del runtime Functions: facile da configurare, ma accoppiato alla versione del runtime e con DI limitata. **Isolated worker** gira in un processo separato — DI completa, middleware, versione di .NET indipendente dal runtime Functions. Microsoft ha deprecato in-process a novembre 2026 e in **.NET 10 esiste solo il modello isolated**: il progetto creato con `--worker-runtime dotnet-isolated` è l'unico disponibile. Tutto il codice di questo capitolo usa già il modello corretto.

**`AuthorizationLevel.Function`** — il trigger HTTP richiede una chiave API (`?code=...`) per essere chiamato. `Anonymous` non richiede chiave — usalo solo per webhook pubblici o con autenticazione separata.

**Quando scegliere Functions vs microservizio** — Functions è ideale per: operazioni event-driven sporadiche, webhook, job schedulati, pipeline di elaborazione. Un microservizio è meglio per: API con stato, connessioni WebSocket, operazioni che richiedono warm-up costoso, logica complessa con molte dipendenze.
