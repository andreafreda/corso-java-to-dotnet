# 11.2 Native AOT: compilazione nativa con .NET

> **Analogia Java**: equivalente di GraalVM Native Image per Spring Boot

---

## Introduzione

Il runtime .NET normalmente compila il codice in bytecode IL (Intermediate Language) e lo JIT-compila a runtime. **Native AOT** (Ahead-Of-Time) compila l'intero programma in un binario nativo già in fase di build — nessun JIT, nessun runtime .NET richiesto sulla macchina target. Il risultato: avvio in millisecondi, memoria ridotta al 30-50%, binario autonomo. Il trade-off: non tutto il codice .NET è compatibile con AOT — reflection dinamica, serializzazione generica, e alcune librerie richiedono annotazioni aggiuntive o alternative.

---

## Come si faceva in Java

```java
// GraalVM Native Image — Spring Boot 3
// pom.xml
<plugin>
    <groupId>org.graalvm.buildtools</groupId>
    <artifactId>native-maven-plugin</artifactId>
</plugin>

// Hint per la reflection (necessario se Spring non lo genera automaticamente)
@RegisterReflectionForBinding(Order.class)
@SpringBootApplication
public class OrderServiceApplication { ... }
```

```bash
# Build con GraalVM
mvn -Pnative native:compile

# Risultato: binario nativo ~50MB, avvio < 100ms
./target/order-service
```

## Compatibilità AOT in .NET 10
Non tutto funziona con Native AOT — queste funzionalità richiedono attenzione:

| Funzionalità | Stato AOT | Alternativa |
|---|---|---|
| `System.Text.Json` (source gen) | ✅ Compatibile | — |
| `System.Text.Json` (reflection) | ⚠️ Richiede annotazioni | `[JsonSerializable]` |
| EF Core (completo) | ❌ Non compatibile | Dapper o EF Core + queries precompilate |
| Minimal API | ✅ Compatibile | — |
| Controller MVC + Razor | ⚠️ Limitato | Minimal API |
| `Activator.CreateInstance` | ❌ Reflection dinamica | Factory esplicite |
| Regex compilate | ✅ con `[GeneratedRegex]` | — |
| gRPC | ✅ Compatibile | — |

> Per `OrderService` completo (EF Core + controller MVC), AOT non è applicabile direttamente. AOT è ideale per **Azure Functions**, **microservizi gRPC**, o **API minimali** con Dapper.

## Progetto AOT — Minimal API con Dapper

```bash
# Crea un progetto con AOT abilitato
dotnet new webapi -n OrderProcessorAot --use-minimal-apis
cd OrderProcessorAot
```

```xml
<!-- OrderProcessorAot.csproj -->
<Project Sdk="Microsoft.NET.Sdk.Web">

  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <PublishAot>true</PublishAot>
    <InvariantGlobalization>true</InvariantGlobalization>
  </PropertyGroup>

  <ItemGroup>
    <!-- Dapper 2.x usa reflection per il mapping — potenzialmente problematico con AOT trimming.
         Per AOT rigoroso sostituire con Dapper.AOT che usa source generators. -->
    <PackageReference Include="Dapper" Version="2.*" />
    <PackageReference Include="Microsoft.Data.SqlClient" Version="5.*" />
  </ItemGroup>

</Project>
```

## JSON con Source Generator

AOT non può usare la reflection per serializzare/deserializzare — bisogna generare il codice a compile-time:

```csharp
// Models/Order.cs
namespace OrderProcessorAot;

public record Order(int Id, string Customer, decimal Total, string Status);
public record CreateOrderRequest(string Customer, decimal Total);
```

```csharp
// JsonContext.cs — source generator per System.Text.Json
using System.Text.Json.Serialization;

namespace OrderProcessorAot;

[JsonSerializable(typeof(Order))]
[JsonSerializable(typeof(CreateOrderRequest))]
[JsonSerializable(typeof(List<Order>))]
[JsonSerializable(typeof(ProblemDetails))]
internal partial class AppJsonContext : JsonSerializerContext { }
```

## Program.cs — Minimal API AOT-compatibile

```csharp
// Program.cs
using Dapper;
using Microsoft.Data.SqlClient;
using OrderProcessorAot;

var builder = WebApplication.CreateSlimBuilder(args);  // SlimBuilder — meno overhead

// Registra il JsonContext per AOT
builder.Services.ConfigureHttpJsonOptions(options =>
{
    options.SerializerOptions.TypeInfoResolverChain.Insert(0, AppJsonContext.Default);
});

var connectionString = builder.Configuration.GetConnectionString("Default")!;
builder.Services.AddScoped(_ => new SqlConnection(connectionString));

var app = builder.Build();

// Health check minimale
app.MapGet("/health/live", () => Results.Ok(new { Status = "Healthy" }));

// GET /orders
app.MapGet("/orders", async (SqlConnection db) =>
{
    var orders = await db.QueryAsync<Order>(
        "SELECT Id, Customer, Total, Status FROM Orders");
    return Results.Ok(orders.ToList());
});

// GET /orders/{id}
app.MapGet("/orders/{id:int}", async (int id, SqlConnection db) =>
{
    var order = await db.QueryFirstOrDefaultAsync<Order>(
        "SELECT Id, Customer, Total, Status FROM Orders WHERE Id = @Id",
        new { Id = id });

    return order is null ? Results.NotFound() : Results.Ok(order);
});

// POST /orders
app.MapPost("/orders", async (CreateOrderRequest req, SqlConnection db) =>
{
    var id = await db.ExecuteScalarAsync<int>(
        "INSERT INTO Orders (Customer, Total, Status) VALUES (@Customer, @Total, 'Pending'); SELECT SCOPE_IDENTITY();",
        new { req.Customer, req.Total });

    return Results.Created($"/orders/{id}", new Order(id, req.Customer, req.Total, "Pending"));
});

app.Run();
```

## Build e publish AOT

```bash
# Publish AOT — richiede il SDK del sistema operativo target
# Su Linux produce un binario Linux nativo
dotnet publish -c Release -r linux-x64 --self-contained

# Su Windows per target Linux — usa Docker
docker run --rm \
    -v $(pwd):/app \
    -w /app \
    mcr.microsoft.com/dotnet/sdk:10.0 \
    dotnet publish -c Release -r linux-x64 --self-contained -o /app/publish
```

```
# Output
publish/
└── OrderProcessorAot        ← binario Linux nativo, ~15MB
    (nessun .dll, nessun runtime .NET)
```

## Dockerfile AOT — immagine minimale

```dockerfile
# Dockerfile.aot
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src
COPY . .
RUN dotnet publish -c Release -r linux-x64 --self-contained -o /app/publish

# Immagine finale: solo il binario — nessun runtime .NET
FROM mcr.microsoft.com/dotnet/runtime-deps:10.0
WORKDIR /app
COPY --from=build /app/publish .
USER app
ENTRYPOINT ["./OrderProcessorAot"]
```

```bash
docker build -f Dockerfile.aot -t order-processor-aot .

# Confronto dimensioni
docker images | grep order
# order-processor-aot    15MB
# order-service          120MB   (standard, con runtime)
```

## Confronto performance

```
                   Standard JIT    Native AOT
─────────────────────────────────────────────
Avvio              300-500ms       20-50ms
Memoria RSS        80-120MB        20-40MB
Dimensione immagine 120MB          15MB
Throughput (req/s) ≈ identico      ≈ identico (dopo warm-up)
Build time         5s              60-120s
```

> Il vantaggio AOT è nello **startup** e nella **memoria** — il throughput stazionario è comparabile perché il JIT ottimizza il codice a runtime in base all'utilizzo reale.

## Avvertimento: AOT e Azure Functions

Azure Functions con .NET 10 isolated worker supporta AOT sperimentalmente. Esempio minimale:

```csharp
// Program.cs — Functions AOT
var host = new HostBuilder()
    .ConfigureFunctionsWorkerDefaults()
    .Build();

host.Run();
```

```csharp
// Function AOT-compatibile
public class ProcessOrderFunction
{
    [Function("ProcessOrder")]
    public HttpResponseData Run(
        [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequestData req)
    {
        // NON usare ReadFromJsonAsync con reflection — usa source generator
        var response = req.CreateResponse(HttpStatusCode.OK);
        return response;
    }
}
```

```xml
<PublishAot>true</PublishAot>
<_FunctionsSkipCleanOutput>true</_FunctionsSkipCleanOutput>
```

## Spiegazione

**`CreateSlimBuilder` vs `CreateBuilder`** — `WebApplication.CreateSlimBuilder` esclude le feature non necessarie per Minimal API (MVC, Razor, ecc.) — binario più piccolo e compatibilità AOT migliore. `CreateBuilder` include tutto lo stack ASP.NET Core.

**`[JsonSerializable]` è obbligatorio con AOT** — il source generator di `System.Text.Json` genera il codice di serializzazione a compile-time. Senza di esso il runtime cerca di usare la reflection, che non funziona in un binario AOT. Ogni tipo che viene serializzato/deserializzato deve comparire nell'attributo `[JsonSerializable]`.

**`runtime-deps` vs `runtime` vs `aspnet`** — l'immagine `runtime-deps` contiene solo le dipendenze del sistema operativo (libc, libssl, ecc.) — non il runtime .NET. Un binario AOT non ha bisogno del runtime, quindi `runtime-deps` è l'immagine base corretta. È la più piccola delle tre.

**Dapper e AOT** — Dapper 2.x usa reflection per mappare le colonne SQL sui campi C#. Con il trimming AOT, la reflection su tipi non annotati può essere rimossa dal compilatore causando errori a runtime. Per scenari AOT rigorosi usa il pacchetto **Dapper.AOT** (`dotnet add package Dapper.AOT`) che genera il codice di mapping a compile-time tramite source generators — esattamente come `[JsonSerializable]` fa per la serializzazione JSON.

**Quando usare AOT** — ideale per: Azure Functions con cold start frequenti, sidecar container con startup critico, CLI tool distribuito come singolo binario, microservizi gRPC. Non ideale per: servizi che usano EF Core, serializzazione complessa con tipi dinamici, librerie che usano reflection estensivamente.

## Confronto con Java

| GraalVM Native Image | .NET Native AOT |
|---|---|
| `mvn -Pnative native:compile` | `dotnet publish -c Release -r linux-x64` |
| `@RegisterReflectionForBinding` | `[JsonSerializable]` source generator |
| `reflect-config.json` | trimming annotations (`[DynamicallyAccessedMembers]`) |
| Spring Boot 3 supporto nativo | Minimal API + `CreateSlimBuilder` |
| Build time: 3-5 minuti | Build time: 1-2 minuti |
| Avvio < 100ms | Avvio < 50ms |
| Immagine ~80MB (distroless) | Immagine ~15MB (runtime-deps) |
