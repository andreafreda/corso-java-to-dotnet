# 8.8 Sicurezza dei dati: redazione log e dati sensibili

> **Analogia Java**: nessun equivalente diretto nativo in Spring

---

## Introduzione

I log sono il primo posto dove finiscono i dati sensibili per errore: password in chiaro, numeri di carta, token JWT, dati personali. In Spring non esiste un meccanismo standard — si usavano soluzioni custom o librerie di terze parti. In .NET il problema è uguale, ma `ILogger<T>` con Serilog offre strumenti per la redazione automatica — e .NET ha il concetto di **`[LoggerMessage]`** per log ad alte prestazioni.

---

## Il problema

```csharp
// ❌ Questi log espongono dati sensibili
_logger.LogInformation("Login: email={Email}, password={Password}", email, password);
_logger.LogDebug("Request body: {Body}", JsonSerializer.Serialize(request));
// → "Login: email=user@example.com, password=secret123"
// → "Request body: {"email":"user@example.com","creditCard":"4111111111111111"}"
```

---

## Come si faceva in Java

```java
// Spring — nessun meccanismo standard
// Soluzioni custom comuni:

// 1. Annotation custom + AOP
@Masked
private String password;

// 2. Serializer custom per Jackson
@JsonSerialize(using = MaskedSerializer.class)
private String creditCardNumber;

// 3. Filtro custom nel log pattern Logback
// Non esiste un'API standard
```

## Destructuring con Serilog — redazione selettiva

Serilog ha un meccanismo di **destructuring** che permette di controllare come gli oggetti vengono serializzati nei log:

```csharp
// Tipo wrapper con ToString() mascherato — semplice e funziona davvero
public readonly struct Redacted
{
    private readonly string _value;
    public Redacted(string value) => _value = value;
    public override string ToString() => "***REDACTED***";
}
```

```csharp
// Uso — il valore non appare mai nel log
_logger.LogInformation("Tentativo login per {Email} con token {Token}",
    email,
    new Redacted(token));  // → "Tentativo login per user@example.com con token ***REDACTED***"
```

```csharp
// Destructuring policy avanzata — filtra i campi sensibili sugli oggetti strutturati
using Serilog.Core;    // IDestructuringPolicy, ILogEventPropertyValueFactory
using Serilog.Events;  // LogEventPropertyValue, ScalarValue, StructureValue

public class SensitiveDataDestructuringPolicy : IDestructuringPolicy
{
    private static readonly HashSet<string> SensitiveFields = new(StringComparer.OrdinalIgnoreCase)
    {
        "password", "token", "secret", "creditcard", "cvv",
        "authorization", "apikey", "connectionstring"
    };

    public bool TryDestructureInstance(object value, ILogEventPropertyValueFactory factory,
        out LogEventPropertyValue? result)
    {
        // Intercetta solo oggetti strutturati (record, classi con proprietà)
        if (value is null) { result = null; return false; }

        var type       = value.GetType();
        var properties = type.GetProperties();
        if (properties.Length == 0) { result = null; return false; }

        var logProperties = properties.Select(p =>
        {
            var propValue = SensitiveFields.Contains(p.Name)
                ? new ScalarValue("***REDACTED***")
                : factory.CreatePropertyValue(p.GetValue(value), destructureObjects: true);
            return new LogEventProperty(p.Name, propValue);
        });

        result = new StructureValue(logProperties);
        return true;
    }
}
```

```csharp
// Registrazione in Program.cs
Log.Logger = new LoggerConfiguration()
    .Destructure.With<SensitiveDataDestructuringPolicy>()
    .WriteTo.Console()
    .CreateLogger();
```

## `[LoggerMessage]` — log ad alte prestazioni

`[LoggerMessage]` è un source generator che elimina il boxing e l'allocazione dei parametri a ogni log:

```csharp
// Api/Logging/OrderLogMessages.cs
using Microsoft.Extensions.Logging;

namespace OrderService.Api.Logging;

public static partial class OrderLogMessages
{
    [LoggerMessage(
        EventId = 1001,
        Level   = LogLevel.Information,
        Message = "Ordine {OrderId} creato per {Customer}")]
    public static partial void OrderCreated(
        this ILogger logger, int orderId, string customer);

    [LoggerMessage(
        EventId = 1002,
        Level   = LogLevel.Warning,
        Message = "Tentativo login fallito per {Email} — IP: {IpAddress}")]
    public static partial void LoginFailed(
        this ILogger logger, string email, string ipAddress);

    [LoggerMessage(
        EventId = 1003,
        Level   = LogLevel.Error,
        Message = "Errore creazione ordine per {Customer}: {ErrorMessage}")]
    public static partial void OrderCreationFailed(
        this ILogger logger, string customer, string errorMessage, Exception exception);
}
```

```csharp
// Uso — type-safe, zero allocazione
_logger.OrderCreated(order.Id, order.Customer);
_logger.LoginFailed(email, httpContext.Connection.RemoteIpAddress?.ToString() ?? "unknown");
```

## Redazione nei middleware — request/response logging

Il logging delle request HTTP è utile ma può esporre header sensibili (`Authorization`, `Cookie`):

```csharp
// Api/Program.cs — configura Serilog request logging per escludere header sensibili
app.UseSerilogRequestLogging(options =>
{
    options.MessageTemplate = "HTTP {RequestMethod} {RequestPath} → {StatusCode} ({Elapsed:0.0000}ms)";
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        // ✓ Aggiungi solo dati sicuri
        diagnosticContext.Set("RequestHost", httpContext.Request.Host.Value);
        diagnosticContext.Set("UserAgent",   httpContext.Request.Headers.UserAgent.ToString());

        // ✓ UserId dal token — non il token completo
        var userId = httpContext.User.FindFirst("sub")?.Value ?? "anonymous";
        diagnosticContext.Set("UserId", userId);

        // ✗ NON aggiungere: Authorization header, Cookie, body della request
    };
});
```

## Protezione dati a riposo — Data Protection API

Per dati sensibili che devono essere persistiti (es. refresh token, dati utente), ASP.NET Core ha la **Data Protection API** — equivalente di Spring Security Crypto:

```csharp
// Api/Program.cs
builder.Services.AddDataProtection()
    .PersistKeysToFileSystem(new DirectoryInfo("/keys"))  // in produzione: Azure Key Vault
    .SetApplicationName("OrderService");
```

```csharp
// Uso — cifratura e decifratura
public class SecureTokenService
{
    private readonly IDataProtector _protector;

    public SecureTokenService(IDataProtectionProvider provider)
    {
        _protector = provider.CreateProtector("OrderService.RefreshTokens");
    }

    public string Protect(string token)   => _protector.Protect(token);
    public string Unprotect(string token) => _protector.Unprotect(token);
}
```

## Checklist sicurezza dei log

```
✓ Non loggare password, token, secret, connection string
✓ Non loggare numeri di carta, CVV, IBAN
✓ Non loggare dati personali sensibili (SSN, documenti)
✓ Loggare l'ID utente, non l'email o il nome completo
✓ Mascherare i valori con Redacted o destructuring policy
✓ Escludere header Authorization e Cookie dal request logging
✓ Configurare retention dei log (non conservare per sempre)
✓ In produzione: solo livello Information o superiore
```

---

## Spiegazione

**`[LoggerMessage]` vs `_logger.LogInformation`** — `LogInformation` con stringhe di formato alloca array e fa boxing dei parametri a ogni chiamata. `[LoggerMessage]` genera codice ottimizzato a compile time — zero allocazione, zero reflection. Per microservizi ad alto throughput la differenza è misurabile.

**Dati personali e GDPR** — in Europa i log con dati personali identificabili (email, nome, IP) sono soggetti al GDPR. Definisci una retention policy: i log non vanno conservati indefinitamente. In Azure Monitor puoi configurare la retention a livello di workspace.

**Data Protection API vs BCrypt** — Data Protection è per cifratura simmetrica reversibile (cifri e decifri). BCrypt è per hashing irreversibile (password). Non sono intercambiabili — usa Data Protection per token che devi decifrare, BCrypt per password che non devi mai decifrare.

## Confronto con Java

| Java / Spring | .NET |
|---|---|
| Nessun meccanismo standard per log redaction | `Redacted` wrapper type + Serilog destructuring |
| `@JsonSerialize` custom per mascherare campi | Destructuring policy di Serilog |
| `LoggerFactory.getLogger()` — overhead a runtime | `[LoggerMessage]` source generator — zero allocazione |
| Spring Security Crypto per cifratura | Data Protection API |
| `PasswordEncoder` (BCrypt) | `BCrypt.Net-Next` package |
| Nessun request logging integrato | `UseSerilogRequestLogging` con filtro header |
