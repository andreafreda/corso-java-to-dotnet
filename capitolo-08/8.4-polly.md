# 8.4 Resilienza con Polly (retry, circuit breaker)

> **Analogia Java**: equivalente di Resilience4j

---

## Introduzione

`InventoryService` ГЁ temporaneamente irraggiungibile. Senza resilienza `OrderService` fallisce immediatamente e restituisce 500 al client. Con **Polly** puoi definire policy di retry, circuit breaker e timeout вҖ” lo stesso ruolo che `Resilience4j` gioca nell'ecosistema Spring. In .NET 8 Polly ГЁ integrato direttamente in `IHttpClientFactory` tramite `Microsoft.Extensions.Http.Resilience`.

---

## Come si faceva in Java

```java
// Resilience4j вҖ” retry + circuit breaker
@Service
public class InventoryClient {

    @Retry(name = "inventory", fallbackMethod = "fallback")
    @CircuitBreaker(name = "inventory", fallbackMethod = "fallback")
    public boolean isAvailable(int productId, int quantity) {
        return restTemplate.getForObject(
            "http://inventory-service/inventory/{id}/available?quantity={qty}",
            Boolean.class, productId, quantity);
    }

    private boolean fallback(int productId, int quantity, Exception e) {
        log.warn("InventoryService non disponibile, fallback a true");
        return true;
    }
}
```

```yaml
# application.yml
resilience4j:
  retry:
    instances:
      inventory:
        maxAttempts: 3
        waitDuration: 500ms
        retryExceptions:
          - java.io.IOException
  circuitbreaker:
    instances:
      inventory:
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
```

## Resilienza standard вҖ” configurazione rapida

```csharp
// Infrastructure/DependencyInjection.cs
services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    client.BaseAddress = new Uri(configuration["Services:InventoryUrl"]!);
})
.AddStandardResilienceHandler();  // retry + circuit breaker + timeout preconfigurati
```

`AddStandardResilienceHandler()` applica automaticamente:
- **Retry**: 3 tentativi con backoff esponenziale
- **Circuit breaker**: si apre dopo il 10% di fallimenti in una finestra di 30 secondi
- **Timeout**: 30s totali, 10s per singolo tentativo

Per la maggior parte dei casi ГЁ sufficiente вҖ” zero configurazione.

## Configurazione personalizzata

```csharp
// Infrastructure/DependencyInjection.cs вҖ” configurazione esplicita
services.AddHttpClient<IInventoryClient, InventoryClient>(client =>
{
    client.BaseAddress = new Uri(configuration["Services:InventoryUrl"]!);
})
.AddResilienceHandler("inventory-pipeline", pipeline =>
{
    // Timeout per singola chiamata
    pipeline.AddTimeout(TimeSpan.FromSeconds(5));

    // Retry con backoff esponenziale
    pipeline.AddRetry(new HttpRetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        Delay            = TimeSpan.FromMilliseconds(200),
        BackoffType      = DelayBackoffType.Exponential,
        UseJitter        = true,  // evita thundering herd
        ShouldHandle     = args => args.Outcome switch
        {
            { Exception: HttpRequestException } => PredicateResult.True(),
            { Result.StatusCode: HttpStatusCode.ServiceUnavailable } => PredicateResult.True(),
            { Result.StatusCode: HttpStatusCode.TooManyRequests }    => PredicateResult.True(),
            _ => PredicateResult.False()
        }
    });

    // Circuit breaker
    pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
    {
        FailureRatio              = 0.5,   // apri se >50% delle chiamate falliscono
        SamplingDuration          = TimeSpan.FromSeconds(30),
        MinimumThroughput         = 5,     // almeno 5 chiamate prima di valutare
        BreakDuration             = TimeSpan.FromSeconds(15)
    });
});
```

## Fallback вҖ” comportamento quando tutto fallisce

```csharp
// Infrastructure/Http/InventoryClient.cs
using Polly.CircuitBreaker;  // BrokenCircuitException

public async Task<bool> IsAvailableAsync(int productId, int quantity, CancellationToken ct = default)
{
    try
    {
        var response = await _client.GetAsync(
            $"/inventory/{productId}/available?quantity={quantity}", ct);

        response.EnsureSuccessStatusCode();
        return await response.Content.ReadFromJsonAsync<bool>(ct);
    }
    catch (BrokenCircuitException ex)
    {
        // Circuit breaker aperto вҖ” InventoryService ГЁ noto come down
        _logger.LogWarning("Circuit breaker aperto per InventoryService: {Message}", ex.Message);
        return false;  // fallback conservativo: non disponibile
    }
    catch (HttpRequestException ex)
    {
        _logger.LogWarning(ex, "InventoryService non raggiungibile dopo i retry");
        return false;
    }
}
```

## Spiegazione

**`UseJitter`** вҖ” aggiunge un ritardo casuale al backoff. Senza jitter, tutti i client che falliscono nello stesso istante riproveranno esattamente allo stesso tempo вҖ” **thundering herd**. Con jitter i retry vengono distribuiti nel tempo.

**`MinimumThroughput`** вҖ” il circuit breaker non si apre finchГ© non ha osservato almeno N chiamate. Evita falsi positivi durante il cold start quando le prime chiamate falliscono per motivi non correlati all'availability.

**Polly v8 vs v7** вҖ” Polly v7 usava `Policy.Handle<>().Retry()` вҖ” un'API fluente. Polly v8 (integrato in `Microsoft.Extensions.Http.Resilience`) usa pipeline e strategy вҖ” piГ№ composabile e performante. La documentazione online mostra spesso Polly v7 вҖ” verifica sempre la versione.

## Confronto con Java

| Resilience4j | Polly v8 / Microsoft.Extensions.Http.Resilience |
|---|---|
| `@Retry(name = "x")` | `.AddRetry(new HttpRetryStrategyOptions {...})` |
| `@CircuitBreaker(name = "x")` | `.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions {...})` |
| `fallbackMethod` sull'annotation | `catch (BrokenCircuitException)` nel client |
| `resilience4j.retry.instances.x.maxAttempts` | `MaxRetryAttempts` nella strategy |
| `waitDurationInOpenState` | `BreakDuration` nella circuit breaker strategy |
| `@TimeLimiter` | `.AddTimeout(TimeSpan.FromSeconds(n))` |
| Configurazione in `application.yml` | Configurazione in codice o `appsettings.json` |
