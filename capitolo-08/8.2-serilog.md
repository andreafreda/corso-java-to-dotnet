# 8.2 Logging strutturato con Serilog

> **Analogia Java**: equivalente di Logback + SLF4J

---

## Introduzione

.NET ha un sistema di logging nativo (`ILogger<T>`) — l'astrazione è ottima, ma il default scrive testo non strutturato su console. In produzione hai bisogno di log in formato JSON, correlabili per `RequestId`, scritti su sink diversi (console, file, Elasticsearch, Application Insights). **Serilog** è il logger strutturato più usato nell'ecosistema .NET — l'equivalente di Logback con un'API più fluente.

## Come si faceva in Java

```xml
<!-- logback-spring.xml — configurazione Logback -->
<configuration>
  <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
  </appender>

  <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/app.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
      <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
    </rollingPolicy>
    <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
  </appender>

  <root level="INFO">
    <appender-ref ref="JSON"/>
    <appender-ref ref="FILE"/>
  </root>
</configuration>
```

## Configurazione in `Program.cs`

```csharp
// Api/Program.cs
using Serilog;
using Serilog.Formatting.Compact;

// Configura Serilog prima di tutto il resto — cattura anche gli errori di startup
Log.Logger = new LoggerConfiguration()
    .ReadFrom.Configuration(builder.Configuration)  // legge da appsettings.json
    .Enrich.FromLogContext()                         // aggiunge proprietà dal LogContext
    .Enrich.WithMachineName()
    .Enrich.WithEnvironmentName()
    .WriteTo.Console(new CompactJsonFormatter())     // JSON su console
    .WriteTo.File(
        formatter: new CompactJsonFormatter(),
        path:      "logs/orderservice-.log",
        rollingInterval: RollingInterval.Day,
        retainedFileCountLimit: 7)
    .CreateLogger();

builder.Host.UseSerilog();  // sostituisce il logger nativo di .NET

// ...

// Log delle request HTTP — equivalente di access log
app.UseSerilogRequestLogging(options =>
{
    options.MessageTemplate =
        "HTTP {RequestMethod} {RequestPath} → {StatusCode} ({Elapsed:0.0000}ms)";
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        diagnosticContext.Set("RequestHost", httpContext.Request.Host.Value);
        diagnosticContext.Set("UserId",
            httpContext.User.FindFirst("sub")?.Value ?? "anonymous");
    };
});
```

## Uso con `ILogger<T>` — nessun cambiamento

Serilog si integra con `ILogger<T>` di .NET — il codice applicativo non cambia:

```csharp
// Core/Services/OrderService.cs — invariato
public class OrderService : IOrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(IOrderRepository repository, ILogger<OrderService> logger)
    {
        _logger = logger;
    }

    public async Task<Order> CreateAsync(CreateOrderRequest request, CancellationToken ct)
    {
        // Logging strutturato — le proprietà diventano campi JSON
        _logger.LogInformation(
            "Creazione ordine per {Customer}, totale {Total:C}",
            request.Customer, request.Total);

        var order = /* ... */;

        _logger.LogInformation(
            "Ordine {OrderId} creato con successo per {Customer}",
            order.Id, order.Customer);

        return order;
    }
}
```

## Log context — arricchire i log con dati custom

```csharp
// Aggiungi proprietà al log context per tutta la durata dello scope
using (LogContext.PushProperty("OrderId", order.Id))
using (LogContext.PushProperty("CustomerId", customerId))
{
    _logger.LogInformation("Inizio elaborazione ordine");
    await ProcessOrderAsync(order);
    _logger.LogInformation("Elaborazione completata");
    // Tutti i log in questo scope avranno OrderId e CustomerId
}
```

## Livelli di log

```csharp
_logger.LogTrace("Dettaglio massimo — solo debug profondo");      // Trace
_logger.LogDebug("Debug — disabilitato in produzione");           // Debug
_logger.LogInformation("Evento normale del flusso applicativo");  // Information
_logger.LogWarning("Situazione anomala ma non bloccante");        // Warning
_logger.LogError(ex, "Errore gestito con eccezione");             // Error
_logger.LogCritical("Errore fatale — servizio non funzionante");  // Critical
```

## Spiegazione

**`ILogger<T>` vs `Log.Logger`** — il codice applicativo usa sempre `ILogger<T>` — l'astrazione .NET. Non referenziare mai `Log.Logger` di Serilog direttamente nel codice business. Serilog è un dettaglio di infrastruttura, come EF Core.

**Logging strutturato vs string interpolation** — non usare `$"Ordine {order.Id} creato"`. Usa `LogInformation("Ordine {OrderId} creato", order.Id)` — Serilog cattura `OrderId` come proprietà searchable nel log, non come parte della stringa. Questa è la differenza fondamentale tra logging strutturato e logging testuale.

**`WriteTo.Console` con `CompactJsonFormatter`** — in sviluppo il JSON è meno leggibile. Puoi usare `WriteTo.Console()` senza formatter per output human-readable in Development, e aggiungere il JSON solo in Production via `appsettings.{Environment}.json`.

## Confronto con Java

| Logback + SLF4J | Serilog + ILogger\<T\> |
|---|---|
| `LoggerFactory.getLogger(T.class)` | `ILogger<T>` iniettato via DI |
| `log.info("msg: {}", value)` | `_logger.LogInformation("msg: {Prop}", value)` |
| `logback-spring.xml` | Configurazione in `appsettings.json` + `Program.cs` |
| `LogstashEncoder` per JSON | `CompactJsonFormatter` |
| Appender (Console, File, Logstash) | Sink (Console, File, Elasticsearch, Application Insights) |
| MDC (Mapped Diagnostic Context) | `LogContext.PushProperty()` |
| `logging.level.root=INFO` in `application.yml` | `Serilog:MinimumLevel:Default` in `appsettings.json` |
