# 8.6 Gestione configurazione centralizzata

> **Analogia Java**: equivalente di Spring Cloud Config

---

## Introduzione

In un'architettura microservizi ogni servizio ha il proprio `appsettings.json`. Cambiare una configurazione comune — una stringa di connessione, una feature flag, un timeout — richiede un deploy per ogni servizio. La configurazione centralizzata risolve questo: un unico store da cui tutti i servizi leggono a runtime. In Java era Spring Cloud Config Server. In Azure è **Azure App Configuration** — con supporto per refresh dinamico e integrazione con Key Vault per i segreti.

---

## Come si faceva in Java

```yaml
# Config Server — repository Git centralizzato
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/org/config-repo
```

```yaml
# Client — legge la configurazione dal Config Server
spring:
  config:
    import: "configserver:http://config-server:8888"
  cloud:
    config:
      name: order-service
      profile: production
```

```java
// Refresh dinamico — ricarica senza restart
@RefreshScope
@RestController
public class ConfigController {
    @Value("${feature.maxOrders}")
    private int maxOrders;
}
```

## La gerarchia di configurazione in ASP.NET Core

Prima di aggiungere un config server esterno, vale la pena capire la gerarchia nativa:

```csharp
// L'ordine di override (ultimo vince)
// 1. appsettings.json                       — valori di default
// 2. appsettings.{Environment}.json         — override per ambiente
// 3. User Secrets (solo Development)        — segreti locali
// 4. Variabili d'ambiente                   — override da container/K8s
// 5. Argomenti da command line              — override al runtime
```

In molti casi questa gerarchia è sufficiente — le variabili d'ambiente in Kubernetes coprono il caso d'uso principale del config server.

---

## Azure App Configuration

```bash
dotnet add OrderService.Api package Microsoft.Azure.AppConfiguration.AspNetCore
```

```csharp
// Api/Program.cs
builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(builder.Configuration["AppConfiguration:ConnectionString"])
           .Select("OrderService:*")           // solo le chiavi del nostro servizio
           .ConfigureRefresh(refresh =>
           {
               // Quando questa chiave cambia, ricarica tutte le configurazioni
               refresh.Register("OrderService:Sentinel", refreshAll: true)
                      .SetCacheExpiry(TimeSpan.FromSeconds(30));
           });
});

// Abilita il refresh dinamico nel middleware
app.UseAzureAppConfiguration();
```

```json
// appsettings.Development.json
{
  "AppConfiguration": {
    "ConnectionString": "Endpoint=https://...;Id=...;Secret=..."
  }
}
```

## Emulatore locale — configurazione da file JSON

Per sviluppo locale senza Azure puoi simulare il config server con un file JSON aggiuntivo:

```json
// config/shared.json — configurazione condivisa tra servizi
{
  "Shared": {
    "DefaultTimeout": 30,
    "MaxRetries": 3,
    "FeatureFlags": {
      "EnableInventoryCheck": true
    }
  }
}
```

```csharp
// Api/Program.cs — carica configurazione da file esterno
builder.Configuration
    .AddJsonFile("config/shared.json", optional: true, reloadOnChange: true);
```

`reloadOnChange: true` ricarica il file quando cambia su disco — utile in sviluppo per testare il comportamento di refresh senza Azure.

## Configurazione e segreti — Azure Key Vault

Le connection string e le API key non vanno in `appsettings.json` e nemmeno in Azure App Configuration in chiaro. Azure Key Vault è lo store per i segreti:

```bash
dotnet add OrderService.Api package Azure.Extensions.AspNetCore.Configuration.Secrets
```

```csharp
// Api/Program.cs
builder.Configuration.AddAzureKeyVault(
    new Uri($"https://{builder.Configuration["KeyVault:Name"]}.vault.azure.net/"),
    new DefaultAzureCredential());
```

Con `DefaultAzureCredential` il codice usa:
- **In sviluppo**: credenziali di Azure CLI (`az login`)
- **In produzione**: Managed Identity del container — nessuna chiave da gestire nel codice

---

## User Secrets — segreti locali in sviluppo

Alternativa leggera a Key Vault per lo sviluppo locale — i segreti sono salvati fuori dalla cartella del progetto (non versionati):

```bash
dotnet user-secrets init --project OrderService.Api
dotnet user-secrets set "Jwt:Key" "my-local-secret-key" --project OrderService.Api
dotnet user-secrets set "ConnectionStrings:Default" "Server=..." --project OrderService.Api
```

I segreti vengono caricati automaticamente in `Development` — sovrascrivono `appsettings.Development.json`.

## Spiegazione

**Variabili d'ambiente come config server** — in Kubernetes puoi iniettare configurazione via `ConfigMap` e segreti via `Secret` come variabili d'ambiente. Per molti team questo è sufficiente — non serve Azure App Configuration se le configurazioni cambiano solo durante i deploy.

**`reloadOnChange` vs refresh dinamico** — `reloadOnChange: true` su file JSON ricarica la configurazione quando il file cambia su disco, senza restart del processo. Azure App Configuration con `SetCacheExpiry` fa la stessa cosa via polling verso il cloud. Il comportamento per il codice è identico — `IOptionsSnapshot<T>` o `IOptionsMonitor<T>` ricevono i nuovi valori automaticamente.

**Segreti mai nel codice** — mai committare connection string, API key o JWT secret nel repository. User Secrets per sviluppo, Key Vault per produzione. Le variabili d'ambiente sono un compromesso accettabile in K8s con `Secret` objects.

## Confronto con Java

| Spring Cloud Config | ASP.NET Core |
|---|---|
| Config Server Git-backed | Azure App Configuration |
| `spring.config.import: configserver:` | `AddAzureAppConfiguration(options => ...)` |
| `@RefreshScope` per reload dinamico | `IOptionsSnapshot<T>` — reload automatico per request |
| `@Value("${key}")` | `IOptions<T>` / `IConfiguration["key"]` |
| `spring.cloud.vault` per segreti | `AddAzureKeyVault` + `DefaultAzureCredential` |
| `.env` / `application-local.yml` per segreti locali | `dotnet user-secrets` |
| Profili Spring (`dev`, `prod`) | `ASPNETCORE_ENVIRONMENT` + `appsettings.{env}.json` |
