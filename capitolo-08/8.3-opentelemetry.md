# 8.3 Distributed tracing con OpenTelemetry

> **Analogia Java**: equivalente di Micrometer Tracing / Spring Cloud Sleuth

---

## Introduzione

Con più microservizi una singola richiesta attraversa `OrderService` → `InventoryService` → `NotificationService`. Se qualcosa va lento o fallisce, come trovi dove? Il **distributed tracing** aggiunge un `TraceId` condiviso tra tutti i servizi coinvolti — ogni span registra l'intervallo di tempo di ogni operazione. **OpenTelemetry** è lo standard aperto per tracing, metrics e logs — indipendente dal vendor, supportato nativamente da .NET 8+.

```xml
<!-- Spring Cloud Sleuth + Zipkin -->

  zipkin:
    base-url: http://zipkin:9411
  sleuth:
    sampler:
      probability: 1.0
```

```java

// e li propaga negli header HTTP (B3 propagation)
// Nessuna modifica al codice applicativo
```

## OpenTelemetry in .NET — installazione

```bash
dotnet add OrderService.Api package OpenTelemetry.Extensions.Hosting
dotnet add OrderService.Api package OpenTelemetry.Instrumentation.AspNetCore
dotnet add OrderService.Api package OpenTelemetry.Instrumentation.Http
dotnet add OrderService.Api package OpenTelemetry.Instrumentation.EntityFrameworkCore
dotnet add OrderService.Api package OpenTelemetry.Exporter.Otlp
dotnet add OrderService.Api package OpenTelemetry.Exporter.Console
```

## Configurazione in `Program.cs`

```csharp
// Api/Program.cs
using OpenTelemetry.Resources;
using OpenTelemetry.Trace;

builder.Services.AddOpenTelemetry()
    .ConfigureResource(resource => resource
        .AddService(
            serviceName:    "OrderService",
            serviceVersion: "1.0.0"))
    .WithTracing(tracing => tracing
        .AddAspNetCoreInstrumentation()     // span per ogni request HTTP in entrata
        .AddHttpClientInstrumentation()     // span per ogni HttpClient in uscita
        .AddEntityFrameworkCoreInstrumentation()  // span per ogni query EF Core
        .AddOtlpExporter(options =>
        {
            options.Endpoint = new Uri(
                builder.Configuration["OpenTelemetry:Endpoint"] ?? "http://localhost:4317");
        })
        .AddConsoleExporter());             // utile in sviluppo
```

```json
// appsettings.Development.json
{
  "OpenTelemetry": {
    "Endpoint": "http://localhost:4317"
  }
}
```

## Span custom — traccia operazioni specifiche

.NET 8+ ha `System.Diagnostics.ActivitySource` nativo — OpenTelemetry lo legge automaticamente:

```csharp
// Core/Services/OrderService.cs
using System.Diagnostics;

public class OrderService : IOrderService
{
    private static readonly ActivitySource ActivitySource =
        new("OrderService.Core", "1.0.0");

    public async Task<Order> CreateAsync(CreateOrderRequest request, CancellationToken ct)
    {
        using var activity = ActivitySource.StartActivity("CreateOrder");
        activity?.SetTag("order.customer", request.Customer);
        activity?.SetTag("order.total",    request.Total);

        var order = await _repository.AddAsync(/* ... */, ct);

        activity?.SetTag("order.id", order.Id);
        activity?.SetStatus(ActivityStatusCode.Ok);

        return order;
    }
}
```

```csharp
// Program.cs — registra l'ActivitySource personalizzata
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing => tracing
        .AddSource("OrderService.Core")   // ← aggiunto
        .AddAspNetCoreInstrumentation()
        /* ... */);
```

## TraceId nei log — correlazione

Con Serilog e OpenTelemetry puoi aggiungere il `TraceId` automaticamente a ogni log entry:

```bash
dotnet add OrderService.Api package Serilog.Enrichers.Span
```

```csharp
// Program.cs — aggiungi l'enricher a Serilog
Log.Logger = new LoggerConfiguration()
    .Enrich.WithSpan()   // aggiunge TraceId e SpanId ai log
    .WriteTo.Console(new CompactJsonFormatter())
    /* ... */
    .CreateLogger();
```

```json
// Log con TraceId — correlabile con Jaeger
{
  "@t": "2025-01-01T10:00:00Z",
  "@m": "Ordine 42 creato con successo",
  "OrderId": 42,
  "TraceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "SpanId":  "00f067aa0ba902b7"
}
```

## Spiegazione

**`ActivitySource` nativo** — .NET ha il tracing built-in con `System.Diagnostics.Activity`. Non devi dipendere da OpenTelemetry nel codice business — usi l'API nativa e OpenTelemetry la legge. È come usare `ILogger<T>` invece di Serilog direttamente.

**Propagation automatica** — quando `OrderService` chiama `InventoryService` via `HttpClient`, OpenTelemetry aggiunge automaticamente gli header `traceparent` e `tracestate` alla request. Il servizio destinatario li legge e crea span figli sotto lo stesso `TraceId`. Nessun codice manuale.

**OTLP exporter** — OTLP (OpenTelemetry Protocol) è il protocollo standard. Puoi puntare all'exporter verso Jaeger (sviluppo), Zipkin, Azure Monitor, Datadog, Grafana Tempo — cambia solo l'endpoint, il codice non tocca.

## Confronto con Java

| Micrometer Tracing / Sleuth | OpenTelemetry in .NET |
|---|---|
| Propagazione automatica via Sleuth | Propagazione automatica via `AddHttpClientInstrumentation` |
| `Tracer` / `Span` API di Micrometer | `ActivitySource` / `Activity` — API nativa .NET |
| `@NewSpan` per span custom | `ActivitySource.StartActivity("name")` |
| Zipkin / Jaeger come backend | Jaeger / Azure Monitor / Grafana via OTLP exporter |
| MDC con TraceId automatico in log | `Serilog.Enrichers.Span` per TraceId nei log |
| `spring.zipkin.base-url` | `OpenTelemetry:Endpoint` in `appsettings.json` |
