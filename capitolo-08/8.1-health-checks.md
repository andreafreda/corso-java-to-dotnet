# 8.1 Health checks (`/health`)

> **Analogia Java**: equivalente di Spring Boot Actuator

---

## Introduzione

Kubernetes, load balancer e orchestratori hanno bisogno di sapere se un'istanza del microservizio è viva e pronta a ricevere traffico. Spring Boot lo faceva con Actuator — `/actuator/health` esposto automaticamente. In ASP.NET Core lo fai con il middleware `AddHealthChecks` — esplicito, configurabile, estendibile.

---

## Come si faceva in Java

```java
// Spring Boot Actuator — zero configurazione
// Aggiungere il dependency è sufficiente
// <dependency>
//   <groupId>org.springframework.boot</groupId>
//   <artifactId>spring-boot-starter-actuator</artifactId>
// </dependency>

// application.yml
management:
  endpoints:
    web:
      exposure:
        include: health, info
  endpoint:
    health:
      show-details: always
```

```java
// Health check custom
@Component
public class DatabaseHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        try {
            dataSource.getConnection().isValid(1);
            return Health.up().withDetail("database", "reachable").build();
        } catch (Exception e) {
            return Health.down().withDetail("error", e.getMessage()).build();
        }
    }
}
```

## Configurazione base

```csharp
// Api/Program.cs
builder.Services.AddHealthChecks()
    // Database — verifica che EF Core riesca a connettersi
    .AddDbContextCheck<AppDbContext>("database")
    // Service Bus — verifica che la queue sia raggiungibile
    .AddAzureServiceBusQueue(
        connectionString: builder.Configuration.GetConnectionString("ServiceBus")!,
        queueName:        "order-events",
        name:             "servicebus")
    // Check custom — logica applicativa
    .AddCheck<OrderServiceHealthCheck>("order-service");

// ...

// Endpoint separati: liveness (è vivo?) e readiness (è pronto?)
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    // Nessun check specifico — se il processo risponde alla request, è vivo.
    // Predicate = _ => false è esplicito: liveness non esegue query al DB.
    Predicate = _ => false
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate      = _ => true,
    ResponseWriter = WriteHealthResponse
});
```

## Health check custom

```csharp
// Api/HealthChecks/OrderServiceHealthCheck.cs
using Microsoft.Extensions.Diagnostics.HealthChecks;
using OrderService.Core.Interfaces.Repositories;

namespace OrderService.Api.HealthChecks;

public class OrderServiceHealthCheck : IHealthCheck
{
    private readonly IOrderRepository _repository;

    public OrderServiceHealthCheck(IOrderRepository repository)
    {
        _repository = repository;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context, CancellationToken ct = default)
    {
        try
        {
            // Query leggera — GetByIdAsync(0) restituisce null ma prova la connessione al repository.
            // Non usare GetAllAsync: caricherebbe tutti gli ordini solo per un health check.
            await _repository.GetByIdAsync(0, ct);
            return HealthCheckResult.Healthy("Repository raggiungibile");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy("Repository non raggiungibile", ex);
        }
    }
}
```

## Response writer — output JSON dettagliato

```csharp
// Api/Program.cs — aggiungi il writer
static Task WriteHealthResponse(HttpContext context, HealthReport report)
{
    context.Response.ContentType = "application/json";
    var result = new
    {
        status  = report.Status.ToString(),
        checks  = report.Entries.Select(e => new
        {
            name        = e.Key,
            status      = e.Value.Status.ToString(),
            description = e.Value.Description,
            duration    = e.Value.Duration.TotalMilliseconds + "ms"
        }),
        totalDuration = report.TotalDuration.TotalMilliseconds + "ms"
    };
    return context.Response.WriteAsJsonAsync(result);
}
```

## Liveness vs Readiness in Kubernetes

```yaml
# k8s/deployment.yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 30

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
```

- **Liveness**: il container è vivo? Se fallisce, Kubernetes lo riavvia
- **Readiness**: il container è pronto a ricevere traffico? Se fallisce, Kubernetes lo rimuove dal load balancer senza riavviarlo

---

## Spiegazione

**`AddDbContextCheck`** — esegue una query leggera (`SELECT 1`) per verificare che il database sia raggiungibile. Non usa `SaveChangesAsync` — non modifica dati.

**Liveness vs Readiness** — Actuator di Spring Boot espone un singolo `/health`. In Kubernetes servono due endpoint separati con semantiche diverse. `/health/live` usa `Predicate = _ => false`: non esegue nessun check, risponde solo con 200 se il processo è in piedi — è esattamente il comportamento corretto per un liveness probe. Se il processo fosse morto, non risponderebbe affatto. `/health/ready` esegue tutti i check (DB, Service Bus, repository) e torna Unhealthy se uno fallisce — Kubernetes smette di mandare traffico al pod senza riavviarlo.

**Health check nel DI** — `OrderServiceHealthCheck` viene risolto dal DI container — può iniettare qualsiasi servizio registrato. In Spring si usa `@Component` con auto-detection; qui la registrazione è esplicita con `.AddCheck<T>()`.

## Confronto con Java

| Spring Boot Actuator | ASP.NET Core Health Checks |
|---|---|
| Endpoint automatico `/actuator/health` | `MapHealthChecks("/health")` esplicito |
| `HealthIndicator` interface | `IHealthCheck` interface |
| `Health.up()` / `Health.down()` | `HealthCheckResult.Healthy()` / `HealthCheckResult.Unhealthy()` |
| Auto-detection con `@Component` | Registrazione esplicita con `.AddCheck<T>()` |
| `management.endpoint.health.show-details` | `ResponseWriter` custom |
| Liveness/Readiness via Actuator groups | `MapHealthChecks` con `Predicate` per tag |
