# 5.4 Repository con EF Core: `InMemoryOrderRepository` va in pensione

> **Analogia Java**: equivalente di passare da una `List<>` in memoria a un `JpaRepository` reale

---

## Introduzione

Il database gira, la migration ha creato la tabella `Orders`. Manca l'ultimo pezzo: un repository che parli davvero con il database. In 4.8 abbiamo registrato `InMemoryOrderRepository` sapendo che sarebbe stato temporaneo — ora lo sostituiamo con `EfCoreOrderRepository`. `IOrderRepository` non cambia. `OrderService` non cambia. `OrderController` non cambia. Solo `Infrastructure` si aggiorna.

---

## Come si faceva in Java

```java
// Spring Data JPA — zero implementazione, tutto generato
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByStatus(OrderStatus status);
}

// Registrazione nel contesto Spring — automatica con @Repository
// La stringa di connessione stava in application.yml
```

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:sqlserver://localhost:1433;databaseName=OrderDb;encrypt=false
    username: sa
    password: Password123!
  jpa:
    hibernate:
      ddl-auto: validate
```

In Spring Data JPA non scrivi l'implementazione del repository — Spring la genera a runtime. In EF Core scrivi l'implementazione tu, ma è semplice: `DbContext` fa il lavoro pesante.

## Sostituire `InMemoryOrderRepository` con EF Core

> 📝 **Code Step 11** — Crea `EfCoreOrderRepository`, aggiorna `AddInfrastructure` e `Program.cs`

### 1. Il repository EF Core

```
OrderService.Infrastructure/
└── Repositories/
    ├── InMemoryOrderRepository.cs   ← rimane per riferimento, non più registrato
    └── EfCoreOrderRepository.cs     ← nuovo
```

```csharp
// Infrastructure/Repositories/EfCoreOrderRepository.cs
using Microsoft.EntityFrameworkCore;
using OrderService.Core.Entities;
using OrderService.Core.Interfaces.Repositories;
using OrderService.Infrastructure.Persistence;

namespace OrderService.Infrastructure.Repositories;

public class EfCoreOrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;

    public EfCoreOrderRepository(AppDbContext db)
    {
        _db = db;
    }

    public async Task<IEnumerable<Order>> GetAllAsync(CancellationToken ct = default) =>
        await _db.Orders.ToListAsync(ct);

    public async Task<Order?> GetByIdAsync(int id, CancellationToken ct = default) =>
        await _db.Orders.FindAsync([id], ct);

    public async Task<Order> AddAsync(Order order, CancellationToken ct = default)
    {
        _db.Orders.Add(order);
        await _db.SaveChangesAsync(ct);
        return order;
    }
}
```

### 2. Aggiorna `AddInfrastructure`

`AddDbContext` è già registrato da 5.3 — qui sostituiamo solo il repository:

```csharp
// Infrastructure/DependencyInjection.cs — sostituisce InMemoryOrderRepository con EfCoreOrderRepository
public static IServiceCollection AddInfrastructure(
    this IServiceCollection services,
    IConfiguration configuration)
{
    services.AddDbContext<AppDbContext>(options =>
        options.UseSqlServer(configuration.GetConnectionString("Default")));
    // PostgreSQL: sostituisci UseSqlServer con UseNpgsql (package Npgsql.EntityFrameworkCore.PostgreSQL)

    services.AddScoped<IOrderRepository, EfCoreOrderRepository>();   // era: AddSingleton<...InMemoryOrderRepository>
    return services;
}
```

## Spiegazione

**`FindAsync([id], ct)`** — `FindAsync` controlla prima la cache del change tracker, poi colpisce il database. Per `GetByIdAsync` è la scelta corretta. Per query più complesse (filtri, join) usa `FirstOrDefaultAsync` con LINQ.

**`SaveChangesAsync` dentro `AddAsync`** — in questo repository ogni operazione di scrittura fa il suo `SaveChangesAsync`. In scenari con più operazioni coordinate (es. create order + update inventory) userai transazioni esplicite — vedi 5.5.

**`AddScoped` per `AppDbContext`** — EF Core registra `DbContext` come `Scoped` per default. `EfCoreOrderRepository` è anch'esso `Scoped`: ogni richiesta HTTP ottiene la propria istanza di entrambi. Il change tracker è pulito a ogni richiesta.

**`IConfiguration` in `AddInfrastructure`** — il metodo ora accetta configurazione dall'esterno. `Program.cs` passa `builder.Configuration` che legge da `appsettings.json`, `appsettings.Development.json` e variabili d'ambiente. In produzione basterà sovrascrivere la variabile d'ambiente `ConnectionStrings__Default` senza toccare il codice.

**`InMemoryOrderRepository` non va cancellato subito** — è utile per i test unitari. Nel capitolo 9 lo useremo come fake nei test che non devono colpire il database.

## Confronto con Java

| Spring Data JPA | EF Core |
|---|---|
| `JpaRepository<T, ID>` — implementazione generata | `DbSet<T>` — scrivi l'implementazione, è semplice |
| `@Repository` — rilevato da component scan | `AddScoped<IOrderRepository, EfCoreOrderRepository>()` |
| `save(entity)` | `_db.Orders.Add(order)` + `SaveChangesAsync()` |
| `findById(id)` | `FindAsync(id)` |
| `findAll()` | `ToListAsync()` |
| `datasource.url` in `application.yml` | `ConnectionStrings:Default` in `appsettings.json` |
| `EntityManager` gestito da Spring | `DbContext` iniettato dal DI container |
