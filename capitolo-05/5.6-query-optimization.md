# 5.6 Ottimizzazione delle query: `AsNoTracking`, proiezioni e `AsSplitQuery`

> **Analogia Java**: equivalente di `FetchType`, query hints e proiezioni con `@SqlResultSetMapping` in JPA

---

## Introduzione

EF Core traccia ogni entità caricata dal database — è il change tracking, il meccanismo che permette a `SaveChangesAsync` di sapere cosa è cambiato. Utile per le scritture, inutile (e costoso) per le letture. Questo paragrafo copre tre strumenti per ottimizzare le query in lettura: `AsNoTracking`, le proiezioni con `Select`, e `AsSplitQuery` per le relazioni.

---

## Come si faceva in Java

```java
// JPA — FetchType per controllare il caricamento delle relazioni
@OneToMany(fetch = FetchType.LAZY)
private List<OrderItem> items;

// Query con proiezione — evita di caricare l'intera entità
@Query("SELECT new com.example.OrderSummary(o.id, o.customer, o.total) FROM Order o WHERE o.status = :status")
List<OrderSummary> findSummaryByStatus(@Param("status") OrderStatus status);

// EntityManager con hint per read-only
Query query = em.createQuery("FROM Order o", Order.class);
query.setHint("org.hibernate.readOnly", true);
List<Order> orders = query.getResultList();
```

---

## `AsNoTracking` — disabilita il change tracking per le letture

Ogni entità caricata senza `AsNoTracking` viene aggiunta alla cache del change tracker. Per endpoint in sola lettura (GET) questo è overhead puro.

```csharp
// Senza AsNoTracking — EF Core traccia ogni Order caricato
var orders = await _db.Orders.ToListAsync(ct);

// Con AsNoTracking — nessun tracking, query più veloce e meno memoria
var orders = await _db.Orders
    .AsNoTracking()
    .Where(o => o.Status == OrderStatus.Pending)
    .OrderByDescending(o => o.CreatedAt)
    .ToListAsync(ct);
```

**Quando usarlo**: su tutti i `GET` che non modificano i dati. Se non chiami `SaveChangesAsync` dopo, il tracking è inutile.

**Quando NON usarlo**: quando carichi un'entità per modificarla — il change tracker deve osservarla per generare l'`UPDATE`.

---

## Proiezioni con `Select` — carica solo i campi che servono

Invece di caricare l'intera entità, proietta su un DTO. EF Core genera una `SELECT` con solo le colonne necessarie.

```csharp
// DTO per la lista ordini — solo i campi utili alla UI
public record OrderSummary(int Id, string Customer, decimal Total, string Status);
```

```csharp
// Senza proiezione — SELECT * FROM Orders
var orders = await _db.Orders.ToListAsync(ct);

// Con proiezione — SELECT Id, Customer, Total, Status FROM Orders
var summaries = await _db.Orders
    .AsNoTracking()
    .Where(o => o.Status == OrderStatus.Pending)
    .Select(o => new OrderSummary(o.Id, o.Customer, o.Total, o.Status.ToString()))
    .ToListAsync(ct);
```

La proiezione è il pattern più efficace per gli endpoint di lista: restituisci solo ciò che la UI consuma, non l'intera entità con tutti i suoi campi.

---

## `AsSplitQuery` — per le relazioni con collezioni

Quando carichi un'entità con una relazione uno-a-molti (es. `Order` con `List<OrderItem>`), EF Core di default genera una singola query con `JOIN`. Su dataset grandi questo produce il problema del **cartesian explosion**: se un ordine ha 10 item, la query restituisce 10 righe con i dati dell'ordine ripetuti 10 volte.

```csharp
// Scenario ipotetico: Order con una collezione di OrderItem
// OrderItem non è nel progetto del corso — esempio per illustrare AsSplitQuery
public class Order
{
    public int Id { get; set; }
    public string Customer { get; set; } = string.Empty;
    public List<OrderItem> Items { get; set; } = [];
}
```

```csharp
// Query singola con JOIN — cartesian explosion su collezioni grandi
var orders = await _db.Orders
    .Include(o => o.Items)
    .ToListAsync(ct);
// → SELECT Orders.*, OrderItems.* FROM Orders JOIN OrderItems ON ...
//   se ogni ordine ha 10 item → N*10 righe trasferite

// AsSplitQuery — due query separate, nessuna moltiplicazione di righe
var orders = await _db.Orders
    .Include(o => o.Items)
    .AsSplitQuery()
    .ToListAsync(ct);
// → SELECT * FROM Orders
// → SELECT * FROM OrderItems WHERE OrderId IN (1, 2, 3, ...)
```

**Quando usarlo**: quando carichi collezioni con `Include` e le entità padre hanno molti figli. Non usarlo per default — due query sono due round trip al database.

## Combinare le tecniche

```csharp
// Lettura ottimizzata: AsNoTracking + proiezione + filtro
public async Task<IEnumerable<OrderSummary>> GetPendingSummariesAsync(CancellationToken ct = default)
{
    return await _db.Orders
        .AsNoTracking()
        .Where(o => o.Status == OrderStatus.Pending)
        .OrderByDescending(o => o.CreatedAt)
        .Select(o => new OrderSummary(o.Id, o.Customer, o.Total, o.Status.ToString()))
        .ToListAsync(ct);
}
```

Questa query:
- non traccia nulla (`AsNoTracking`)
- trasferisce solo le colonne necessarie (`Select`)
- filtra sul database, non in memoria (`Where` → SQL `WHERE`)
- ordina sul database (`OrderByDescending` → SQL `ORDER BY`)

## Query complesse: LINQ come alternativa a HQL

In Java scrivi query complesse in HQL/JPQL come stringhe. In EF Core le stesse query si scrivono in LINQ — tradotte in SQL dal provider. Per i casi in cui LINQ non basta, EF Core offre SQL raw.

### Aggregazioni e raggruppamenti

```java
// HQL — totale degli ordini per stato
"SELECT o.status, COUNT(o), SUM(o.total) FROM Order o GROUP BY o.status"
```

```csharp
// LINQ equivalente — tradotto in SQL con GROUP BY
var report = await _db.Orders
    .AsNoTracking()
    .GroupBy(o => o.Status)
    .Select(g => new
    {
        Status = g.Key,
        Count  = g.Count(),
        Total  = g.Sum(o => o.Total)
    })
    .ToListAsync(ct);
// → SELECT Status, COUNT(*), SUM(Total) FROM Orders GROUP BY Status
```

### Query con più condizioni e paginazione

```java
// HQL — filtro multiplo + paginazione
Query q = em.createQuery(
    "FROM Order o WHERE o.total > :min AND o.status = :status ORDER BY o.createdAt DESC");
q.setFirstResult(page * size);
q.setMaxResults(size);
```

```csharp
// LINQ equivalente
var orders = await _db.Orders
    .AsNoTracking()
    .Where(o => o.Total > minTotal && o.Status == status)
    .OrderByDescending(o => o.CreatedAt)
    .Skip(page * pageSize)
    .Take(pageSize)
    .ToListAsync(ct);
// → SELECT ... FROM Orders WHERE Total > @min AND Status = @status
//   ORDER BY CreatedAt DESC OFFSET @skip ROWS FETCH NEXT @take ROWS ONLY
```

### SQL raw — quando LINQ non basta

Per query che LINQ non riesce a tradurre bene (window functions, CTE, stored procedure), EF Core offre SQL raw:

```java
// JPA — native query per SQL non supportato da JPQL
@Query(value = "SELECT * FROM orders WHERE YEAR(created_at) = :year", nativeQuery = true)
List<Order> findByYear(@Param("year") int year);
```

```csharp
// EF Core — SQL raw con parametri (sicuro, no SQL injection)
var orders = await _db.Orders
    .FromSqlRaw("SELECT * FROM Orders WHERE YEAR(CreatedAt) = {0}", year)
    .AsNoTracking()
    .ToListAsync(ct);

// Oppure con parametri nominali (più leggibile)
var yearParam = new SqlParameter("year", year);
var orders = await _db.Orders
    .FromSqlRaw("SELECT * FROM Orders WHERE YEAR(CreatedAt) = @year", yearParam)
    .AsNoTracking()
    .ToListAsync(ct);
```

`FromSqlRaw` restituisce un `IQueryable` — puoi concatenarci `Where`, `OrderBy`, `Select` normalmente e EF Core li aggiunge alla query SQL.

## Spiegazione

**`AsNoTracking` vs `@Transactional(readOnly = true)`** — in Spring `readOnly = true` ottimizza la sessione Hibernate disabilitando il dirty checking. `AsNoTracking` fa la stessa cosa in EF Core: nessun tracking, nessuna memoria per lo snapshot delle entità.

**Proiezione vs `@SqlResultSetMapping`** — in JPA le proiezioni su DTO richiedono `@SqlResultSetMapping` o costruttori speciali nell'HQL. In EF Core basta `Select` con un record o una classe anonima — è LINQ standard.

**`Include` vs `FetchType.EAGER`** — `Include` in EF Core è opt-in per ogni query, non un'impostazione globale sull'entità. Non c'è l'equivalente di `FetchType.EAGER` che carica sempre la relazione — devi esplicitarlo nella query.

---

## Confronto con Java

| JPA / Hibernate | EF Core |
|---|---|
| `FetchType.LAZY` (default per collezioni) | Lazy loading disabilitato per default — usa `Include` esplicitamente |
| `FetchType.EAGER` | `Include(o => o.Items)` nella query |
| `query.setHint("readOnly", true)` | `AsNoTracking()` |
| HQL con costruttore per proiezione | `Select(o => new MyDto(...))` con LINQ |
| N+1 problem con LAZY | N+1 problem senza `Include` — stesso rischio |
| Cartesian explosion con `JOIN FETCH` su collezioni | Cartesian explosion con `Include` — `AsSplitQuery()` come rimedio |
| `@Transactional(readOnly = true)` | `AsNoTracking()` sul DbSet |
