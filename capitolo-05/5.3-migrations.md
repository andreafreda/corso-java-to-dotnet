# 5.3 Code-first migrations

> **Analogia Java**: equivalente di Flyway/Liquibase, ma le migration sono generate dal codice

---

## Introduzione

> "SONO TABELLE, NON LE SCALE DI HOGWARTS!"

Il database è in ascolto, `AppDbContext` è configurato. Manca la tabella `Orders`. In Java avevi due strade: lasciare che Hibernate generasse lo schema dalle entità con `ddl-auto`, oppure gestirlo esplicitamente con Flyway o Liquibase. EF Core funziona allo stesso modo — puoi usare `EnsureCreatedAsync()` per lo schema automatico, oppure le **migration** per un controllo esplicito e versionato. Nel corso usiamo le migration: sono la scelta corretta per qualsiasi ambiente che non sia un test locale usa-e-getta.

---

## Come si faceva in Java

```sql
-- Flyway — migration scritte a mano in SQL
-- resources/db/migration/V1__create_orders.sql
CREATE TABLE orders (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer VARCHAR(100) NOT NULL,
    total DECIMAL(18,2),
    status VARCHAR(20),
    created_at TIMESTAMP
);
```

```sql
-- Seconda migration: aggiunta colonna
-- resources/db/migration/V2__add_description.sql
ALTER TABLE orders ADD COLUMN description VARCHAR(500);
```

Con Flyway scrivi le migration a mano in SQL. Con EF Core le generi — poi le puoi modificare se serve.

Hibernate aveva anche un'altra via: `spring.jpa.hibernate.ddl-auto=create` (o `update`) generava lo schema direttamente dalle entità `@Entity` a runtime, senza migration. Comodo in sviluppo, PERICOLOSO IN PRODUZIONE — poteva droppare e ricreare tabelle. EF Core ha l'equivalente con `EnsureCreatedAsync()`, usato principalmente nei test.

## Creare ed applicare la prima migration

> 📝 **Code Step 10** — Crea la prima migration e applica al database

```bash
# Crea la migration — EF Core confronta il modello con il database e genera il diff
dotnet ef migrations add InitialCreate --project OrderService.Infrastructure --startup-project OrderService.Api

# Applica la migration al database
dotnet ef database update --project OrderService.Infrastructure --startup-project OrderService.Api
```

EF Core crea automaticamente la cartella `Migrations/` in `Infrastructure`:

```
OrderService.Infrastructure/
└── Migrations/
    ├── 20251201120000_InitialCreate.cs        ← migration generata
    ├── 20251201120000_InitialCreate.Designer.cs
    └── AppDbContextModelSnapshot.cs           ← snapshot del modello corrente
```

Il file generato appare così:

```csharp
// Migrations/20251201120000_InitialCreate.cs — generato da EF Core — verifica prima di applicare
public partial class InitialCreate : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.CreateTable(
            name: "Orders",
            columns: table => new
            {
                Id       = table.Column<int>(nullable: false)
                               .Annotation("SqlServer:Identity", "1, 1"),
                Customer = table.Column<string>(maxLength: 100, nullable: false),
                Total    = table.Column<decimal>(type: "decimal(18,2)", nullable: false),
                Status   = table.Column<string>(nullable: false),
                CreatedAt = table.Column<DateTime>(nullable: false)
            },
            constraints: table => table.PrimaryKey("PK_Orders", x => x.Id));
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropTable(name: "Orders");
    }
}
```

`Up` applica la migration, `Down` la annulla. EF Core li genera entrambi.

## Perché `--startup-project OrderService.Api`

EF Core ha bisogno di avviare l'applicazione per leggere la configurazione del `DbContext` — la stringa di connessione, i provider registrati. Il progetto di startup è `Api` perché lì vive `Program.cs`. Il progetto con le migration è `Infrastructure` perché lì vive `AppDbContext`.

Per il corso usiamo i flag espliciti — è più chiaro di qualsiasi scorciatoia.

## Il ciclo di vita delle migrations

```bash
# 1. Modifichi un'entità in Core (es. aggiungi una proprietà)
# 2. Generi una nuova migration
dotnet ef migrations add AddOrderDescription --project OrderService.Infrastructure --startup-project OrderService.Api

# 3. Verifichi il file generato — modificalo se necessario
# 4. Applichi
dotnet ef database update --project OrderService.Infrastructure --startup-project OrderService.Api

# Comandi utili
dotnet ef migrations list   # lista tutte le migrations
dotnet ef migrations remove # rimuove l'ultima migration non ancora applicata
dotnet ef database update 0 # rollback completo (solo sviluppo!)
```

## Applicare le migrations all'avvio (opzionale)

In sviluppo puoi applicare le migrations automaticamente all'avvio dell'applicazione — comodo, ma da non usare in produzione dove preferisci controllare ogni migration manualmente:

```csharp
// Program.cs — solo in Development, opzionale
if (app.Environment.IsDevelopment())
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();
}
```

## Spiegazione

**Migration generate vs scritte a mano** — EF Core confronta il modello C# con lo snapshot dell'ultima migration e genera il diff. Non devi scrivere SQL — ma devi *leggere* la migration generata prima di applicarla. EF Core fa un buon lavoro, ma in casi complessi (rename di colonne, split di tabelle) può servire un aggiustamento manuale.

**`AppDbContextModelSnapshot`** — EF Core mantiene uno snapshot del modello corrente. Quando generi una nuova migration, confronta il modello attuale con lo snapshot — non con il database. Non cancellare mai questo file.

**`Down` sempre generato** — il metodo `Down` permette il rollback. In produzione il rollback di una migration è un'operazione delicata (puoi perdere dati se hai droppato colonne), ma averlo è meglio che non averlo.

## Confronto con Java

| Flyway | EF Core Migrations |
|---|---|
| Migration scritte a mano in SQL | Migration generate automaticamente dal modello C# |
| Versioning con prefisso `V1__`, `V2__` nel nome del file | Timestamp nel nome del file |
| Flyway si esegue automaticamente all'avvio di Spring Boot | `dotnet ef database update` da CLI (o `MigrateAsync()` all'avvio, opzionale) |
| `flyway migrate` da CLI | `dotnet ef database update` |
| Nessun rollback automatico (serve `V1__undo.sql` in Flyway Pro) | Metodo `Down` generato automaticamente |
| File in `resources/db/migration/` — versionati insieme al codice | Cartella `Migrations/` in Infrastructure — versionata insieme al codice |
| `ddl-auto=create/update` — schema da entità a runtime (solo dev) | `EnsureCreatedAsync()` — stesso scopo, stesso avvertimento |
| `ddl-auto=validate` — verifica schema senza modificarlo | nessun equivalente diretto — le migration coprono questo caso |
