# 5.7 Dapper: micro-ORM per query raw

> **Analogia Java**: equivalente di MyBatis / JdbcTemplate

---

## Introduzione

EF Core è potente ma a volte è troppo. Per query analitiche complesse, report, o situazioni dove vuoi il massimo controllo sull'SQL senza rinunciare al mapping automatico sui tipi C#, esiste **Dapper**. È un micro-ORM: scrivi SQL, Dapper mappa il risultato sulle tue classi. Zero magia, zero overhead.

Questo paragrafo è un **esempio standalone** — non modifica `OrderService`. Puoi seguirlo su un progetto separato o direttamente su `OrderService.Infrastructure` se vuoi integrarlo.

---

## Come si faceva in Java

```java
// MyBatis — mapper interface con SQL esplicito
@Mapper
public interface OrderMapper {
    @Select("SELECT id, customer, total, status FROM orders WHERE status = #{status}")
    List<Order> findByStatus(String status);

    @Select("""
        SELECT status, COUNT(*) as count, SUM(total) as total
        FROM orders
        GROUP BY status
        """)
    List<OrderReport> getReport();
}
```

```java
// JdbcTemplate — ancora più raw
List<Order> orders = jdbcTemplate.query(
    "SELECT * FROM orders WHERE total > ?",
    (rs, row) -> new Order(rs.getInt("id"), rs.getString("customer"), rs.getBigDecimal("total")),
    minTotal
);
```

## Setup

> 🔀 **Esempio standalone** — eseguibile su `OrderService` esistente o progetto separato

```bash
dotnet add OrderService.Infrastructure package Dapper
```

Dapper estende `IDbConnection` con metodi di query. Non serve alcuna configurazione aggiuntiva — lavora con la stessa stringa di connessione già presente in `appsettings.Development.json`.

## Esempi di utilizzo

### Query semplice con mapping automatico

```csharp
using Dapper;
using Microsoft.Data.SqlClient;

var connectionString = configuration.GetConnectionString("Default")!;
await using var connection = new SqlConnection(connectionString);

// Dapper mappa automaticamente le colonne sui campi del tipo
var orders = await connection.QueryAsync<Order>(
    "SELECT Id, Customer, Total, Status FROM Orders WHERE Status = @Status",
    new { Status = "Pending" });
```

### Report con aggregazione

```csharp
public record OrderReport(string Status, int Count, decimal TotalAmount);

var report = await connection.QueryAsync<OrderReport>("""
    SELECT
        Status,
        COUNT(*)   AS Count,
        SUM(Total) AS TotalAmount
    FROM Orders
    GROUP BY Status
    ORDER BY TotalAmount DESC
    """);
```

### Filtro multiplo e paginazione

```csharp
var orders = await connection.QueryAsync<Order>("""
    SELECT Id, Customer, Total, Status, CreatedAt
    FROM Orders
    WHERE Total > @MinTotal
      AND CreatedAt >= @From
    ORDER BY CreatedAt DESC
    OFFSET @Skip ROWS FETCH NEXT @Take ROWS ONLY
    """,
    new { MinTotal = 100m, From = DateTime.UtcNow.AddDays(-30), Skip = 0, Take = 20 });
```

### Operazioni scalari e write

```csharp
// Conteggio
var count = await connection.ExecuteScalarAsync<int>(
    "SELECT COUNT(*) FROM Orders WHERE Status = @Status",
    new { Status = "Pending" });

// Update
var affected = await connection.ExecuteAsync(
    "UPDATE Orders SET Status = @Status WHERE Id = @Id",
    new { Status = "Confirmed", Id = orderId });
```

## Spiegazione

**Dapper vs EF Core** — non sono in competizione. EF Core copre il 90% dei casi: CRUD, relazioni, change tracking, migrations. Dapper è la scelta per il restante 10%: report, query analitiche, CTE, window functions — dove vuoi SQL esplicito senza nessun layer di astrazione. I due coesistono nello stesso progetto senza problemi.

**Dapper vs `FromSqlRaw`** — `FromSqlRaw` di EF Core ti dà SQL raw ma rimani nel contesto EF Core: change tracking attivo, puoi concatenare LINQ. Dapper è completamente separato: apri una connessione, esegui SQL, ottieni risultati. Più semplice, più prevedibile per le query di sola lettura.

**Mapping automatico** — Dapper mappa le colonne del risultato sui campi della classe per nome (case-insensitive). Se la colonna si chiama `TotalAmount` e il campo C# si chiama `TotalAmount`, il mapping è automatico. Nessun `RowMapper` da scrivere.

**Sicurezza SQL injection** — i parametri con `@NomeParametro` e oggetti anonimi `new { }` sono sempre parametrizzati. Non usare mai interpolazione di stringhe per costruire query SQL.

## Confronto con Java

| Java | .NET |
|---|---|
| MyBatis `@Select` con SQL esplicito | Dapper `QueryAsync<T>` |
| Mapping automatico colonne → campo | Dapper mapping automatico per nome |
| `JdbcTemplate.query()` con `RowMapper` | Dapper `QueryAsync<T>` — nessun RowMapper |
| `JdbcTemplate.queryForObject()` | Dapper `ExecuteScalarAsync<T>` |
| `JdbcTemplate.update()` | Dapper `ExecuteAsync` |
