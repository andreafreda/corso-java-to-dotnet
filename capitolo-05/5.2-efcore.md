# 5.2 Entity Framework Core: cos'è e come funziona

> **Analogia Java**: equivalente di Hibernate / Spring Data JPA

---

## Introduzione

Entity Framework Core è l'ORM ufficiale di .NET. Se hai usato Hibernate o Spring Data JPA, il modello mentale è lo stesso: scrivi classi C#, EF Core genera le query SQL, gestisce il tracking delle modifiche e astrae il database sottostante. Le differenze sono nei dettagli — e vale la pena capirle bene prima di scrivere codice.

---

## Come si faceva in Java

```java
// Hibernate / Spring Data JPA — il modello che conosci
@Entity
@Table(name = "orders")
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String customer;

    private BigDecimal total;

    @Enumerated(EnumType.STRING)
    private OrderStatus status = OrderStatus.PENDING;
}

// Repository con Spring Data
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByStatus(OrderStatus status);
}

// EntityManager come unità di lavoro
@PersistenceContext
private EntityManager em;
```

## I concetti chiave di EF Core

### DbContext — l'equivalente di `EntityManager`

`DbContext` è la classe centrale di EF Core. Rappresenta una sessione con il database — gestisce la connessione, traccia le modifiche alle entità, e coordina le operazioni di lettura e scrittura.

In Spring Data JPA hai `EntityManager` (o lo usi implicitamente tramite `JpaRepository`). In EF Core hai `DbContext` — più esplicito, più controllabile.

```csharp
// AppDbContext — una per applicazione, una istanza per richiesta HTTP
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Order> Orders { get; set; }
}
```

`DbSet<Order>` è l'equivalente di `JpaRepository<Order, Long>` — rappresenta la tabella e ti dà i metodi per interrogarla.

### Change Tracking — come EF Core sa cosa salvare

Quando carichi un'entità da EF Core, il `DbContext` la "osserva". Se modifichi una proprietà e chiami `SaveChangesAsync()`, EF Core genera automaticamente l'`UPDATE` corretto — esattamente come fa Hibernate con la dirty checking.

```csharp
// EF Core traccia automaticamente le modifiche
var order = await context.Orders.FindAsync(id);
order.Status = OrderStatus.Confirmed;  // EF Core "vede" la modifica
await context.SaveChangesAsync();       // genera UPDATE orders SET status = 'Confirmed' WHERE id = 1
```

In Java è lo stesso principio — Hibernate traccia le entità nel persistence context e genera le query al flush.

### Configurazione delle entità — Fluent API vs annotazioni

In Hibernate usi annotazioni JPA sull'entità (`@Column`, `@Table`, ecc.). In EF Core puoi fare lo stesso con Data Annotations, ma il modo idiomatico è la **Fluent API** nel `DbContext` — le regole di mapping stanno separate dall'entità, che rimane pulita.

```csharp
// Fluent API in OnModelCreating — il modo raccomandato
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Order>(entity =>
    {
        entity.ToTable("Orders");
        entity.HasKey(o => o.Id);
        entity.Property(o => o.Customer).IsRequired().HasMaxLength(100);
        entity.Property(o => o.Total).HasColumnType("decimal(18,2)");
        entity.Property(o => o.Status).HasConversion<string>();
    });
}
```

### IQueryable e LINQ — le query in C#

Invece di JPQL o HQL, in EF Core scrivi query con LINQ. Il provider EF Core traduce le espressioni LINQ in SQL a runtime.

```csharp
// LINQ → SQL generato automaticamente
var orders = await context.Orders
    .Where(o => o.Status == OrderStatus.Pending)
    .OrderBy(o => o.CreatedAt)
    .ToListAsync();

// SELECT * FROM Orders WHERE Status = 'Pending' ORDER BY CreatedAt
```

## `AppDbContext`

```
OrderService.Infrastructure/
└── Persistence/
    └── AppDbContext.cs    ← nuovo
```

```csharp
// Infrastructure/Persistence/AppDbContext.cs
using Microsoft.EntityFrameworkCore;
using OrderService.Core.Entities;

namespace OrderService.Infrastructure.Persistence;

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Order> Orders { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(entity =>
        {
            entity.ToTable("Orders");
            entity.HasKey(o => o.Id);
            entity.Property(o => o.Customer).IsRequired().HasMaxLength(100);
            entity.Property(o => o.Total).HasColumnType("decimal(18,2)");
            entity.Property(o => o.Status).HasConversion<string>();
        });
    }
}
```

## Spiegazione

**Perché Fluent API e non annotazioni** — le annotazioni JPA sull'entità mescolano dominio e persistenza. Qui `Order` è un'entità di dominio pulita, senza attributi. La configurazione EF Core sta in `AppDbContext`, in `Infrastructure` — esattamente dove deve stare in Clean Architecture.

**`HasConversion<string>()`** — dice a EF Core di salvare l'enum `OrderStatus` come stringa nel database invece che come intero. Più leggibile, più sicuro durante le migrations.

**`DbSet<Order>`** — EF Core usa questo come punto di accesso alla tabella `Orders`. Non serve dichiarare esplicitamente la tabella se usi `ToTable()` nel Fluent API o se il nome del `DbSet` corrisponde al nome della tabella.

**Lifetime del `DbContext`** — va registrato come `Scoped` nel DI container: una istanza per richiesta HTTP. Se lo registri come `Singleton` ottieni problemi di concorrenza e memory leak sul change tracker.

## Confronto con Java

| Hibernate / JPA | EF Core |
|---|---|
| `EntityManager` | `DbContext` |
| `@Entity`, `@Table`, `@Column` | Fluent API in `OnModelCreating` (o Data Annotations) |
| `@GeneratedValue(IDENTITY)` | `entity.Property(o => o.Id).ValueGeneratedOnAdd()` (default per `int`) |
| `@Enumerated(EnumType.STRING)` | `.HasConversion<string>()` |
| `JpaRepository<T, ID>` | `DbSet<T>` nel `DbContext` |
| `persistence.xml` / `application.yml` | `DbContextOptions` passato via DI |
| Dirty checking automatico | Change tracking automatico |
| JPQL / HQL | LINQ con provider EF Core |
