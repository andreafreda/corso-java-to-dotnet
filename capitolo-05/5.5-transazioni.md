# 5.5 Gestione delle transazioni: implicite ed esplicite

> **Analogia Java**: equivalente di `@Transactional` in Spring

---

## Introduzione

In Spring metti `@Transactional` su un metodo e Spring apre, commita o fa rollback della transazione per te. In EF Core il meccanismo è simile ma più esplicito: hai transazioni **implicite** (gestite automaticamente da `SaveChangesAsync`) e transazioni **esplicite** (quando hai bisogno di coordinare più operazioni).

---

## Come si faceva in Java

```java
// Spring — @Transactional apre e commita automaticamente
@Service
public class OrderService {

    @Transactional
    public Order createOrder(CreateOrderRequest request) {
        Order order = new Order(request.customer(), request.total());
        orderRepository.save(order);
        auditLogRepository.save(new AuditLog("ORDER_CREATED", order.getId()));
        return order;  // Spring commita tutto qui
    }

    @Transactional(rollbackFor = Exception.class)
    public void processOrder(int id) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.setStatus(OrderStatus.CONFIRMED);
        // se lancia eccezione → rollback automatico
    }
}
```

```java
// Transazione programmatica con TransactionTemplate
transactionTemplate.execute(status -> {
    orderRepository.save(order);
    auditLogRepository.save(log);
    return order;
});
```

## Transazioni implicite

`SaveChangesAsync()` è già atomico: tutte le modifiche tracciate dal `DbContext` vengono committate in un'unica transazione. Se una operazione fallisce, EF Core fa rollback di tutto.

```csharp
// Questo è già transazionale — nessuna configurazione richiesta
// Nota: AuditLog è un'entità ipotetica usata come esempio — non è nel progetto del corso
public async Task<Order> AddAsync(Order order, CancellationToken ct = default)
{
    _db.Orders.Add(order);
    _db.AuditLogs.Add(new AuditLog { Action = "ORDER_CREATED", CreatedAt = DateTime.UtcNow });
    await _db.SaveChangesAsync(ct);  // commit atomico di entrambe le operazioni
    return order;
}
```

Finché le operazioni usano lo stesso `DbContext` e lo stesso `SaveChangesAsync`, sono nella stessa transazione. Non serve nulla di più.

---

## Transazioni esplicite

Quando hai bisogno di più `SaveChangesAsync` nella stessa transazione — o di coordinare operazioni che non puoi raggruppare in un unico `SaveChanges` — usi una transazione esplicita:

```csharp
// Servizio con transazione esplicita
public async Task<Order> CreateOrderWithInventoryAsync(
    CreateOrderRequest request,
    CancellationToken ct = default)
{
    await using var transaction = await _db.Database.BeginTransactionAsync(ct);
    try
    {
        var order = new Order
        {
            Customer = request.Customer,
            Total    = request.Total,
            Status   = OrderStatus.Pending
        };

        _db.Orders.Add(order);
        await _db.SaveChangesAsync(ct);  // primo SaveChanges — dentro la transazione

        // operazione separata che dipende dall'Id generato
        // AuditLog è un'entità ipotetica — non è nel progetto del corso
        _db.AuditLogs.Add(new AuditLog
        {
            OrderId = order.Id,
            Action  = "ORDER_CREATED"
        });
        await _db.SaveChangesAsync(ct);  // secondo SaveChanges — stessa transazione

        await transaction.CommitAsync(ct);
        return order;
    }
    catch
    {
        await transaction.RollbackAsync(ct);
        throw;
    }
}
```

## Transazioni e scope del DbContext

La transazione è legata all'istanza del `DbContext`. Poiché `DbContext` è `Scoped` (una istanza per richiesta HTTP), una transazione esplicita aperta in un service è visibile a tutti i repository che usano lo stesso `DbContext` nella stessa richiesta.

```csharp
// Entrambi usano la stessa istanza di DbContext → stessa transazione
public class OrderService
{
    private readonly AppDbContext _db;
    private readonly IOrderRepository _orderRepo;
    private readonly IAuditRepository _auditRepo;

    // _orderRepo e _auditRepo ricevono lo stesso _db per DI → stessa transazione
}
```

---

## Spiegazione

**Quando usare la transazione implicita** — nella maggior parte dei casi. Se tutte le operazioni usano lo stesso `DbContext` e un solo `SaveChangesAsync`, sei già atomico. Non serve codice aggiuntivo.

**Quando usare la transazione esplicita** — quando hai più `SaveChangesAsync` distinti che devono essere atomici, oppure quando chiami stored procedure o SQL raw che devono partecipare alla stessa transazione.

**Rollback automatico** — in EF Core non esiste l'equivalente di `@Transactional(rollbackFor = ...)`. Il rollback avviene se non chiami `CommitAsync`. Nelle transazioni implicite, se `SaveChangesAsync` lancia un'eccezione, EF Core non ha committato nulla — il rollback è automatico a livello di database.

**`await using`** — `IDbContextTransaction` implementa `IAsyncDisposable`. `await using` garantisce il rollback automatico se esci dallo scope senza aver chiamato `CommitAsync` — equivalente al rollback automatico di Spring in caso di eccezione.

---

## Confronto con Java

| Spring | EF Core |
|---|---|
| `@Transactional` sul metodo | `SaveChangesAsync()` implicito (caso semplice) |
| `@Transactional` con più repository | Stesso `DbContext` iniettato — stessa transazione implicita |
| `TransactionTemplate` programmatico | `BeginTransactionAsync` + `CommitAsync` / `RollbackAsync` |
| Rollback automatico su eccezione non checked | `await using` + nessun `CommitAsync` → rollback automatico |
| Propagazione `REQUIRED` (default) | `DbContext` Scoped — stesso scope = stessa transazione |
| `@Transactional(readOnly = true)` | Nessun equivalente diretto — usa `AsNoTracking()` (vedi 5.6) |
